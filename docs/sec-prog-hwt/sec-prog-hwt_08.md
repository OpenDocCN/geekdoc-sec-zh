几乎没有程序是完全自包含的；几乎所有程序都会调用其他程序以获取资源，例如操作系统提供的程序、软件库等。有时这种调用其他资源并不明显或涉及大量的“隐藏”基础设施，这些基础设施必须依赖，例如实现动态库的机制。显然，你必须小心你程序所信任的其他资源，你必须确保你向它们发送请求的方式。

# 8.1. 仅调用安全库例程

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/call-only-safe.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/call-only-safe.html)

有时安全性与抽象（信息隐藏）和重用的发展原则之间可能存在冲突。问题在于，某些高级库例程可能安全或不安全，而它们的规范不会告诉你。即使特定的实现是安全的，也可能无法确保其他版本的例程将是安全的，或者相同的接口在其他平台上将是安全的。

最后，如果你的应用程序必须安全，有时你必须重新实现库例程的自己的版本。基本上，如果你不能确定库例程将执行你为安全所需的必要操作，你必须重新实现例程。是的，在某些情况下，库的实现应该被修复，但如果选择了一个具有安全弱点的库例程，你的用户将会受到伤害。如果可能，当你必须重新实现某些内容时，尝试使用高级接口 - 这样，你可以在使用安全的高级接口的系统上切换到高级接口。

如果可能，测试该例程是否安全，如果是，则使用它 - 理想情况下，你可以将其作为编译或安装的一部分（例如，作为“autoconf”脚本的一部分）进行此测试。对于某些条件，这种运行时测试是不切实际的，但对于其他条件，这可以消除许多问题。如果你不想重新实现库，至少测试以确保它是安全的，如果不安全，则停止安装。这样，用户就不会意外安装不安全的程序，并且会知道问题是什么。

# 8.2\. 限制调用到有效值

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/limit-call-outs.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/limit-call-outs.html)

确保任何调用其他程序的调用只允许每个参数的有效和预期值。这比听起来要困难，因为许多库调用或命令以可能令人惊讶的方式调用底层例程。例如，许多系统调用是通过调用外壳间接实现的，这意味着传递外壳元字符可能会产生危险的效果。因此，让我们讨论元字符。

# 8.3\. 处理元字符

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/handle-metacharacters.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/handle-metacharacters.html)

许多系统，例如 SQL 解释器和命令行外壳，都有 *元字符*，也就是说，它们输入中的这些字符不会被解释为数据。这些字符可能是命令，或者用于将数据与命令或其他数据分隔开。如果你使用该系统的接口有语言规范，那么它肯定有元字符。如果你的程序调用其他系统并允许攻击者插入这样的元字符，通常的结果是攻击者可以完全控制你的程序。

## 8.3.1\. SQL 注入

大多数数据库系统包括一种语言，可以让你创建任意查询，通常还有许多其他功能。SQL 语言特别常见，许多其他数据库语言都与 SQL 语言相似。

SQL 及其相关语言，不出所料，也包括元字符。当元字符作为输入触发 SQL 元字符时，通常称为 *SQL 注入*。即使语言技术上不是 SQL，如果它是对数据库系统语言的攻击，通常仍然被称为 SQL 注入攻击。有许多触发 SQL 注入攻击的方法；攻击者可以插入单引号或双引号、分号（作为命令分隔符）、"--"（这是一个注释标记）等等。参见 [SPI Dynamic 的论文“SQL Injection: Are your Web Applications Vulnerable?”](http://www.spidynamics.com/papers/SQLInjectionWhitePaper.pdf) 以进一步讨论这个问题。

防止 SQL 注入的最佳单一方法可能是使用预处理语句。预处理语句允许程序员识别占位符；然后现有的库会为该特定实现正确地转义它。这种方法有许多优点。首先，由于库为您执行转义，因此它更简单，也更可能正确。其次，它往往会产生更容易维护的代码，因为代码更容易阅读。预处理语句在处理 SQL 时尤为重要，因为不同的 SQL 引擎有不同的语法规则。

当然，还有其他方法。您可以编写自己的转义代码，但这很难正确实现，而且通常浪费时间，因为通常有现成的库来完成这项工作。您还可以使用存储过程，这也有助于防止 SQL 注入。

有其他限制输入的解决方案。不同的 SQL 实现有不同的元字符，因此黑名单在防止 SQL 注入方面甚至更不是一个好主意。如第五章中所述，定义一个非常有限的模式，并且只允许匹配该模式的数据进入；如果您将模式限制为^[0-9]$或^[0-9A-Za-z]*$，那么您就不会有问题。如果您必须处理可能包含 SQL 元字符的数据，一个很好的方法是在存储之前（尽可能早地）将其转换为其他编码，例如 HTML 编码（在这种情况下，您还需要对任何和号字符进行编码）。此外，即使数据是数字的，也要在所有用户输入前添加和后添加引号；这样，空格和其他类型数据的插入就不会那么危险。

## 8.3.2\. 壳注入

许多元字符问题涉及 shell 元字符。试图利用 shell 元字符处理中的漏洞的攻击被称为*shell 注入*攻击。例如，标准的类 Unix 命令 shell（通常存储在/bin/sh 中）对一些字符有特殊的解释。如果这些字符被发送到 shell，那么除非它们被转义，否则将使用它们的特殊解释；这一事实可以用来破坏程序。根据 WWW 安全 FAQ [Stein 1999, Q37]，这些元字符是：

```
 & ; ` ' \ " &#124; * ? ~ < > ^ ( ) [ ] { } $ \n \r
```

#字符是注释字符，因此它也是一个元字符。分隔符的值可以通过设置 IFS 环境变量来更改，但如果您无法信任该变量的来源，您应该将其丢弃或将其重置，作为您环境变量处理的一部分。

不幸的是，在现实生活中这不是一个完整的列表。以下是一些可能引起问题的其他字符：

+   '!' 在表达式中表示“非”（正如它在 C 语言中那样）；如果测试程序的返回值，前置'!'可能会欺骗脚本认为某些成功时失败了，反之亦然。在某些 shell 中，“!”还可以访问命令历史，这可能会引起真正的问题。在 bash 中，这仅发生在交互模式下，但 tcsh（在某些 Linux 发行版中发现的 csh 克隆）即使在脚本中也使用"!"。

+   '#' 是注释字符；该行上所有后续文本都将被忽略。

+   '-' 可能会被误解为引导一个选项（或者，作为 --，禁用所有后续选项）。即使它在文件名“中间”，如果它前面是 shell 认为的空白字符，你可能会遇到问题。

+   ' '（空格）、'\t'（制表符）、'\n'（换行）、'\r'（回车）、'\v'（垂直空格）、'\f'（换页）和其他空白字符可能具有许多危险效果。它们可能会将一个“单个”文件名变成多个参数，例如，或者在存储时将单个参数变成多个参数。换行和回车有若干额外的危险，例如，它们可以用于在某些程序中创建“伪造”的日志条目，或者插入在单独的命令之前，然后执行该命令（如果底层协议使用换行或回车作为命令分隔符）。

+   其他控制字符（特别是 NIL）可能会对某些 shell 实现造成问题。

+   根据你的使用情况，甚至可以想象“.”（在当前 shell 中运行）和“=”（用于设置变量）可能是令人担忧的字符。然而，到目前为止，我所找到的任何这些字符有问题的情况都有其他（更严重的）安全问题。

忘记这些字符中的一个可能会造成灾难，例如，许多程序在作为 shell 元字符时省略反斜杠 [rfp 1999]。正如在第五章中讨论的，一些人的建议是在输入时立即转义这些字符中的所有字符。

因此，简单地创建一个禁止字符的列表是一个坏主意（因为这相当于一个*黑名单*）。相反，确定可接受的字符，然后禁止或正确转义所有其他字符（一个*白名单*）。

使 shell 元字符特别普遍的是，几个重要的库调用，如 popen(3)和 system(3)，是通过调用命令 shell 来实现的，这意味着它们也会受到 shell 元字符的影响。同样，execlp(3)和 execvp(3)可能会调用 shell。许多指南建议完全避免使用 popen(3)、system(3)、execlp(3)和 execvp(3)，并在尝试生成进程时直接在 C 中使用 execve(3) [Galvin 1998b]。至少，当可以使用 execve(3)时，避免使用 system(3)；因为 system(3)使用 shell 来展开字符，所以在 system(3)中有更多的机会进行恶作剧。以类似的方式，Perl 和 shell 反引号(`)也会调用命令 shell；有关 Perl 的更多信息，请参阅第 10.2 节。

## 8.3.3. 有问题的路径名和文件名

“路径名”是一系列字节序列，用于描述如何找到文件系统对象。在类 Unix 系统中，路径名是由一个或多个通过一个或多个“/”分隔的文件名序列组成。在 Windows 系统中，路径名更为复杂，但基本概念相同。在实践中，许多人使用“文件名”一词来指代路径名。

不幸的是，路径名通常至少部分由不受信任的用户控制。例如，使用文件/目录名作为识别相关数据的键通常很有用，但这可能导致不受信任的用户控制文件名。另一个例子是在监控或管理共享系统（例如，虚拟机或容器化文件系统）时；在这种情况下，不受信任的监控对象控制文件名。即使攻击者不应该能够获得这种控制，通常也很重要作为深度防御措施来应对这种问题，以对抗那些获得少量控制权的攻击者。

一个明显的例子是，系统不应该允许在某些方向之外进行重定向（例如，Web 服务器的“文档根目录”）。如果一个 Web 应用程序允许“.”、“/”和/或“\”，那么可能很容易违反该规则。例如，如果一个程序尝试访问由“trusted_root_path”和“username”拼接而成的路径，攻击者可能能够创建一个名为“../../../mysecrets”的用户名并绕过限制。一如既往，对于将用于创建文件名的信息，请使用非常有限的白名单。

微软 Windows 路径名在安全处理上可能比较困难。Windows 路径名的解释取决于 Windows 的版本和使用的 API（许多调用使用 CreateFile，它支持\\.\ -，并且这些调用对路径名的解释与其他不支持的调用不同）。最明显的是，“letter:”和“\\server\share...”在 Windows 中有特殊含义。一个更严重的问题是存在保留的文件名，其形式取决于使用的 API 和本地配置。内置的保留设备名称如下：CON, PRN, AUX, NUL, COM1, COM2, COM3, COM4, COM5, COM6, COM7, COM8, COM9, LPT1, LPT2, LPT3, LPT4, LPT5, LPT6, LPT7, LPT8, 和 LPT9。更糟糕的是，驱动程序可以创建更多的保留名称——因此你实际上无法提前知道哪些名称被保留。你应该避免创建带有保留名称的文件名，无论是带扩展名还是不带扩展名；如果攻击者能够诱骗程序读取/写入该名称（例如，com1.txt），则可能会（取决于 API）导致对设备而不是文件的读取或写入。在这种情况下，即使是简单的字母数字也可能造成灾难，并被解释为元字符——这是一个罕见的情况，因为通常字母数字是安全的。Windows 支持“/”作为目录分隔符，但传统上使用“\"作为目录分隔符（这很烦人，因为\广泛用作转义字符）。在 Windows 中，不要以空格或句点结束文件或目录名；底层文件系统可能支持它，但 Windows 外壳和用户界面通常不支持。更多信息请参阅[`msdn.microsoft.com/en-us/library/aa365247.aspx`](http://msdn.microsoft.com/en-us/library/aa365247.aspx)。

类 Unix 系统上的文件名和路径名处理起来也不总是容易。在大多数类 Unix 系统上，文件名可以是任何不包含\0（终止符）或斜杠的字节序列。一个常见的误解是 Unix 文件名是一系列字符。Unix 文件名不是由一个或多个字符组成的字符串；它们仅仅是字节序列，因此文件名不需要是合法字符序列。例如，虽然将文件名解释为字符的 UTF-8 编码是一个常见的约定，但大多数系统实际上并不强制执行这一点。实际上，它们往往不强制执行任何规则，因此可以创建许多有问题的文件名，包括包含空格（或仅包含空格）、控制字符（包括换行符、制表符、转义符等）、非法 UTF-8 字节或以“-”开头（命令选项的标记）的文件名。这些有问题的文件名可能会在以后造成麻烦。一些文件名的问题可能特定于 shell，但文件名问题并不局限于 shell。

常见问题在于许多命令中“-”是选项标志，但它却是文件名的一个合法开头。一个简单的解决方案是在需要的地方给所有全局匹配符或文件名前加上“./”，这样它们就不能以“-”开头。所以例如，永远不要用“*.pdf”来指代一组 PDF 文件；使用“./*.pdf”。

在显示或存储路径名时要小心，因为它们可能包含换行符、制表符、转义符（这可能开始终端控制），或者不是合法字符串的序列。在某些系统中，仅仅显示文件名就可能触发终端控制，这可能导致以显示者的权限运行命令。

想要获取更详细的信息，请参阅[Filenames and Pathnames in Shell: How to do it correctly](http://www.dwheeler.com/essays/filenames-in-shell.html)。

## 8.3.4. 其他注入问题

许多程序，尤其是那些为人类交互设计的程序，都有执行“额外”活动的“转义”码。其中更常见（也更危险）的一种转义码是弹出命令行的转义码。确保这些“转义”命令不能被包含（除非你确信特定的命令是安全的）。例如，许多面向行的邮件程序（如 mail 或 mailx）使用波浪号（~）作为转义字符，然后可以用来发送多个命令。因此，看似无害的命令如“mail admin < file-from-user”可以用来执行任意程序。交互式程序如 vi、emacs 和 ed 有允许用户从会话中运行任意 shell 命令的“转义”机制。始终检查你调用的程序的文档，以寻找转义机制。最好是调用仅适用于其他程序使用的程序；参见第 8.4 节。

避免转义码的问题甚至涉及到低级硬件组件及其仿真器。大多数调制解调器实现了所谓的“Hayes”命令集。除非禁用命令集，引入延迟，否则短语“+++”，然后是另一个延迟会迫使调制解调器将任何随后的文本解释为发送给调制解调器的命令。这可以用来实施拒绝服务攻击（通过发送“ATH0”，一个挂断命令）或者甚至强迫用户连接到其他人（一个复杂的攻击者可以将用户的连接重新路由到攻击者控制的机器）。对于调制解调器的特定情况，这很容易应对（例如，在调制解调器初始化字符串中添加“ATS2-255”），但一般问题仍然存在：如果你正在控制一个低级组件，或者一个组件的仿真器，确保你禁用或以其他方式处理它们内置的任何转义码。

许多“终端”接口实现了早已消失的物理终端（如 VT100）的转义码。这些代码可能很有用，例如，用于加粗字符、更改字体颜色或移动到终端界面中的特定位置。然而，不要允许任意不受信任的数据直接发送到终端屏幕，因为其中一些代码可能会引起严重问题。在某些系统上，您可以重新映射键（例如，当用户按下“Enter”或功能键时，它会发送您希望他们运行的命令）。在某些系统上，甚至可以发送清除屏幕、显示您希望受害者运行的命令集，然后将该集“发送回”，迫使受害者运行攻击者选择的命令，甚至不需要等待按键。这通常是通过“页面模式缓冲区”实现的。这就是为什么模拟的 tty（通常表示为设备文件，通常位于`/dev/`中）应该只能由其所有者写入，永远不应该由其他人写入 - 它们永远不应该设置“其他写入”权限，并且除非只有用户是该组的成员（即，“用户专用组”方案），否则“组写入”权限也不应该为终端设置 [Filipski 1986]。如果您在（模拟的）终端向用户显示数据，您可能需要从发送回用户的数据中过滤掉所有控制字符（值小于 32 的字符），除非它们被您识别为安全。更糟糕的是，您可以识别制表符和换行符（以及可能还有回车符）为安全，移除所有其他字符。具有高位的字符（即，值大于 127）在某种程度上更难以处理；一些旧系统将它们实现为未设置，但简单地过滤它们会抑制许多国际使用。在这种情况下，您需要查看您具体情况的具体细节。

相关问题之一是空字符（字符 0）可能会产生意想不到的效果。大多数 C 和 C++函数都假设这个字符标志着字符串的结束，但其他语言（如 Perl 和 Ada95）中的字符串处理例程可以处理包含空字符的字符串。由于许多库和内核调用都使用 C 约定，结果是检查的内容并不是实际使用的 [rfp 1999]。

当调用另一个程序或引用一个文件时，指定其完整路径（例如，`/usr/bin/sort`）可能是明智的。对于程序调用，这将消除调用“错误”命令时可能出现的错误，即使 PATH 值设置不正确。对于其他文件引用，这减少了从“不良”起始目录产生的问题。

# 8.4. 仅调用面向程序员的接口

> [原文链接](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/call-intentional-apis.html)

只调用旨在由程序使用的应用程序编程接口 (API)。通常，一个程序可以调用任何其他程序，包括那些真正为人类交互设计的程序。然而，通常不建议以人类的方式调用旨在人类交互的程序。问题是程序的人类界面故意功能丰富，并且往往难以完全控制。如第 8.3 节处理元字符中所述，交互式程序通常有“逃生”代码，这可能允许攻击者执行不希望的功能。此外，交互式程序通常试图直观地推断“最可能的”默认值；这可能不是你预期的默认值，攻击者可能找到利用这一点的方法。

你通常不应该直接调用的程序示例包括 mail、mailx、ed、vi 和 emacs。至少，在检查它们的输入之前不要调用这些程序。

通常有参数可以提供更安全的程序功能访问，或者有针对程序使用的不同 API 或应用程序；使用这些代替。例如，不要调用文本编辑器来编辑一些文本（例如 ed、vi 或 emacs），而应该使用 sed。

# 8.5. 检查所有系统调用返回值

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/check-returns.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/check-returns.html)

每个可能返回错误条件的系统调用都必须检查该错误条件。一个原因是几乎所有系统调用都需要有限的系统资源，用户可以通过各种方式影响资源。可以通过 setuid/setgid 程序通过 setrlimit(3) 和 nice(2) 等调用设置限制。服务器程序和 CGI 脚本的外部用户可能仅通过大量并发请求就能导致资源耗尽。如果错误无法优雅地处理，那么就按照前面讨论的进行安全失败。

# 8.6. 避免使用 vfork(2)

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/avoid-vfork.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/avoid-vfork.html)

在类 Unix 系统中创建新进程的便携式方法是使用 fork(2)调用。BSD 引入了一个名为 vfork(2)的变体，作为一种优化技术。在 vfork(2)中，与 fork(2)不同，子进程会借用父进程的内存和控制线程，直到调用 execve(2V)或发生退出；父进程在子进程使用其资源时被挂起。其理由是，在旧的 BSD 系统中，fork(2)实际上会导致内存被复制，而 vfork(2)则不会。Linux 从未有过这个问题；因为 Linux 内部使用写时复制语义，Linux 只有在页面发生变化时才复制页面（实际上，还有一些必须复制的表；在大多数情况下，它们的开销并不显著）。尽管如此，由于一些程序依赖于 vfork(2)，最近 Linux 实现了 BSD vfork(2)语义（之前 vfork(2)是 fork(2)的一个别名）。

vfork(2)存在一些问题。从可移植性的角度来看，vfork(2)的问题实际上对于进程来说不干扰其父进程相当棘手，尤其是在高级语言中。不干扰的要求适用于实际生成的机器代码，许多编译器生成隐藏的临时变量和其他代码结构，导致意外的干扰。结果是，使用 vfork(2)的程序在代码更改或甚至编译器版本更改时很容易失败。

对于安全程序来说，在 Linux 系统上问题更严重，因为 Linux（至少 2.2 版本到 2.2.17）在 vfork()实现中存在竞争条件。如果一个特权进程在 Linux 中使用 vfork(2)/execve(2)对来执行用户命令，当子进程已经以用户的 UID 运行但尚未进入 execve(2)时，会存在竞争条件。用户可能能够向此进程发送信号，包括 SIGSTOP。由于 vfork(2)的语义，特权父进程也会被阻塞。因此，一个非特权进程可能导致特权进程停止，从而导致特权进程服务的拒绝服务。至少 FreeBSD 和 OpenBSD 有专门处理这种情况的代码，据我所知，它们不受到这个问题的影响。感谢 Solar Designer，他在 2000 年 10 月 7 日的“security-audit”邮件列表上注意并记录了这个问题。

关于 vfork(2)的底线很简单：*不要*在你的程序中使用 vfork(2)。这不应该很难；vfork(2)的主要用途是支持需要 vfork 语义的旧程序。

# 8.7. 在检索嵌入式内容时防止 Web 虫

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/embedded-content-bugs.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/embedded-content-bugs.html)

一些数据格式可以嵌入对内容的引用，当数据被查看时（无需等待用户选择），这些内容会自动检索（不是等待用户选择）。如果可以通过互联网（例如，通过万维网）触发这些数据的检索，那么就有可能利用这种能力在不让读者知情的情况下获取关于读者的信息，在某些情况下甚至可以强迫读者执行某些活动。这种隐私问题有时被称为“网络虫子”。

在网络虫子中，一个引用被有意插入到文档中，并由内容作者用来跟踪谁、在哪里以及多久阅读一次文档。作者还可以基本上监视一个“被虫子感染的”文档是如何从一个个人传到另一个个人，或者从一个组织传到另一个组织的。

HTML 格式已经存在这个问题一段时间了。根据[隐私基金会](http://www.privacyfoundation.org)的说法：

> 网络虫子今天被互联网广告公司广泛用于网页和基于 HTML 的电子邮件消息中进行跟踪。它们通常大小为 1x1 像素，以在屏幕上使其不可见，从而掩盖它们被用于跟踪的事实。然而，它们可以是任何图像（使用 img 标签）；其他可以实施网络虫子的 HTML 标签，例如框架、表单调用和脚本。仅通过调用网络虫子本身，就会为“虫子感染”的网站提供读者的 IP 地址、读者访问的页面以及有关浏览器的各种信息；通过同时使用 cookies，通常可以确定读者的具体身份。关于网络虫子的调查可在[`www.securityspace.com/s_survey/data/man.200102/webbug.html`](http://www.securityspace.com/s_survey/data/man.200102/webbug.html)找到。

更令人担忧的是，其他文档格式似乎也具有这种能力。当使用网络浏览器查看网站上的 HTML 时，有其他方式可以获取有关谁正在浏览数据的资料，但当你从电子邮件中查看另一种格式的文档时，很少有用户会预期仅仅阅读文档这一行为可以被监控。然而，对于许多格式来说，阅读文档的行为是可以被监控的。例如，最近确定微软 Word 可以支持网络虫子；参见[隐私基金会咨询信息](http://www.privacyfoundation.org/advisories/advWordBugs.html) 。正如他们在咨询中提到的，微软 Excel 和微软 PowerPoint 的最新版本也可以被植入虫子。在某些情况下，可以使用 cookies 获取更多信息。

网络虫子主要是文件格式设计的问题。如果你的用户重视他们的隐私，你可能希望限制包含文件的自动下载。一个例外可能是当文件本身正在下载时（例如，通过网络浏览器）；同时从同一位置下载其他文件则不太可能引起用户的关注。

# 8.8. 隐藏敏感信息

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/hide-sensitive-information.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/hide-sensitive-information.html)

敏感信息应该在输入、输出以及存储在系统中时都隐藏起来，不让好奇的目光窥视。敏感信息当然包括信用卡号码、账户余额和家庭地址，在许多应用程序中也包括姓名、电子邮件地址和其他私人信息。

基于 Web 的应用程序应该对所有包含敏感信息的用户通信进行加密；通常的方法是使用“https:”协议（在 SSL 或 TLS 之上的 HTTP）。根据 HTTP 1.1 规范（IETF RFC 2616 第 15.1.3 节），使用 HTTP 协议的服务提供者**不应**使用基于 GET 的表单提交敏感数据，因为这会导致这些数据被编码在请求 URI 中。许多现有的服务器、代理和用户代理会在可能被第三方看到的地方记录请求 URI。相反，应使用基于 POST 的提交，这些提交正是为此目的而设计的。

这类敏感数据的数据库也应该在任何存储设备上（例如磁盘上的文件）进行加密。当然，这种加密并不能防止攻击者破解安全应用程序，因为显然应用程序也需要一种方式来访问加密数据。然而，它确实提供了一些防御措施，以防止攻击者获取数据的备份磁盘，但没有获取解密这些数据的密钥。如果攻击者未能破解应用程序，但设法部分破解了相关系统，足以查看存储的数据，这也提供了一些防御措施——他们现在必须破解加密算法才能获取数据。在许多情况下，数据可能会无意中传输（例如，核心文件），这也防止了这种情况。然而，值得注意的是，这种防御并不像你想象的那么强大，因为服务器本身可能被篡改或破解。
