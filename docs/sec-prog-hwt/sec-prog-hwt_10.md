本书余下部分讨论的问题通常适用于所有语言（尽管某些问题在特定语言中更为常见或不存在）。然而，也存在许多特定于语言的安全问题。其中许多可以概括如下：

+   在实际可行的情况下，打开所有相关的警告和保护机制。对于编译型语言，这包括编译时机制和运行时机制。一般来说，与安全相关的程序应该在没有警告的情况下编译干净。

+   如果可以使用“安全模式”（例如，限制可执行文件活动的模式），请使用它。许多解释型语言包括这样的模式。一般来说，不要依赖安全模式提供绝对的保护；大多数语言的安全模式尚未得到充分的安全分析，并且当它们被分析时，人们通常会发现许多利用它的方法。然而，通过编写安全模式之外的代码，然后添加安全模式，你最终实现了多层次防御（因为在许多情况下，攻击者必须破坏你的应用程序代码和安全模式）。

+   避免语言中的危险和已弃用操作。所谓“危险”，是指难以正确使用的操作。例如，许多语言包括一些“魔法”机制或函数，即它们试图通过启发式方法推断出“正确”的操作。通常你应该避免使用它们，因为攻击者可能能够利用启发式方法执行危险的操作而不是预期的操作。一个常见的错误是“偏移量错误”，其中边界偏移了一个单位，有时这些错误可能导致可利用的错误。一般来说，编写代码时应尽量减少偏移量错误的可能性。如果语言中有标准约定（例如，编写循环），则应使用它们。

+   确保语言的底层基础设施（例如，运行时库）可用且安全。

+   自动垃圾回收字符串的语言应特别注意立即擦除秘密数据（特别是秘密密钥和密码）。

+   精确了解你所使用的操作的语义。在其文档中查找每个操作的语义。除非你确定它们不可能相关，否则不要忽略返回值。不要忽略“有符号”和“无符号”值之间的区别。这在不支持异常的语言中尤其困难，比如 C，但就是这样。

这里有一些特定语言的要点。然而，不要忘记其他地方讨论过的问题。例如，大多数语言都有一个格式化库，因此请确保攻击者无法控制格式命令（有关更多信息，请参阅第 9.4 节）。

# 10.1\. C/C++

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/c-cpp.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/c-cpp.html)

使用 C 或 C++ 开发安全代码是可能的，但这两种语言都包含了一些基本的设计决策，使得编写安全代码变得更加困难。C 和 C++ 容易导致缓冲区溢出，强迫程序员进行自己的内存管理，并且在类型系统中相当宽松。对于系统程序（如操作系统内核），C 和 C++ 是不错的选择。对于应用程序，C 和 C++ 往往被过度使用。强烈考虑使用更高层次的编程语言，至少对于应用程序的大部分部分。但显然，有许多现有的 C 和 C++ 程序不会完全重写，许多开发者可能会选择在 C 和 C++ 中进行开发。

C 和 C++ 程序最大的安全问题是缓冲区溢出；有关更多信息，请参阅第六章。

C 和 C++的另一个问题是开发者必须自己进行内存管理（例如，使用 malloc()、alloc()、free()、new 和 delete），如果操作不当可能会导致安全漏洞。更严重的问题是程序可能会错误地释放不应该释放的内存（例如，因为它已经被释放）。这可能导致程序立即崩溃或被利用，允许攻击者执行任意代码；参见[匿名 Phrack 2001]。一些系统（如许多 GNU/Linux 系统）默认根本不保护双重释放，而且不清楚那些试图保护自己的系统是否真正不可攻破。尽管我没有看到关于这个主题的任何书面材料，但我怀疑在 C++中使用错误的调用（例如，混合 new 和 malloc()）可能产生类似的效果。例如，2002 年 3 月 11 日，宣布 zlib 库存在这个问题，影响了使用它的许多程序。因此，在 GNU/Linux 上测试程序时，你应该将环境变量 MALLOC_CHECK_ 设置为 1 或 2，并且你可能考虑在设置该环境变量为 0、1、2 的情况下执行你的程序。这个变量的原因在 GNU/Linux malloc(3)手册页中解释如下：

> Linux libc（晚于 5.4.23）和 GNU libc（2.x）的最近版本包括一个可以通过环境变量调整的 malloc 实现。当 MALLOC_CHECK_ 被设置时，将使用一个特殊（效率较低）的实现，该实现旨在对简单错误具有容错性，例如具有相同参数的双重 free()调用，或单个字节的越界（off-by-one 错误）。然而，并非所有此类错误都可以得到保护，并且可能会导致内存泄漏。如果 MALLOC_CHECK_ 设置为 0，则任何检测到的堆损坏都将被静默忽略；如果设置为 1，则会在 stderr 上打印诊断信息；如果设置为 2，则立即调用 abort()。这可能是有用的，因为否则崩溃可能发生得晚得多，而真正的问题原因可能很难追踪。

有各种工具可以处理这个问题，例如 Electric Fence 和 Valgrind；有关更多信息，请参阅第 11.7 节。如果未使用的内存没有被释放（例如，使用 free()），那么这些未使用的内存可能会积累——如果足够的未使用内存积累，程序可能会停止工作。结果，未使用的内存可能被攻击者利用来创建拒绝服务。理论上，攻击者可能造成内存碎片化并导致拒绝服务，但通常这种攻击相当不切实际且风险较低。

在声明类型时尽可能严格。在可能的情况下，使用“enum”来定义枚举值（而不仅仅是具有特殊值的“char”或“int”）。这对于 switch 语句中的值特别有用，编译器可以用来确定是否覆盖了所有合法值。如果适当，如果值不能为负，则使用“unsigned”类型。

C 和 C++ 中一个复杂的问题是字符类型 “char” 可以是有符号或无符号的，这取决于编译器和机器；C 标准允许两者。当一个设置了高位的带符号 char 被保存到整数中时，结果将是一个负数；在某些情况下这可能被利用。通常，在处理可能具有大于 127（0x7f）值的字符数据时，使用 “unsigned char” 而不是 char 或 signed char 作为缓冲区、指针和类型转换。在编译时，尝试调用一个强制未指定的 "char" 为无符号的编译器选项。可移植程序不应依赖于 char 是否有符号，通过强制它为无符号，生成的可执行文件可以避免一些讨厌的安全漏洞。在 gcc 中，你可以使用 "-funsigned-char" 选项来实现这一点。

C 和 C++ 在类型检查支持方面定义上相当宽松，但你至少可以增加它们的检查级别，以便自动检测一些错误。尽可能打开编译器警告，并更改代码以干净地编译它们，并在单独的头文件 (.h) 中严格使用 ANSI 原型，以确保所有函数调用都使用正确的类型。对于使用 gcc 的 C 或 C++ 编译，至少使用以下作为编译标志（这些标志会打开大量警告消息）并尝试消除所有警告（注意，使用 -O2，因为一些警告只能在更高优化级别执行的数据流分析中检测到）：

```
 gcc -Wall -Wpointer-arith -Wstrict-prototypes -O2
```

IBM 的 Doc Shankar 建议在使用 gcc 时使用以下编译器选项集；使现有程序符合所有这些检查可能需要一些努力，但这些检查也可以帮助发现许多问题：

```
 gcc -Werror -Wall \
    -Wmissing-prototypes -Wmissing-declarations \
    -Wstrict-prototypes -Wpointer-arith \
    -Wwrite-strings -Wcast-qual -Wcast-align \
    -Wbad-function-cast \
    -Wformat-security  -Wformat-nonliteral \
    -Wmissing-format-attribute \
    -Winline
```

你可能还想使用 “-W -pedantic”。记住要添加 "-funsigned-char" 选项到这个集合中。

许多 C/C++ 编译器可以检测不准确的格式字符串。例如，如果你使用 gcc 的 __attribute__() 功能（一个 C 扩展）来标记这样的函数，并且你可以使用这个功能而不使你的代码不可移植，gcc 可以警告你创建的函数中不准确的格式字符串。以下是在你的头文件 (.h) 中可能包含的示例：

```
  /* in header.h */
 #ifndef __GNUC__
 #  define __attribute__(x) /*nothing*/
 #endif

 extern void logprintf(const char *format, ...)
    __attribute__((format(printf,1,2)));
 extern void logprintva(const char *format, va_list args)
    __attribute__((format(printf,1,0)));
```

"format" 属性可以是 "printf" 或 "scanf"，后面的数字分别是格式字符串的参数数和第一个可变参数（分别）。GNU 文档对此有很好的说明。注意，还有其他 __attribute__ 功能，例如 "noreturn" 和 "const"。

避免由 C/C++ 开发者犯的常见错误。使用警告系统和风格检查器可以帮助避免常见错误。例如，当你的意思是 “==” 时，要小心不要使用 “=”。上面推荐的 gcc 编译器的 -Wall 选项打开了 -Wparenthesis 选项。这个 -Wparenthesis 选项会在错误地使用 “=” 时警告你，并且如果你真的想使用 “=”，则需要添加额外的括号）。

一些组织已经定义了一个知名语言的子集，试图使其中的常见错误要么不可能发生，要么更加明显。C 的一个更知名的子集是 MISRA C 指南[MISRA 1998]。如果你打算使用子集，明智的做法是使用自动化工具来检查你是否实际上只使用了子集。有一个名为[Safer C](http://www.oakcomp.co.uk/SCT_About.html)的专有工具，它可以检查代码是否满足大多数 MISRA C 要求（它并不完全达到 100%，因为一些 MISRA C 要求难以自动检查）。

其他方法包括在语言中构建更多的安全检查，或者将语言本身改变为一个更容易编写安全程序的变体方言。我还没有使用过它们：安全 C 编译器（Safe C Compiler，简称 SCC）是一个 C 到 C 的编译器，它增加了扩展的指针和数组访问语义，以自动检测内存访问错误。它的[首页](http://www.cs.wisc.edu/~austin/scc.html)和[演讲](http://www.cs.wisc.edu/~austin/talk.scc)提供了有趣的信息，但截至 2004 年，其分发似乎有限。[Cyclone](http://www.research.att.com/projects/cyclone)是 C 的一个变体，它具有更多的“编译时、链接时和运行时检查，旨在确保安全性”（其中他们将安全性定义为无崩溃、缓冲区溢出、格式字符串攻击和一些其他问题）。此时，你实际上开始使用一种不同的（尽管相似）语言，在使用之前你应该仔细选择语言。

# 10.2. Perl

> [`dwheeler.com/secure-programs/Secure-Programs-HOWTO/perl.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/perl.html)

Perl 程序员首先应该阅读 man 页面的 perlsec(1)，它描述了在 Perl 中编写安全程序涉及的一些问题。特别是，perlsec(1)描述了“污染”模式，大多数安全的 Perl 程序都应该使用。如果真实用户或组 ID 与有效用户或组 ID 不同，或者你可以使用-T 命令行标志（如果你代表其他人运行，例如 CGI 脚本，请使用后者），则自动启用污染模式。污染模式会开启各种检查，例如检查路径目录以确保它们不会被其他人写入。

然而，taint 模式最明显的影响是，你可能会意外地使用来自程序外部的数据来影响程序外部的其他事物。在 taint 模式下，所有外部获取的输入都被标记为“污染”，包括命令行参数、环境变量、区域信息（参见 perllocale(1)）、某些系统调用的结果（readdir、readlink、getpw*调用的 gecos 字段）以及所有文件输入。污染数据不能直接或间接地用于任何调用子 shell 的命令，也不能用于任何修改文件、目录或进程的命令。有一个重要的例外：如果你向 system 或 exec 传递一个参数列表，该列表的元素不会被检查是否污染，所以在 taint 模式下使用 system 或 exec 时要格外小心。

从污染数据派生出的任何数据值也会变成污染的。对此有一个例外；取消数据污染的方法是提取污染数据的一个子串。不过，不要盲目地使用“.*”作为子串，因为这会破坏污染机制的防护。相反，确定由你的程序允许的“安全”模式，并使用它们来提取“良好”的值。提取值之后，你可能还需要检查它（特别是其长度）。

open、glob 和反引号函数调用 shell 来展开文件名通配符；这可能会被用来打开安全漏洞。你可以尝试完全避免使用这些函数，或者像在 perlsec(1)中描述的那样，在更受限制的“沙盒”中使用它们。特别是，反引号应该被重写为 system()调用（或者更好，完全改为更安全的东西）。

Perl 的 open()函数对于大多数安全程序来说，“魔法”太多了；它解释文本，如果不仔细过滤，可能会创建很多安全问题。在编写打开或锁定文件的代码之前，请查阅 perlopentut(1)手册页。在大多数情况下，sysopen()提供了打开文件的一种更安全（尽管更复杂）的方法。[新的 Perl 5.6 添加了一个带有 3 个参数的 open()调用，以关闭魔法行为，而不需要 sysopen()的复杂操作](http://www.xray.mpe.mpg.de/mailing-lists/perl5-porters/2000-03/msg02596.html)。

Perl 程序应该打开警告标志（-w），它会警告潜在危险或过时的语句。

你还可以在受限环境中运行 Perl 程序。更多信息请参阅标准 Perl 发行版中的“Safe”模块。我无法确定它经历了多少审计，所以请勿依赖它来保证安全。你还可以调查“Penguin Model for Secure Distributed Internet Scripting”，尽管在撰写本文时，代码和文档似乎不可用。

许多安装包括一个名为“suidperl”的 setuid root 版本的 perl。然而，perldelta 手册页面的 5.6.1 版本建议使用 sudo，如下所述：

> "请注意，在最近版本的 Perl 中，suidperl 既没有编译也没有默认安装。强烈不建议使用 suidperl。如果你认为你需要它，首先尝试使用 sudo 等替代方案。更多信息请参阅 http://www.courtesan.com/sudo/"。

# 10.3\. Python

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/python.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/python.html)

与任何语言一样，要小心任何允许数据作为程序一部分执行的功能，以确保不受信任的用户无法影响他们的输入。这包括 exec()、eval() 和 execfile()（坦白说，你应该仔细检查任何编译()的调用）。input() 语句也出奇地危险。[Watters 1996, 150]。

具有可由无权限用户调用的权限的 Python 程序（例如，setuid/setgid 程序）不得导入“用户”模块。用户模块会导致 pythonrc.py 文件被读取和执行。由于此文件将受不受信任的用户控制，导入用户模块允许攻击者强制受信任的程序运行任意代码。

Python 在编译时进行的检查非常少——例如，它几乎没有编译时类型信息。这是不幸的，导致了许多潜在的错误（我和 John Viega 都经历过这个问题）。希望有一天 Python 将实现可选的静态类型和类型检查，这是一个已经讨论了一段时间的想法。目前的部分解决方案是 PyChecker，这是一个类似 lint 的程序，用于检查 Python 源代码中的常见错误。您可以从 [`pychecker.sourceforge.net`](http://pychecker.sourceforge.net) 获取 PyChecker。

在 Python 2.3 版本之前，Python 通过其 RExec 和 Bastion 类支持“受限执行”。RExec 类主要用于执行小程序和移动代码，但它也可以用来尝试在代码未外部提供的情况下限制程序权限。Bastion 模块旨在支持对另一个对象的受限访问。更多信息，请参阅 Kuchling [2000]。本书的早期版本识别了这些函数，但将其标记为“程序员注意”，我确实有理由担心。最近的分析发现，RExec 和 Bastion 存在根本性的缺陷，并且有无法修复的安全漏洞。因此，这些类已被从 Python 2.3 中移除，不应在任何版本的 Python 中用于强制执行安全。目前正在进行开发替代方法来运行不受信任的 Python 代码，例如实验性的 [Sandbox.py 模块](http://www.procoders.net/download.php?fname=SandBox.py)。请勿将此实验性的 Sandbox.py 模块用于重要目的。

在 Python 中支持不受信任代码的安全执行证明是一个相当困难的问题。例如，允许用户无限制地向类添加属性允许以各种方式破坏环境，因为 Python 的实现调用了许多“隐藏”的方法。默认情况下，大多数 Python 对象是通过引用传递的；如果你将一个可变值的引用插入到受限程序的环境中，受限程序可以以在受限环境之外可见的方式更改对象。因此，如果你想要访问一个可变值，在许多情况下你应该复制这个可变值。从根本上说，Python 被设计成一个干净且高度自反的语言，这对于通用语言来说是好的，但使得处理恶意代码变得更加困难。

Python 支持名为“pickle”和“unpickling”的操作，以便方便地存储和检索对象集合。绝对不要从不受信任的源中反序列化数据。Python 2.2 尝试半心半意地支持从不受信任的源反序列化（通过 __safe_for_unpickling__ 属性），但它从未经过审计，可能从未真正工作过。Python 2.3 移除了所有这些，并明确指出反序列化不是一个安全操作。更多信息，请参阅[PEP 307](http://www.python.org/peps/pep-0307.html)。

# 10.4. Shell 脚本语言（sh 和 csh 派生）

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/shell.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/shell.html)

我强烈反对使用标准的命令行脚本语言（如 csh、sh 和 bash）来编写 setuid/setgid 安全代码。一些系统（如 Linux）完全禁用了 setuid/setgid 的 shell 脚本，因此创建 setuid/setgid 的 shell 脚本会带来不必要的可移植性问题。在一些旧系统中，由于竞争条件（如第 3.1.3 节中讨论的），它们本身就不安全。即使对于其他系统，这也不是一个好主意。

实际上，有许多情况下不应该在安全程序中使用 shell 脚本语言。标准的命令行外壳程序因其受非明显输入影响而臭名昭著——通常是因为命令行外壳程序是为了尝试为交互式用户“自动”执行事情而设计的，而不是为了防御有决心的攻击者。对于不需要安全性的程序（例如，它们以无特权的用户相同的权限运行且不接受“不受信任”的数据），shell 程序是合适的。只要所有输入（例如，文件、目录、命令行、环境等）都来自受信任的用户，它们也可以在启动/关闭脚本中使用，这也是为什么它们经常在启动/关闭脚本中非常成功的原因。

在存在恶意输入的情况下编写安全的 shell 程序比许多其他语言更困难，因为 shell 会受到许多因素的影响。例如，“隐藏”的环境变量（例如，ENV、BASH_ENV 和 IFS 的值）可能会影响它们的操作，甚至在脚本执行之前执行任意用户定义的代码。即使是可执行文件的文件名或目录内容也可能影响执行。如果攻击者可以创建包含一些控制字符（例如，换行符）、空白字符、shell 元字符或以短横线开头（选项标志语法）的文件名，通常有方法可以利用这些漏洞。例如，在许多 Bourne shell 实现中，执行以下操作将授予 root 访问权限（感谢 NCSA 描述了这个漏洞）：

```
  % ln -s /usr/bin/setuid-shell /tmp/-x
 % cd /tmp
 % -x
```

一些系统可能已经关闭了这个漏洞，但这个观点仍然成立：大多数命令 shell 并不是为了编写安全的 setuid/setgid 程序而设计的。出于编程目的，避免创建 setuid shell 脚本，即使在允许它们的系统上也是如此。相反，用另一种语言编写一个小程序来清理环境，然后让它调用其他可执行文件（其中一些可能是 shell 脚本）。

如果你仍然坚持使用 shell 脚本语言，至少将脚本放在一个无法移动或更改的目录中。在脚本开始时尽早设置 PATH 和 IFS 为已知值；实际上，应该在调用脚本之前清理环境。此外，尽早“cd”到一个安全的目录。仅使用受信任用户控制的目录中的数据，例如，/etc，这样攻击者就不能将这些目录中插入恶意命名的文件。确保在命令行上引用每个文件名，例如，使用"$1"而不是$1，因为包含空白的文件名将被分割。使用"--"调用命令以禁用可能的地方的附加选项，因为攻击者可能会创建或传递以短横线开头的文件名，希望欺骗程序将其作为选项处理。特别小心嵌入其他字符（例如，换行符和其他控制字符）的文件名。特别仔细检查输入文件名，并对允许的文件名非常严格。

如果您不介意将您的程序限制为仅与 GNU 工具一起工作（或者如果您检测到并可选地使用可用的 GNU 工具），您可能希望使用 NIL 字符作为文件名终止符而不是换行符。通过使用 NIL 字符，而不是空白或换行符，处理讨厌的文件名（例如，包含嵌入换行符的文件）要简单得多。几个输出或输入文件名的 GNU 工具可以使用这种格式而不是更常见的“每行一个文件名”格式。不幸的是，这个选项的名称在不同的工具之间并不一致；对于许多工具，这个选项的名称是“--null”或“-0”。GNU 程序 xargs 和 cpio 允许使用--null 或-0，tar 使用--null，find 使用-print0，grep 使用--null 或-Z，而 sort 使用-z 或--zero-terminated。那些发现这种不一致性特别令人烦恼的人被邀请向 GNU 作者提供补丁；我建议确保每个程序都支持“--null”，因为这似乎是最常见的选项名称。例如，以下是将文件移动到目标目录的一种方法，即使可能有大量文件，其中一些可能具有不规则的名称，包含嵌入的换行符（感谢 Jim Dennis 提醒我这一点）：

```
  find . -print0 &#124; xargs --null mv --target-dir=$TARG
```

同样地，我建议**不要**信任“受限外壳”来实现安全策略。受限外壳是有意阻止用户执行大量活动的外壳 - 它们的目标是强制用户只运行一小套程序。受限外壳可以作为多层次防御措施的一部分是有用的，但受限外壳在正确配置上非常困难，并且一旦配置好，通常容易被绕过。例如，一些受限外壳会首先以不受限制的模式运行一些文件（例如，“.profile”） - 如果用户可以更改此文件，他们可以强制执行该代码。受限外壳应该仅配置为运行少数几个程序，但如果其中任何程序都有“外壳逃逸”功能以允许用户运行更多程序，攻击者可以使用这些外壳逃逸来绕过受限外壳。即使程序没有外壳逃逸功能，也很可能各种程序可以一起使用（包括外壳的功能）来绕过限制。当然，如果您不设置受限外壳的 PATH（并允许运行任何程序），那么攻击者可以使用许多程序的外壳逃逸（包括文本编辑器、邮件程序等）。问题是外壳的目的是运行其他程序，但这些其他程序可能允许未预期的操作 -- 而外壳本身并不介入以阻止这些操作。

# 10.5\. Ada

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/ada.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/ada.html)

在 Ada95 中，Unbounded_String 类型通常比 String 类型更灵活，因为它会根据需要自动调整大小。然而，不要将特别敏感的秘密值（如密码或密钥）存储在 Unbounded_String 中，因为核心转储和页面区域可能会在以后仍然保留它们。相反，使用 String 类型来存储此类数据，在它被使用时将其锁定在内存中，并尽快用一些常数值（如 (others => ' ')）覆盖数据。在擦除内存后，使用 Ada 的 Inspection_Point 约束对包含秘密的对象进行操作。这样，你可以确信包含秘密的对象确实被删除了（并且覆盖不会被优化掉）。

与许多其他语言一样，Ada 的字符串类型（包括 String 和 Unbounded_String）可以存储 ASCII 0。如果将其传递给 C 库（包括内核），库可能会将其解释为与调用者意图完全不同。

对于初学 Ada 的程序员来说，认为 String 类型的第一个索引值总是 1 是很常见的，但如果字符串被截断，这就不正确了。避免这种错误。

值得注意的是，SPARK 是 Ada 编程语言的一个“高完整性子集”；SPARK 用户使用一个名为“SPARK Examiner”的工具来检查对 SPARK 规则的符合性，包括流程分析，如果需要的话，还有对代码的完整形式化证明的支持。[更多信息请参见 SPARK 网站](http://www.sparkada.com)。据我所知，目前没有开源的 SPARK 工具。如果你正在存储密码和私钥，如果适当的话，你应该仍然将它们锁定在内存中，并尽快覆盖它们。请注意，SPARK 通常用于不会发生分页的环境。

# 10.6\. Java

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/java.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/java.html)

如果你正在使用 Java 开发安全程序，坦白说，你的第一步（在学习 Java 之后）是阅读 Java 安全性的两本主要文献，即 Gong [1999] 和 McGraw [1999]（对于后者，特别关注第 7.1 节）。你还应该查看 Sun 发布的安全代码指南，网址为 [`java.sun.com/security/seccodeguide.html`](http://java.sun.com/security/seccodeguide.html)，以及 Sahu 等人 [2002] 的优秀文章 [http://www-106.ibm.com/developerworks/java/library/j-staticsec.html?loc=dwmain]。描述 Java 安全模型的幻灯片在 [`www.dwheeler.com/javasec`](http://www.dwheeler.com/javasec) 上免费提供。你还可以查看 McGraw [1998]。

显然，很大程度上取决于你正在开发的应用程序类型。打算在客户端使用的 Java 代码与服务器端代码的环境（和信任模型）完全不同。当然，一般原则适用；例如，你必须检查和过滤来自不受信任来源的任何输入。然而，在 Java 中，有一些“隐藏”的输入或潜在的输入，你需要小心，如下所述。Johnathan Nightingale [2000] 作出了一些有趣的陈述，总结了 Java 编程中的许多问题：

> ... 在 Java 编程中，最重要的是注意你的继承。如果你从父类、接口或父类的接口继承方法，你可能会打开你的代码的大门。

以下是一些关键指南，基于 Gong [1999]、McGraw [1999]、Sun 的指导以及我的个人经验：

1.  不要使用公共字段或变量；将它们声明为私有并提供访问器，以便您可以限制它们的可访问性。

1.  除非有充分的理由，否则不要将方法设置为私有（如果这样做，请记录原因）。这些非私有方法必须保护自己，因为它们可能会收到受污染的数据（除非你以某种方式安排了保护它们）。

1.  JVM 可能实际上不会在应用程序运行时强制执行可访问修饰符（例如，“私有”）（与 applet 相反）。感谢 John Steven（Cigital Inc.），他在 2000 年 11 月 7 日的“Secure Programming”邮件列表上指出了这一点。问题是这完全取决于请求访问的类所使用的类加载器。如果类是由可信的类加载器（包括 null/原初类加载器）加载的，则访问检查返回“TRUE”（允许访问）。例如，这可行（至少在 Sun 的 1.2.2 VM 中；它可能不适用于其他实现）：

    1.  编写一个受害者类（V），其中包含一个公共字段，然后编译它。

    1.  编写一个“攻击”类（A），它访问该字段，然后编译它

    1.  将 V 的公共字段改为私有，重新编译

    1.  运行 A - 它将访问 V 的（现在是私有的）字段。

    然而，applet 的情况不同。如果你将 A 转换为 applet 并以 applet 的形式运行它（例如，使用 appletviewer 或浏览器），它的类加载器就不再是可信的（或 null）类加载器。因此，代码将抛出 java.lang.IllegalAccessError，消息是你正在尝试从类 A 访问字段 V.secret。

1.  避免使用静态字段变量。此类变量附加到类（而非类实例）上，并且任何其他类都可以定位到类。因此，静态字段变量可以被任何其他类找到，这使得它们更难以保证安全性。

1.  永远不要向可能恶意代码返回可变对象（因为代码可能会决定更改它）。请注意，数组是可变的（即使数组内容不是），因此不要返回指向包含敏感数据的内部数组的引用。

1.  不要直接存储用户提供的可变对象（包括对象数组）。否则，用户可能会将对象交给安全代码，让安全代码“检查”该对象，并在安全代码尝试使用数据时更改数据。在内部保存之前，先克隆数组，并在此处小心（例如，警惕用户编写的克隆例程）。

1.  不要依赖于初始化。有几种方法可以分配未初始化的对象。

1.  除非有很好的理由不这样做，否则请将所有内容都设置为 final。如果一个类或方法不是 final，攻击者可能会尝试以危险和不可预见的方式扩展它。请注意，这会导致可扩展性的损失，以换取安全性。

1.  不要依赖于包作用域进行安全。一些类，如 java.lang，默认是封闭的，一些 Java 虚拟机（JVM）允许您关闭其他包。否则，Java 类不是封闭的。因此，攻击者可以在您的包内引入一个新的类，并使用这个新类来访问您认为您正在保护的东西。

1.  不要使用内部类。当内部类被转换为字节码时，内部类会被转换为对包中任何类都可见的类。更糟糕的是，封装类的私有字段会静默地变为非私有，以便内部类可以访问！

1.  最小化权限。在可能的情况下，不要要求任何特殊权限。McGraw 进一步建议不要签名任何代码；我说继续签名代码（这样用户可以决定“只运行此列表中的发送者的签名代码”），但尽量编写程序，使其不需要比沙盒权限集更多的权限。如果您必须拥有更多权限，请特别严格地审计该代码。

1.  如果您必须签名您的代码，请将其全部放入一个存档文件中。这里最好引用 McGraw [1999]的话：

    > 本规则的目标是防止攻击者执行混合匹配攻击，攻击者构建一个新的小程序或库，将一些您的签名类与恶意类链接在一起，或者将您从未打算一起使用的签名类链接在一起。通过将一组类一起签名，您使这种攻击更加困难。现有的代码签名系统在防止混合匹配攻击方面做得不够好，因此本规则不能完全防止此类攻击。但使用单个存档文件并不会造成伤害。

1.  使您的类不可克隆。Java 的对象克隆机制允许攻击者实例化一个类，而不运行其任何构造函数。要使您的类不可克隆，只需在每个类中定义以下方法：

    ```
 public final Object clone() throws java.lang.CloneNotSupportedException {
       throw new java.lang.CloneNotSupportedException();
       }
    ```  |
    | --- |

    如果您确实需要使您的类可克隆，那么您可以采取一些保护措施来防止攻击者重新定义您的克隆方法。如果您正在定义自己的克隆方法，只需将其设置为 final。如果您没有，您至少可以通过添加以下内容来防止克隆方法被恶意覆盖：

    ```
 public final void clone() throws java.lang.CloneNotSupportedException {
      super.clone();
      }
    ```  |
    | --- |

1.  使你的类不可反序列化。序列化允许攻击者查看你的对象的内部状态，甚至包括私有部分。为了防止这种情况，请向你的类中添加以下方法：

    ```
 private final void writeObject(ObjectOutputStream out)
      throws java.io.IOException {
         throw new java.io.IOException("Object cannot be serialized");
      }
    ```  |
    | --- |

    即使在序列化是可接受的情况下，也务必使用 transient 关键字来标记包含直接引用系统资源的信息以及包含与地址空间相关的信息的字段。否则，反序列化该类可能会允许不恰当的访问。你也许还希望将敏感信息标记为 transient。

    如果你为某个类定义了自己的序列化方法，那么它不应该将内部数组传递给任何接收数组的 DataInput/DataOutput 方法。理由是：所有 DataInput/DataOutput 方法都可以被覆盖。如果一个可序列化类直接将私有数组传递给 DataOutput（write(byte [] b)）方法，那么攻击者可以创建 ObjectOutputStream 的子类并覆盖 write(byte [] b)方法，从而能够访问和修改私有数组。请注意，默认的序列化不会将私有字节数组字段暴露给 DataInput/DataOutput 字节数组方法。

1.  使你的类不可反序列化。即使你的类不是可序列化的，它仍然可能是可反序列化的。攻击者可以创建一个字节序列，该序列恰好反序列化为攻击者选择的你的类的一个实例。换句话说，反序列化是一种公开的构造函数，允许攻击者选择对象的状态——这显然是一个危险的操作！为了防止这种情况，请向你的类中添加以下方法：

    ```
 private final void readObject(ObjectInputStream in)
      throws java.io.IOException {
        throw new java.io.IOException("Class cannot be deserialized");
      }
    ```  |
    | --- |

1.  不要通过名称比较类。毕竟，攻击者可以定义具有相同名称的类，如果你不小心，可能会通过授予这些类不希望的权限而造成混淆。因此，以下是一个*错误*的确定对象是否具有特定类的方法的例子：

    ```
   if (obj.getClass().getName().equals("Foo")) {
    ```  |
    | --- |

    如果你需要确定两个对象是否具有完全相同的类，请在两边都使用 getClass()，并使用==运算符进行比较。因此，你应该使用以下形式：

    ```
   if (a.getClass() == b.getClass()) {
    ```  |
    | --- |

    如果你确实需要确定一个对象是否具有特定的类名，你需要严谨并确保使用当前命名空间（当前类的 ClassLoader 的命名空间）。因此，你需要使用以下格式：

    ```
   if (obj.getClass() == this.getClassLoader().loadClass("Foo")) {
    ```  |
    | --- |

    这条指南来自 McGraw 和 Felten，它是一个很好的指南。我还要补充说，在可能的情况下，通常避免比较类值也是一个好主意。通常，尝试设计类方法和接口，以便根本不需要这样做会更好。然而，这并不总是实用的，因此了解这些技巧很重要。

1.  不要在代码或数据中存储秘密（加密密钥、密码或算法）。敌对的 JVM 可以快速查看这些数据。代码混淆并不能真正隐藏代码，以防止严重攻击者。

# 10.7\. Tcl

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/tcl.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/tcl.html)

Tcl 代表“工具命令语言”，发音为“tickle”。Tcl 分为两部分：一种语言和一个库。这种语言是一种简单的语言，最初旨在向交互式程序发出命令，并包括基本的编程能力。库可以嵌入到应用程序中。您可以在[Tcl.tk](http://www.tcl.tk/)和[Tcl WWW Info](http://www.sco.com/Technology/tcl/Tcl.html)网页以及[`www.tclfaq.wservice.com/tcl-faq`](http://www.tclfaq.wservice.com/tcl-faq)的 comp.lang.tcl FAQ 启动页面上找到更多关于 Tcl 的信息。我要感谢 Wojciech Kocjan 提供了一些关于在安全应用程序中使用 Tcl 的详细资料。

对于某些安全应用程序，Tcl 的一些特别有趣的组件包括 Safe-Tcl（在 Tcl 中创建沙盒）和 Safe-TK（为 Safe Tcl 实现沙盒化的便携式 GUI），以及允许 Tcl 包从全球互联网上的任何地方自动定位和加载的 WebWiseTclTk 工具包。您可以从[`www.cbl.ncsu.edu/software/WebWiseTclTk`](http://www.cbl.ncsu.edu/software/WebWiseTclTk)了解更多关于后者的信息。对我来说，不清楚它收到了多少代码审查。

Tcl 最初的设计目标是成为一个小型、简单的语言，这导致了一种最初有些限制和缓慢的语言。例如，在原始语言中的限制性弱点，请参阅[理查德·斯托尔曼的“为什么你不应该使用 Tcl”](http://sdg.lcs.mit.edu/~jchapin/6853-FT97/Papers/stallman-tcl.html)。例如，Tcl 最初设计为仅真正支持一种数据类型（字符串）。幸运的是，这些问题随着时间的推移得到了解决。特别是，版本 8.0 增加了对更多数据类型的支持（整数内部以整数存储，列表以列表存储等）。这提高了其能力，特别是提高了其速度。

与几乎所有脚本语言一样，Tcl 有一个“eval”命令，它可以解析并执行任意 Tcl 命令。并且像所有这样的脚本语言一样，这个 eval 命令需要特别小心地使用，否则攻击者可能会在输入中插入字符，导致恶意事件发生。例如，攻击者可能能够插入对 Tcl 有特殊意义的字符，如嵌入的空白（包括空格和新行）、双引号、花括号、方括号、美元符号、反斜杠、分号或井号（或创建输入以在处理过程中创建这些字符）。这也适用于任何将数据传递给 eval 的函数（取决于如何调用 eval）。

这里有一个小例子，可能会使这个概念更清晰；首先，让我们定义一个小函数，然后直接交互式地调用它——请注意，这些用法是可以的：

```
  proc something {a b c d e} {
       puts "A='$a'"
       puts "B='$b'"
       puts "C='$c'"
       puts "D='$d'"
       puts "E='$e'"
 }

 % # This works normally:
 % something "test 1" "test2" "t3" "t4" "t5"
 A='test 1'
 B='test2'
 C='t3'
 D='t4'
 E='t5'

 % # Imagine that str1 is set by an attacker:
 % set str1 {test 1 [puts HELLOWORLD]}

 % # This works as well
 % something $str1 t2 t3 t4 t5
 A='test 1 [puts HELLOWORLD]'
 B='t2'
 C='t3'
 D='t4'
 E='t5'
```

然而，继续这个例子，让我们看看"eval"如何被错误和正确地调用。如果你以错误（危险）的方式调用 eval，它允许攻击者滥用它。然而，通过使用 list 或 lrange 等命令正确地分组输入，你可以避免这个问题：

```
  % # This is the WRONG way - str1 is interpreted.
 % eval something $str1 t2 t3
 HELLOWORLD
 A='test'
 B='1'
 C=''
 D='t2'
 E='t3'

 % # Here&rsquo;s one solution, using "list".
 % eval something [list $str1 t2 t3 t4 t5]
 A='test 1 [puts HELLOWORLD]'
 B='t2'
 C='t3'
 D='t4'
 E='t5'

 % # Here&rsquo;s another solution, using lrange:
 % eval something [lrange $str1 0 end] t2
 A='test'
 B='1'
 C='[puts'
 D='HELLOWORLD]'
 E='t2'
```

使用 lrange 在连接调用函数的参数时很有用，例如，使用更复杂的库和回调。在 Tcl 中，eval 经常用于创建一个接受可变数量参数的函数的单参数版本，使用它时需要小心。以下是一个例子（假设你已经定义了一个"printf"函数）：

```
  proc vprintf {str arglist} {
      eval printf [list $str] [lrange $arglist 0 end]
 }

 % printf "1+1=%d  2+2=%d" 2 4
 % vprintf "1+1=%d  2+2=%d" {2 4}
```

基本上，当传递最终将被评估的命令时，你必须以正确构建的列表形式传递 Tcl 命令，而不是作为（可能连接的）字符串。例如，“after”命令在给定数量的毫秒后运行 Tcl 命令；如果$param1 中的数据可以被攻击者控制，则此 Tcl 代码是危险错误的：

```
   # DON'T DO THIS if param1 can be controlled by an attacker
  after 1000 "someCommand someparam $param1"
```

这是错误的，因为如果攻击者可以控制$param1 的值，攻击者就可以控制程序。例如，如果攻击者可以导致$param1 具有“[exit]”，则程序将退出。此外，如果$param1 是“;exit”，它也会退出。

因此，合适的替代方案将是：

```
  after 1000 [list someCommand someparam $param1]
```

更好的方法可能是以下这样：

```
  set cmd [list someCommand someparam]
 after 1000 [concat $cmd $param1]
```

这里还有一个例子，展示了你不应该做什么，假设$params 是可能恶意用户控制的数据：

```
  set params "%-20s TESTSTRING"
 puts "'[eval format $params]'"
```

将导致：

```
  'TESTSTRING       '
```

但是，当不受信任的用户发送包含嵌入换行符的数据时，就像这样：

```
  set params "%-20s TESTSTRING\nputs HELLOWORLD"
 puts "'[eval format $params]'"
```

结果将是这样的（注意攻击者的代码被执行了！）：

```
  HELLOWORLD
 'TESTINGSTRING       '
```

Wojciech Kocjan 建议在这种情况下最简单的解决方案是将此转换为列表，使用 lrange 这样做：

```
  set params "%-20s TESTINGSTRING\nputs HELLOWORLD"
 puts "'[eval format [lrange $params 0 end]]'"
```

结果将是：

```
  'TESTINGSTRING       '
```

注意，这个解决方案假设潜在的恶意文本被连接到文本的末尾；就像所有语言一样，确保攻击者不能控制文本格式。

作为一种风格，始终在 if、while、for、expr 和任何其他使用 expr/eval/subst 解析参数的命令中使用花括号。这样做将避免使用 Tcl 时常见的错误，即意外的双重替换（也称为双重替换）。这最好通过例子来解释；以下代码是错误的：

```
  while ![eof $file] {
     set line [gets $file]
 }
```

代码错误是因为在第一次执行 while 命令时，“![eof $file]”文本将被 Tcl 解析器评估，而不是在每次迭代中重新评估，正如它应该做的那样。相反，你应该这样做：

```
  while {![eof $file]} {
      set line [gets $file]
 }
```

注意，条件和要执行的操作都被大括号包围。尽管在某些情况下大括号是多余的，但它们并不会造成伤害，而且当您忘记在需要的地方包含大括号时（比如，在做一些小的更改时），往往会发生微妙且难以发现的错误。

关于良好的 Tcl 风格的更多信息可以在诸如 [Ray Johnson 的 Tcl 风格指南](http://www.tcl.tk/doc/styleGuide.pdf) 等文档中找到。

在过去，我曾表示不推荐使用 Tcl 编写必须处理安全边界的程序。自那时起，Tcl 似乎有所改进，因此虽然我不能保证 Tcl 会满足您的需求，但我也不能保证其他任何语言会满足您的需求。再次感谢 Wojciech Kocjan，他提供了一些关于如何编写安全应用中 Tcl 代码的建议。

# 10.8\. PHP

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/php.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/php.html)

SecureReality 发布了一篇非常有趣的论文，题为“血色研究 - PHP 中常见漏洞的利用” [Clowes 2001]，讨论了在 PHP 中编写安全程序的一些问题，尤其是在 PHP 4.1.0 之前的版本中。Clowes 总结说：“即使在默认配置下尝试编写安全的 PHP 应用程序（在 PHP 的默认配置中），也是非常困难的”。

当然，任何语言都存在安全问题，但 PHP 的一个特定问题在旧版本中尤为突出，这使旧版本的 PHP 相比大多数语言来说安全性更低：它将数据加载到其命名空间的方式。默认情况下，在 PHP（版本 4.1.0 及以下）中，所有环境变量和通过网络发送到 PHP 的值都会自动加载到与普通变量相同的命名空间（全局变量）中——因此攻击者可以设置任意变量为任意值，除非 PHP 程序明确重置，这些值才会保持不变。此外，PHP 在首次请求时自动创建具有默认值的变量，因此 PHP 程序不初始化变量是很常见的。如果您忘记设置变量，PHP 可以报告它，但默认情况下 PHP 不会——请注意，这只是一个错误报告，它不会阻止找到异常方法来引发它的攻击者。因此，默认情况下，PHP 允许攻击者完全控制程序中所有变量的值，除非程序特别小心地覆盖攻击者。一旦程序接管，它可以重置这些变量，但未能重置任何变量（即使是那些不明显变量）可能会在 PHP 程序中打开漏洞。

例如，以下 PHP 程序（来自 Clowes 的示例）旨在仅允许知道密码的人获取一些重要信息，但攻击者可以在他们的网络浏览器中设置“auth”并绕过授权检查：

```
  <?php
  if ($pass == "hello")
   $auth = 1;
  ...
  if ($auth == 1)
   echo "some important information";
 ?>
```

我和许多其他人都对这个问题表示了特别的担忧；这尤其是一个问题，因为 PHP 被广泛使用。毕竟，一种本应易于使用的语言应该使得编写安全的程序变得容易。通过将设置“register_globals”设置为“off”，可以在 PHP 中禁用这个错误特性，但默认情况下，PHP 版本直到 4.1.0 都将此设置为“on”，而 4.1.0 之前的 PHP 版本在关闭 register_globals 时更难使用。PHP 开发者在 PHP 4.1.0 的公告中警告说：“从 PHP 的下一个半主要版本开始，PHP 的新安装将默认将 register_globals 设置为 off。”现在这已经发生了；从 PHP 版本 4.2.0 开始，默认情况下不再在全局范围内注册外部变量（来自环境、HTTP 请求、cookie 或 Web 服务器）。访问这些外部变量的首选方法是使用在 PHP 4.1.0 中引入的新超级全局数组。

将“register_globals”设置为“on”的 PHP 对于非平凡程序来说是一个危险的选择——编写不安全的程序真是太容易了。然而，一旦将“register_globals”设置为“off”，PHP 就成为一个相当合理的开发语言。

安全的默认设置应包括将“register_globals”设置为“off”，并包括几个函数，使用户能够更容易地指定和限制他们从外部来源接受的输入。然后，Web 服务器（如 Apache）可以分别配置这个安全的 PHP 安装。可以将例程放入 PHP 库中，使用户能够列出他们想要接受的输入变量；一些函数可以检查这些变量必须具有的模式和/或变量必须被强制转换的类型。在我看来，如果你将 register_globals 设置为 on，PHP 不是一个安全的 Web 开发选择。

正如我在本书的早期版本中所建议的，PHP 已经被修改，成为安全 Web 开发的合理选择。然而，请注意，PHP 在安全漏洞方面的记录并不特别出色（例如，register_globals、文件上传问题以及错误报告库中的格式字符串问题）；我相信在 PHP 的早期版本中，安全问题没有得到足够的考虑；我也认为 PHP 开发者现在正在强调安全性，并且这些安全问题最终得到了解决。一个证据是 PHP 开发者对关闭 register_globals 所做的重大改变；这对 PHP 用户产生了重大影响，他们愿意做出这个改变是一个好兆头。不幸的是，目前尚不清楚 PHP 实际上有多安全；PHP 的开发者现在才开始认真检查安全问题，PHP 的记录还没有太多。希望这会很快变得清楚。

如果你已经决定使用 PHP，以下是我的一些推荐（许多这些建议是基于解决 Clowes 提出的问题的方法）：

+   将 PHP 配置选项“register_globals”关闭，并使用 PHP 4.2.0 或更高版本。PHP 4.1.0 添加了几个特殊数组，特别是 $_REQUEST，这使得在关闭“register_globals”时开发 PHP 软件变得简单得多。关闭 register_globals（PHP 4.2.0 中的默认设置）完全消除了最常见的 PHP 攻击。如果你假设 register_globals 已关闭，你应该首先检查这一点（如果实际情况不是这样，则停止操作）——这样，安装你的程序的人会很快知道存在问题。请注意，许多第三方 PHP 应用程序无法使用此设置，因此可能很难在整个网站上保持关闭状态。对于某些程序，可以仅关闭 register_globals。例如，对于 Apache，你可以在 PHP 目录中的 .htaccess 文件中插入以下行（或使用 Directory 指令进一步控制）：

    ```
  php_flag register_globals Off
     php_flag track_vars On
    ```  |
    | --- |

    然而，除非 Apache 网络服务器配置为允许覆盖，否则 .htaccess 文件本身会被忽略；通常 Apache 的全局配置设置为将 AllowOverride 设置为 None。因此，对于 Apache 用户来说，如果你能说服你的网络托管服务在你的主机配置文件（通常是 /etc/http/conf/http.conf）中设置“AllowOverride Options”，那么请这样做。然后编写辅助函数以简化加载数据（仅加载数据）。

+   如果你必须开发在运行时可能开启 register_globals 的软件（例如，广泛部署的 PHP 应用程序），始终设置用户未提供的值。不要依赖 PHP 默认值，也不要信任任何你没有明确设置的变量。请注意，你必须为 *每个* 入口点（例如，每个 PHP 程序或使用 PHP 的 HTML 文件）这样做。最佳做法是在每个 PHP 程序开始时设置所有你将使用的变量，即使你只是将它们重置为通常的默认值（如 "" 或 0）。这包括在包含文件中引用的全局变量，甚至所有库，间接地。不幸的是，这使得这个建议难以实施，因为很少有开发者真正了解并理解他们可能调用的所有函数中可能使用的所有全局变量。一个较小的替代方案是搜索 HTTP_GET_VARS、HTTP_POST_VARS、HTTP_COOKIE_VARS 和 HTTP_POST_FILES，以查看用户是否提供了数据——但程序员经常忘记检查所有来源，如果 PHP 添加了新的数据源（例如，HTTP_POST_FILES 在 PHP 的旧版本中并不存在），会发生什么。当然，这仅仅告诉你如何最大限度地利用一个糟糕的情况；如果你还没有注意到，请关闭 register_globals！

+   将错误报告级别设置为 E_ALL，并在测试期间解决它报告的所有错误。除此之外，这还会对未初始化的变量提出警告，这是 PHP 中的关键问题。无论如何，当你开始使用 PHP 时，这是一个好主意，因为这也有助于调试程序。设置错误报告级别有许多方法，包括在“php.ini”文件（全局）、“.htttpd.conf”文件（单主机）、“.htaccess”文件（多主机）中，或者在脚本顶部通过 error_reporting 函数。我建议在 php.ini 文件和脚本顶部都设置错误报告级别；这样，如果你（1）忘记在脚本顶部插入命令，或者（2）将程序移到另一台机器并忘记更改 php.ini 文件，你都会受到保护。因此，每个 PHP 程序都应该像这样开始：

    ```
   <?php error_reporting(E_ALL);?>
    ```  |
    | --- |

    可以争论说，这种错误报告应该在开发期间开启，但在实际运行在真实网站上时关闭（因为这样的错误信息可能会给攻击者提供有用的信息）。问题是，如果它们在“实际使用”期间被禁用，那么在开发期间也容易忘记启用它们。所以，目前我建议简单的方法，就是将其包含在每一个入口点。一个更好的方法是记录所有错误，但将错误报告直接导向日志文件（而不是将它们报告给攻击者）。

+   仔细过滤用于创建文件名的任何用户信息，特别是为了防止远程文件访问。PHP 默认自带“远程文件”功能——这意味着像 fopen() 这样的文件打开命令，在其他语言中只能打开本地文件，但实际上可以用来从另一个网站发起网络或 ftp 请求。

+   不要使用旧式的 PHP 文件上传；使用 HTTP_POST_FILES 数组和相关函数。PHP 通过将文件上传到某个临时目录并使用特殊文件名来支持文件上传。PHP 最初设置了一系列变量来指示该文件名所在的位置，但由于攻击者可以控制变量名及其值，攻击者可以利用这一能力造成很大的麻烦。相反，始终使用 HTTP_POST_FILES 和相关函数来访问上传的文件。请注意，即使在这种情况下，PHP 的方法也允许攻击者暂时上传任意内容的文件到你的服务器，这本身就具有风险。

+   只在文档树中放置受保护的入口点；将所有其他代码（应该是大部分）放置在文档树之外。PHP 在这方面有不幸的历史建议。最初，PHP 用户应该使用“ .inc”（包含）扩展名来为“包含”文件，但这些包含文件通常包含密码和其他信息，当 Apache 在文档树中请求时，会直接将“ .inc”文件的内容提供给请求者。然后开发者给所有文件都添加了“ .php”扩展名——这意味着内容不会被看到，但现在原本不是入口点的文件变成了入口点，有时可能被利用。如前所述，通常的安全建议是最好的：只在文档树中放置受保护的入口点（文件），并将其他代码（例如库）放置在文档树之外。文档树中根本不应该有任何“ .inc”文件。

+   避免使用会话机制。虽然“会话”机制对于存储持久数据来说很方便，但其当前实现存在许多问题。首先，默认情况下会话将信息存储在临时文件中——因此，如果你在一个多主机系统上，你将使自己容易受到许多攻击和泄露。即使那些目前没有多主机的人也可能在将来发现自己处于多主机状态！你可以将此信息“绑定”到数据库而不是文件系统，但如果其他多主机数据库的用户可以以相同的权限访问该数据库，问题仍然是相同的。如果不小心，也会存在一些歧义（“这是会话值还是攻击者的值？”），而且这也是攻击者可以强制将文件或密钥放置在服务器上，并包含他们选择的内容的另一个案例——这是一个危险的情况——攻击者甚至可以控制到一定程度将此数据放置的文件或密钥的名称。

+   使用指令来限制权限（例如 safe_mode、disable_function 和 open_basedir），但不要过分依赖它们。这些指令可以帮助限制一些简单的偶然攻击，因此它们值得应用。然而，它们可能不足以保护免受真实攻击；它们仅依赖于用户空间 PHP 程序进行保护，而该程序并不是真正设计来执行此操作的。相反，你应该使用操作系统保护（例如，运行单独的进程和用户）来进行严肃的保护。

+   对于所有输入，检查它们是否匹配可接受的模式（就像任何语言一样），然后使用类型转换将非字符串数据强制转换为应有的类型。开发“辅助”函数以轻松检查和导入所选的（预期）输入列表。PHP 是弱类型语言，这可能会引起麻烦。例如，如果输入数据有值为"000"，它不会等于"0"也不会是空()。这对于关联数组尤其重要，因为它们的索引是字符串；这意味着$data["000"]与$data["0"]不同。例如，为了确保$bar 具有 double 类型（在确保它只有 double 合法格式之后）：

    ```
   $bar = (double) $bar; 
    ```  |
    | --- |

+   小心任何将 PHP 代码作为字符串执行的函数 - 确保攻击者不能控制字符串内容。这包括 eval()、exec()、include()、passthru()、popen()、preg_replace()（当使用/e 修饰符时）、require()、system()和反引号操作符。

+   特别小心有风险的函数。例如，这包括打开文件的函数（例如，fopen()、readfile() 和 file()）；确保攻击者不能强迫程序打开任意文件。PHP 的旧版本（4.3.0 之前）在 wordwrap()函数中存在缓冲区溢出漏洞，所以如果你使用旧版本要小心（或者更好的做法是升级，并确保你的客户通过检查安装程序中的版本号来升级）。

+   在适当的地方使用 magic_quotes_gpc() - 这消除了许多种类的攻击。

+   避免文件上传，并考虑修改 php.ini 文件来禁用它们（file_uploads = Off）。文件上传过去存在安全漏洞，所以在较旧的 PHP 版本中这是必需的，并且直到有更多经验表明它们是安全的，这并不是一个坏主意来移除。记住，一般来说，为了确保系统安全，你应该禁用或移除你不需要的任何东西。
