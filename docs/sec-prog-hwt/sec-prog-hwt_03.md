在讨论如何使用 Linux 或 Unix 安全特性的指南之前，了解从程序员的角度来看这些特性是什么非常有用。本节简要描述了在几乎所有 Unix-like 系统上都广泛可用的特性。然而，请注意，不同版本的 Unix-like 系统之间存在相当大的差异，并非所有系统都具有这里描述的能力。本章还指出了一些 Linux 特定的扩展或特性；Linux 发行版在安全编程方面往往相当相似，因为它们都使用基本上相同的内核和 C 库（以及基于 GPL 的许可证鼓励任何创新的快速传播）。它还指出了一些不同 Unix 实现之间的安全相关差异，但请注意，这不是一个详尽的列表。本章不讨论诸如强制访问控制 (MAC) 的实现等问题，许多 Unix-like 系统并不实现这些功能。如果您已经知道这些特性是什么，请随时跳过本节。

许多编程指南只是简要地浏览了 Linux 或 Unix 中的安全相关部分，并跳过了重要信息。特别是，它们通常以一般性术语讨论“如何使用”某物，但忽略了影响其使用的安全属性。相反，手册页中关于个别函数的详细信息非常多，但手册页有时会通过详细讨论如何使用每个个别函数来掩盖关键的安全问题。本节试图弥合这一差距；它概述了 Linux 中程序员可能使用的安全机制，但专注于安全影响。本节比典型的编程指南更深入，专注于与安全相关的问题，并指向可以获取更多详细信息的参考资料。

首先，基础概念。Linux 和 Unix 在本质上分为两部分：内核和“用户空间”。大多数程序都在用户空间（在内核之上）执行。Linux 支持所谓的“内核模块”概念，这仅仅是动态将代码加载到内核中的能力，但请注意，它仍然有这种基本的划分。一些其他系统（如 HURD）是基于“微内核”的系统；它们有一个小型内核，功能更有限，并有一组“用户”程序，这些程序实现了内核传统上实现的底层功能。

一些类 Unix 系统已经进行了大量修改以支持强大的安全性，特别是为了支持美国国防部对强制访问控制（B1 级或更高）的要求。本书的这一版本不涵盖这些系统或问题；我希望能在未来版本中扩展到这一点。关于其中的一些系统，更详细的信息在其他地方可用，例如，SGI 的“可信 IRIX/B”的详细信息可在 NSA 的[最终评估报告（FERs）](http://www.radium.ncsc.mil/tpep/library/fers/index.html)中找到。

当用户登录时，他们的用户名会被映射到表示他们“UID”（即用户 ID）和“GID”（即组 ID）的整数上，这些 ID 代表他们所属的组。UID 0 是一个特殊的特权用户（角色），传统上被称为“root”；在大多数类 Unix 系统中（包括标准的 Linux 内核），root 可以绕过大多数安全检查，并用于系统管理。在某些 Unix 系统中，GID 0 也是特殊的，允许对组级别的资源进行无限制访问 [Gay 2000, 228]；在其他系统（如 Linux）中则不是这样，但即使在那些系统中，组 0 基本上是全能的，因为许多特殊的系统文件都属于组 0。进程是安全意义上的唯一“主体”（即，只有进程是活动的对象）。进程可以访问各种数据对象，特别是文件系统对象（FSOs）、System V 进程间通信（IPC）对象和网络端口。进程还可以设置信号。其他与安全相关的主题包括配额和限制、库、审计和 PAM。接下来的几个小节将详细说明这一点。

# 3.1\. 进程

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/processes.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/processes.html)

在类 Unix 系统中，用户级活动是通过运行进程来实现的。大多数 Unix 系统支持“线程”作为一个独立的概念；线程在进程内部共享内存，系统调度器实际上调度线程。Linux 的做法不同（在我看来，这是一种更好的方法）：线程和进程之间没有本质的区别。相反，在 Linux 中，当进程创建另一个进程时，它可以选择共享哪些资源（例如，内存可以共享）。然后 Linux 内核执行优化以获得线程级别的速度；更多信息请参见 clone(2)。值得注意的是，Linux 内核开发者倾向于使用“任务”这个词，而不是“线程”或“进程”，但外部文档倾向于使用“进程”这个词（所以在这里我将使用“进程”这个词）。当编写多线程应用程序时，通常最好使用标准线程库之一，这些库隐藏了这些差异。这不仅使线程更加可移植，而且一些库通过将多个应用级线程实现为单个操作系统线程，提供了一层额外的间接层；这可以在某些系统上为某些应用程序提供一些性能提升。

## 3.1.1\. 进程属性

下面是 Unix-like 系统中每个进程的典型属性：

+   RUID, RGID - 进程运行用户的实际 UID 和 GID

+   EUID, EGID - 用于权限检查的有效 UID 和 GID（除了文件系统）

+   SUID、SGID - 保存的 UID 和 GID；用于支持下面讨论的“开/关”权限切换。并非所有类 Unix 系统都支持此功能，但绝大多数都支持（包括 Linux 和 Solaris）；如果你想检查给定系统是否在 POSIX 标准中实现了此选项，可以使用 sysconf(2) 来确定 _POSIX_SAVED_IDS 是否生效。

+   补充组 - 该用户所属的组（GIDs）列表。在原始的版本 7 Unix 中，这不存在 - 进程一次只能属于一个组，必须执行特殊命令来更改该组。BSD 为每个进程添加了对组列表的支持，这更加灵活，现在这种添加已被广泛实现（包括 Linux 和 Solaris）。

+   umask - 一组位，用于确定创建新的文件系统对象时的默认访问控制设置；请参阅 umask(2)。

+   调度参数 - 每个进程都有一个调度策略，默认策略 SCHED_OTHER 的进程有额外的参数 nice、优先级和计数器。有关更多信息，请参阅 sched_setscheduler(2)。

+   限制 - 每个进程的资源限制（见下文）。

+   文件系统根 - 进程认为根文件系统（“/”）开始的地方；请参阅 chroot(2)。

这里是与进程关联的较少见的属性：

+   FSUID、FSGID - 用于文件系统访问检查的 UID 和 GID；这通常等于 EUID 和 EGID。这是一个 Linux 特有的属性。

+   能力 - POSIX 能力信息；一个进程实际上有三组能力：有效能力、可继承能力和允许能力。有关 POSIX 能力的更多信息，请参阅下文。Linux 内核版本 2.2 及以上支持此功能；一些其他类 Unix 系统也支持，但并不像 Linux 那样普遍。

在 Linux 中，如果你真的需要确切地知道每个进程关联了哪些属性，最权威的来源是 Linux 源代码，特别是 `/usr/include/linux/sched.h` 中对 task_struct 的定义。

创建新进程的可移植方法是使用 fork(2) 调用。BSD 引入了一个名为 vfork(2) 的变体作为优化技术。vfork(2) 的底线很简单：如果可能，不要使用它。有关更多信息，请参阅第 8.6 节。

Linux 支持 Linux 特有的 clone(2) 调用。这个调用与 fork(2) 类似，但允许指定哪些资源应该被共享（例如，内存、文件描述符等）。各种 BSD 系统实现了 rfork() 系统调用（最初在 Plan9 中开发）；它有不同的语义，但基本思想相同（它也创建了一个对共享有更严格控制的进程）。如果可能，可移植的程序不应直接使用这些调用；如前所述，它们应依赖于使用此类调用实现线程的线程库。

本书不是关于编写程序的完整教程，因此我将跳过广泛的信息处理过程。有关 wait(2)、exit(2)等更多信息，请参阅文档。

## 3.1.2\. POSIX 能力

POSIX 能力是一组允许将通常由 root 持有的权限分割成更大、更具体权限集的位。POSIX 能力由一个 IEEE 标准草案定义；它们不仅限于 Linux，但也不是所有类 Unix 系统都普遍支持。Linux 内核 2.0 不支持 POSIX 能力，而版本 2.2 为进程添加了对 POSIX 能力的支持。当 Linux 文档（包括这一篇）说“需要 root 权限”时，在几乎所有情况下它实际上意味着“需要能力”，正如能力文档中所记录的那样。如果您需要知道所需的具体能力，请在能力文档中查找。

在 Linux 中，最终目的是允许将能力附加到文件系统中的文件；然而，截至本文撰写时，这尚未得到支持。虽然支持传输能力，但默认情况下是禁用的。Linux 版本 2.2.11 增加了一个使能力更直接有用的功能，称为“能力边界集”。能力边界集是允许系统上任何进程持有的能力列表（否则，只有特殊的 init 进程可以持有它）。如果一个能力没有出现在边界集中，那么无论权限有多高，任何进程都不能使用它。这个功能可以用来，例如，禁用内核模块加载。利用这一功能的示例工具是 LCAP，位于[`pweb.netcom.com/~spoon/lcap/`](http://pweb.netcom.com/~spoon/lcap/)。

更多关于 POSIX 能力的信息可在 ftp://linux.kernel.org/pub/linux/libs/security/linux-privs 找到。

## 3.1.3\. 进程创建和管理

可以使用 fork(2)、不建议使用的 vfork(2)或 Linux 特有的 clone(2)创建进程；所有这些系统调用都会复制现有进程，从中创建两个进程。一个进程可以通过调用 execve(2)或其各种前端（例如，参见 exec(3)、system(3)和 popen(3)）来执行不同的程序。

当一个程序被执行，并且其文件设置了 setuid 或 setgid 位时，进程的 EUID 或 EGID（分别）通常会被设置为文件值。当用于支持 setuid 或 setgid 脚本时，这种功能曾是旧 Unix 安全弱点的来源，因为存在竞争条件。在内核打开文件以查看要运行的解释器之间，以及（现在已设置 ID 的）解释器转身重新打开文件以解释它之间，攻击者可能会更改文件（直接或通过符号链接）。

不同的类 Unix 系统以不同的方式处理 setuid 脚本的安全问题。一些系统，如 Linux，在执行脚本时完全忽略 setuid 和 setgid 位，这显然是一种安全的方法。大多数现代的 SysVr4 和 BSD 4.4 版本使用不同的方法来避免内核竞争条件。在这些系统中，当内核将 set-id 脚本的名称传递给解释器时，而不是使用路径名（这会允许竞争条件），它反而传递了文件名/dev/fd/3。这是一个已经打开在脚本上的特殊文件，这样攻击者就不能利用竞争条件。即使在这些系统上，我也建议不要使用 setuid/setgid shell 脚本语言来编写安全程序，如下文所述。

在某些情况下，进程可以影响各种 UID 和 GID 值；请参阅 setuid(2)、seteuid(2)、setreuid(2)和 Linux 特有的 setfsuid(2)。特别是保存的用户 ID（SUID）属性是为了允许受信任的程序临时切换 UID。支持 SUID 的类 Unix 系统使用以下规则：如果 RUID 改变，或者 EUID 设置为不等于 RUID 的值，则 SUID 设置为新的 EUID。无特权的用户可以从他们的 SUID 设置 EUID，将 RUID 设置为 EUID，并将 EUID 设置为 RUID。

Linux 特有的 FSUID 进程属性旨在允许像 NFS 服务器这样的程序仅限于某些给定 UID 的文件系统权限，而不给该 UID 发送信号到进程的权限。每当 EUID 改变时，FSUID 就会改变为新 EUID 值；FSUID 值可以使用 setfsuid(2)，一个 Linux 特有的调用，单独设置。请注意，非 root 调用者只能将 FSUID 设置为当前的 RUID、EUID、SEUID 或当前的 FSUID 值。

# 3.2\. 文件

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/files.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/files.html)

在所有类 Unix 系统中，信息的主要存储库是文件树，以“/”为根。文件树是一组目录的分层集合，每个目录都可能包含文件系统对象（FSOs）。

在 Linux 中，文件系统对象（FSOs）可以是普通文件、目录、符号链接、命名管道（也称为先进先出或 FIFOs）、套接字（见下文）、字符特殊（设备）文件或块特殊（设备）文件（在 Linux 中，此列表在 find(1)命令中给出）。其他类 Unix 系统有一个相同或类似的 FSO 类型列表。

文件系统对象收集在文件系统上，这些文件系统可以在文件树中的目录上挂载和卸载。文件系统类型（例如，ext2 和 FAT）是一组特定的约定，用于在磁盘上安排数据以优化速度、可靠性等；许多人使用“文件系统”一词作为文件系统类型的同义词。

## 3.2.1\. 文件系统对象属性

不同的类 Unix 系统支持不同的文件系统类型。文件系统可能具有略微不同的访问控制属性，并且访问控制可能会受到在挂载时选择的选项的影响。在 Linux 中，ext2 文件系统目前是最受欢迎的文件系统，但 Linux 支持大量的文件系统。大多数类 Unix 系统也倾向于支持多个文件系统。

大多数类 Unix 系统的文件系统至少存储以下内容：

+   拥有者 UID 和 GID - 识别文件系统对象的“所有者”。除非另有说明，否则只有所有者或 root 可以更改访问控制属性。

+   权限位 - 对于用户（所有者）、组和其他的读取、写入、执行位。对于普通文件，读取、写入和执行具有它们典型的含义。在目录中，“读取”权限是显示目录内容所必需的，而“执行”权限有时被称为“搜索”权限，并且是实际进入目录以使用其内容所必需的。在目录中，对目录的“写入”权限允许添加、删除和重命名该目录中的文件；如果您只想允许添加，请设置下面提到的粘性位。请注意，符号链接的权限值永远不会使用；只有它们的包含目录和链接到的文件的值才是重要的。

+   “粘性位” - 当设置在目录上时，该目录中文件的解除链接（删除）和重命名操作仅限于文件所有者、目录所有者或 root 权限。这是一个非常常见的 Unix 扩展，并在 Open Group 的单一 Unix 规范版本 2 中指定。旧版本的 Unix 称此为“保存程序文本位”，并使用此位来指示应保留在内存中的可执行文件。执行此操作的系统确保只有 root 可以设置此位（否则用户可以通过将“一切”强制进入内存来使系统崩溃）。在 Linux 中，此位对普通文件没有影响，普通用户可以修改他们拥有的文件上的此位：Linux 的虚拟内存管理使这种旧用途变得无关紧要。

+   setuid, setgid - 当设置在可执行文件上时，执行文件将设置进程的有效 UID 或有效 GID 为文件所有者的 UID 或 GID（分别）。所有类 Unix 系统都支持这一点。在 Linux 和 System V 系统中，当 setgid 设置在没有任何执行权限的文件上时，这表示在访问期间将受到强制锁定的文件（如果文件系统已挂载以支持强制锁定）；这种意义的重叠让许多人感到惊讶，并且并非在所有类 Unix 系统中都是通用的。实际上，Open Group 的 Single Unix 规范版本 2 的 chmod(3)允许系统忽略对非可执行文件启用 setgid 的请求，如果这种设置没有意义。在 Linux 和 Solaris 中，当 setgid 设置在目录上时，在该目录中创建的文件将自动将 GID 重置为目录的 GID。这种方法的目的是支持“项目目录”：用户可以将文件保存到这种特别设置的目录中，并且组所有者会自动更改。然而，在目录上设置 setgid 位并未由如 Single Unix 规范[Open Group 1997]之类的标准指定。

+   时间戳 - 每个文件系统对象都存储了访问和修改时间。然而，所有者可以任意设置这些值（参见 touch(1)），因此在使用这些信息时要小心。所有类 Unix 系统都支持这一点。

以下属性是 Linux 特有的 ext2 文件系统扩展，尽管许多其他文件系统具有类似的功能：

+   不可变位 - 不允许对文件系统对象进行更改；只有 root 可以设置或清除此位。这仅由 ext2 支持，并且不能跨所有 Unix 系统（或甚至所有 Linux 文件系统）移植。

+   只追加位 - 只允许对文件系统对象进行追加；只有 root 可以设置或清除此位。这仅由 ext2 支持，并且不能跨所有 Unix 系统（或甚至所有 Linux 文件系统）移植。

其他常见扩展包括某种表示“不能删除此文件”的位。

一些类 Unix 系统也支持扩展属性（在 Macintosh 世界中被称为“资源分支”），这些属性本质上是与文件或目录关联的名称/值对，但并不存储在文件或目录的数据本身中。扩展属性可以存储更详细的安全控制信息、MIME 类型等。Linux 内核 2.6 增加了这一功能，但由于许多系统和文件系统不支持它，许多程序选择不使用它们。

一些类 Unix 系统支持 POSIX 访问控制列表（ACLs），允许用户更详细地指定谁可以具体访问文件以及如何访问。有关更多信息，请参阅第 3.2.2 节。

许多这些值可以在挂载时受到影响，例如，某些位可以被视为具有特定值（无论它们在媒体上的值如何）。有关更多信息，请参阅 mount(1)。这些位很有用，但请注意，其中一些旨在简化易用性，实际上并不足以防止某些操作。例如，在 Linux 中，使用“noexec”挂载将禁用该文件系统上程序的执行；正如手册中所述，这是用于挂载包含与不兼容系统二进制的文件系统的。在 Linux 中，此选项不会完全阻止某人运行文件；他们可以将文件复制到其他地方运行，或者甚至使用命令“/lib/ld-linux.so.2”直接运行该文件。

一些文件系统不支持这些访问控制值中的某些；再次提醒，请参阅 mount(1) 了解这些文件系统的处理方式。特别是，许多类 Unix 系统支持 MS-DOS 磁盘，默认情况下，这些磁盘支持的属性非常少（并且没有标准的方式来定义这些属性）。在这种情况下，类 Unix 系统会模拟标准属性（可能通过特殊的磁盘文件来实现），这些属性通常受 mount(1) 命令的影响。

重要的是要注意，对于添加和删除文件，除非 Unix-like 系统支持更复杂的方案（如 POSIX ACLs），否则只有文件的目录的权限位和所有者真正重要。除非系统有其他扩展，而标准的 Linux 2.2 和 2.4 没有，一个在其权限位中没有权限的文件仍然可以被删除，如果其包含的目录允许这样做（例外：标记为“粘性”的目录有特殊规则）。此外，如果祖先目录允许某些用户或组更改其子目录，则该目录的任何后代都可以被该用户或组替换。

值得注意的是，在 Linux 中，Linux ext2 文件系统默认为 root 用户预留了一小部分空间。这是对拒绝服务攻击的部分防御；即使用户填满了与 root 用户共享的磁盘，root 用户仍然会剩下一点空间（例如，用于关键功能）。默认值为文件系统空间的 5%；请参阅 mke2fs(8)，特别是其“-m”选项。

## 3.2.2\. POSIX 访问控制列表 (ACLs)

### 3.2.2.1\. POSIX 访问控制列表 (ACLs) 的历史

原始 Unix 访问控制位（用户、组以及读取、写入和执行的其他值）在各种用途中非常有效。尽管如此，许多用户抱怨在某些情况下，当人们之间共享数据时，这个模型使用起来过于困难。许多人希望给特定的文件或目录添加一组组，或描述特定组的一些特殊权限，而原始方法并没有使这变得容易。

IEEE 成立了一个 POSIX 标准工作组，以确定大量与安全相关接口的通用接口，包括如何创建更复杂的访问控制列表（称为“POSIX 访问控制列表”）。然而，经过 13 年的工作，该小组最终没有就最终草案标准达成一致意见而解散。IEEE 草案标准规范（IEEE 1003.1e 和 IEEE 1003.2c）最后一次编辑是在 1997 年 10 月 14 日，并于 1999 年 3 月 10 日正式解散。我相信这个努力失败的关键原因是因为规范试图涵盖太多不同的领域。因此，不可能就他们所指定的所有内容达成共识，而且漫长的时间意味着最终每个人都放弃了。[草案标准的副本可以免费获取。](http://wt.xpilot.org/publications/posix.1e/download.html)

趣味的是，故事并没有结束。尽管很少有供应商对实现工作组设计的所有接口感兴趣，但对实现更灵活的访问控制列表却有很大的兴趣。虽然还有其他实现访问控制列表的方法，但工作组已经提出了一种合理的方案并将其记录下来。最重要的是，他们详细且合理地解释了为什么实现者应该这样去做。这比最初看起来更重要——尽管更复杂的访问控制列表是一个老想法，但问题是用户需要一个向上兼容的方法，这样就不会与许多现有应用程序产生问题。一个“纯访问控制列表”的方法，即忽略旧方法，将需要重新审查许多现有程序以确保它们不会引起安全问题——任何遗漏都可能导致安全漏洞。工作组还考虑了其他几种替代方案，并在仔细审查后创建了他们的最终方案，该方案强调了与现有应用程序的兼容性。

因此，Unix-like 系统的开发者们逐渐开始添加 POSIX 访问控制列表，或多或少地按照上一份工作草案中的描述进行。这包括 SGI Irix、Sun Solaris、FreeBSD 以及 Linux 内核 2.6（该版本增加了 POSIX 访问控制列表以及扩展属性）。有关 Linux 内核对这些以及一些用户空间工具的实现信息，请参阅[`acl.bestbits.at`](http://acl.bestbits.at)。

然而，虽然编写与 POSIX ACLs 一起工作的程序很重要，但在编写可移植应用程序时，现在依赖它们可能还不明智。Linux 内核版本在 2.6 之前的版本没有 POSIX ACLs，值得注意的是，许多用户空间工具（尤其是像 tar 这样的备份程序）和文件系统格式也不一定支持它们。尽管 NFSv4 规范支持 POSIX ACLs，但许多 NFS 实现并不支持或仅部分支持它们。简而言之，POSIX ACLs 正在逐渐可用，但如果您大量依赖它们，在某些情况下可能会遇到一些问题。

### 3.2.2.2\. POSIX 访问控制列表（ACLs）中使用的数据

在 POSIX ACLs 中，一个 FSO 可能有一组额外的“ACL 条目”，用于确定谁可以访问 FSO；每个目录也可以有一组默认的 ACL 条目，用于在其内部创建 FSO 时使用。每个 ACL 条目可以是多种不同类型之一，并且每个条目还指定了授予的访问权限（r 表示读取，w 表示写入，x 表示执行）。不幸的是，POSIX 草案中这些 ACL 条目类型的名称真的很丑；实际上这是一个简单的系统，但由于名称不佳而变得复杂。有“短形式”和“长形式”两种方式来显示和设置这些信息。

这里是它们的官方名称，以及解释和短形式和长形式：

**表 3-1\. POSIX ACL 条目类型**

| POSIX ACL 条目名称 | 含义 | 短形式 | 长形式 |
| --- | --- | --- | --- |
| ACL_USER_OBJ | 所有者的权限 | u:: | user:: |
| ACL_USER | 除所有者之外的一些特定用户的权限 | u:USERNAME: | user:USERNAME: |
| ACL_GROUP_OBJ | 文件所属组的权限 | g:: | group:: |
| ACL_GROUP | 不拥有文件的某些其他组的权限 | g:GROUPNAME: | group:GROUPNAME: |
| ACL_OTHER | 未被其他方式覆盖的任何人的权限 | o:: | other:: |
| ACL_MASK | 除了所有者和 OTHER 之外，所有人的最大可能权限 | m:: | mask:GROUPNAME: |

“mask”是使这些扩展 POSIX ACLs 与未设计为使用它们的程序良好工作的技巧。如果您指定了除所有者或组所有者之外的用户或组（即，您使用 ACL_USER 或 ACL_GROUP），那么您自动必须有一个 mask 条目。有关 POSIX ACL 的更多信息，请参阅 acl(5)。

## 3.2.3\. 创建时间初始值

在创建时，以下规则适用。在大多数 Unix 系统中，当通过 creat(2)或 open(2)创建新的文件系统对象时，FSO UID 被设置为进程的 EUID，FSO 的 GID 被设置为进程的 EGID。由于 Linux 的 FSUID 扩展，Linux 的工作方式略有不同；FSO 的 UID 被设置为进程的 FSUID，FSO GID 被设置为进程的 FSGUID；如果包含目录的 setgid 位被设置或文件系统的“GRPID”标志被设置，FSO GID 实际上被设置为包含目录的 GID。包括 Sun Solaris 和 Linux 在内的许多系统也支持 setgid 目录扩展。如前所述，这个特殊情况支持“项目”目录：要创建一个“项目”目录，为项目创建一个特殊的组，为该组创建一个项目目录，然后使目录设置为 setgid：放置在那里的文件将自动属于项目。同样，如果在设置了 setgid 位的目录内创建新的子目录（且文件系统 GRPID 未设置），新的子目录也将设置其 setgid 位（这样项目子目录就会“做正确的事情”）；在其他所有情况下，新文件中的 setgid 位将被清除。这是“用户私有组”方案（由 Red Hat Linux 和一些其他系统使用）的原理。在这个方案中，每个用户都是仅包含他们自己的“私有”组的成员，因此他们的默认设置可以允许组读取和写入任何文件（因为他们是组的唯一成员）。因此，当以这种方式转移文件组成员资格时，读取和写入权限也会转移。FSO 基本访问控制值（读取、写入、执行）是从（请求值 & ~ 进程 umask）计算得出的。新文件始终以清除的粘性位和清除的 setuid 位开始。有关 POSIX ACLs 的更多信息，请参阅 acl(5)。

## 3.2.4\. 修改访问控制属性

您可以使用 chmod(2)、fchmod(2)或 chmod(1)来设置这些值中的大多数，但也可以参考 chown(1)和 chgrp(1)。在 Linux 中，一些 Linux 特有的属性是通过 chattr(1)来操作的。

注意，在 Linux 中，只有 root 用户可以更改指定文件的拥有者。一些类 Unix 系统允许普通用户将文件的所有权转让给他人，但这会导致问题，并且 Linux 系统禁止这种行为。例如，如果您正在尝试限制磁盘使用，允许此类操作将使用户声称大文件实际上属于某个其他“受害者”。有关 POSIX ACLs 的更多信息，请参阅 acl(5)。

## 3.2.5\. 使用访问控制属性

在 Linux 和大多数类 Unix 系统中，读取和写入属性值仅在文件打开时进行检查；在每次读取或写入时不会重新检查。尽管如此，大量的调用都会检查这些属性，因为文件系统在类 Unix 系统中至关重要。检查这些属性的调用包括 open(2)、creat(2)、link(2)、unlink(2)、rename(2)、mknod(2)、symlink(2)和 socket(2)。有关 POSIX ACLs 的更多信息，请参阅 acl(5)。

## 3.2.6. 文件系统层次结构

经过多年的发展，已经形成了“将文件放在哪里”的惯例。在可能的情况下，请遵循常规用法在层次结构中放置信息。例如，将全局配置信息放在/etc 中。文件系统层次结构标准（FHS）试图以逻辑方式定义这些惯例，并被 Linux 系统广泛使用。FHS 是之前 Linux 文件系统结构标准（FSSTND）的更新，吸收了 Linux、BSD 和 System V 系统所学的经验和方法。有关 FHS 的更多信息，请参阅[`www.pathname.com/fhs`](http://www.pathname.com/fhs)。Linux 的这些惯例摘要可在 hier(5)中找到，Solaris 的摘要可在 hier(7)中找到。有时不同的惯例不一致；在可能的情况下，在编译或安装时使这些情况可配置。

我应该指出，FHS 已被[Linux 标准基](http://www.linuxbase.org)采用，该组织正在开发和推广一套标准，以提高 Linux 发行版之间的兼容性，并使软件应用程序能够在任何符合规范的 Linux 系统上运行。

# 3.3. 系统 V IPC

> [`dwheeler.com/secure-programs/Secure-Programs-HOWTO/sysv-ipc.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/sysv-ipc.html)

许多类 Unix 系统，包括 Linux 和 System V 系统，支持 System V 进程间通信（IPC）对象。事实上，System V IPC 是 Open Group 的单一 UNIX 规范版本 2 [Open Group 1997]的要求。System V IPC 对象可以是以下三种类型之一：System V 消息队列、信号量集和共享内存段。每个此类对象都有以下属性：

+   创建者、创建者组和其他人的读写权限。

+   创建者 UID 和 GID - 对象创建者的 UID 和 GID。

+   拥有者 UID 和 GID - 对象拥有者的 UID 和 GID（最初等于创建者 UID）。

当访问此类对象时，规则如下：

+   如果进程具有 root 权限，则允许访问。

+   如果进程的 EUID 是对象的拥有者或创建者 UID，则检查适当的创建者权限位以确定是否允许访问。

+   如果进程的 EGID 是对象的拥有者或创建者 GID，或者进程的某个组是拥有或创建对象的 GID，则检查适当的创建者组权限位以确定访问权限。

+   否则，将检查适当的“其他”权限位以确定访问权限。

注意，root 或具有所有者或创建者 EUID 的进程可以设置所有者 UID 和所有者 GID 以及/或删除对象。更多信息请参阅 ipc(5)。

# 3.4. 套接字和网络连接

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/sockets.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/sockets.html)

套接字用于通信，尤其是在网络中。套接字最初是由 Unix 系统的 BSD 分支开发的，但它们通常可以移植到其他类似 Unix 的系统：Linux 和 System V 变体也支持套接字，并且 Open Group 的单一 Unix 规范要求支持套接字[Open Group 1997]。System V 系统传统上使用不同的（不兼容的）网络通信接口，但值得注意的是，像 Solaris 这样的系统包括对套接字的支持。Socket(2) 创建一个通信端点并返回一个描述符，这与文件 open(2) 的方式类似。套接字的参数指定了协议族和类型，例如互联网域（TCP/IP 版本 4）、Novell 的 IPX 或“Unix 域”。服务器通常会调用 bind(2)、listen(2) 和 accept(2) 或 select(2)。客户端通常会调用 bind(2)（尽管这可能被省略）和 connect(2)。有关更多信息，请参阅这些例程各自的 man 页面。从它们的 man 页面理解如何使用套接字可能很困难；你可能想查阅其他论文，例如 Hall "Beej" [1999]，以了解这些调用是如何一起使用的。

“Unix 域套接字”实际上并不代表一个网络协议；它们只能连接到同一台机器上的套接字。（在撰写本文时针对标准 Linux 内核）。当用作流时，它们与命名管道相当相似，但具有显著的优势。特别是，Unix 域套接字是面向连接的；每个新的套接字连接都会导致一个新的通信通道，这与命名管道的情况非常不同。正因为这个特性，Unix 域套接字通常被用于实现许多重要服务的 IPC。就像你可以有未命名的管道一样，你也可以使用 socketpair(2) 有未命名的 Unix 域套接字；未命名的 Unix 域套接字在 IPC 方面与未命名的管道类似。

Unix 域套接字有几个有趣的安全影响。首先，尽管 Unix 域套接字可以出现在文件系统中，并且可以对其应用 stat(2)，但你不能使用 open(2)来打开它们（你必须使用 socket(2)和相关接口）。其次，Unix 域套接字可以用于在进程之间传递文件描述符（不仅仅是文件的内容）。这种在其他任何 IPC 机制中都不存在的奇特功能，已被用于破解各种方案（描述符基本上可以用作计算机科学术语中“能力”的有限版本）。文件描述符是通过 sendmsg(2)发送的，其中 msg（消息）的字段 msg_control 指向一个控制消息头数组的指针（字段 msg_controllen 必须指定数组中包含的字节数）。每个控制消息都是一个 struct cmsghdr，后面跟着数据，为此你希望 cmsg_type 设置为 SCM_RIGHTS。通过 recvmsg(2)检索文件描述符，然后以类似的方式追踪。坦白说，这个功能相当复杂，但了解它是有价值的。

Linux 2.2 及以后的版本支持 Unix 域套接字的一个附加功能：你可以获取对等方的“凭证”（pid、uid 和 gid）。以下是一些示例代码：

```
  /* fd= file descriptor of Unix domain socket connected
    to the client you wish to identify */

 struct ucred cr;
 int cl=sizeof(cr);

 if (getsockopt(fd, SOL_SOCKET, SO_PEERCRED, &cr, &cl)==0) {
   printf("Peer&rsquo;s pid=%d, uid=%d, gid=%d\n",
           cr.pid, cr.uid, cr.gid);
```

标准 Unix 约定是，绑定到 TCP 和 UDP 本地端口号小于 1024 需要 root 权限，而任何进程都可以绑定到 1024 或更大的未绑定端口号。Linux 遵循这个约定，更具体地说，Linux 要求进程具有 CAP_NET_BIND_SERVICE 能力才能绑定到小于 1024 的端口号；这个能力通常只由具有 EUID 为 0 的进程拥有。有冒险精神的用户可以在 Linux 中通过检查其源代码来验证这一点；在 Linux 2.2.12 中，是文件`/usr/src/linux/net/ipv4/af_inet.c`中的函数 inet_bind()。

# 3.5\. 信号

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/signals.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/signals.html)

在类 Unix 操作系统中，信号是一种简单的“中断”形式，并且是 Unix 的古老部分。一个进程可以在另一个进程上设置“信号”（例如使用 kill(1)或 kill(2)），并且那个进程会异步接收和处理信号。为了使一个进程有权向另一个进程发送任意信号，发送进程必须要么具有 root 权限，要么发送进程的真实或有效用户 ID 必须等于接收进程的真实或保存的 set-user-ID。然而，某些信号可以通过其他方式发送。特别是，SIGURG 可以通过 TCP/IP 的带外（OOB）消息在网络中传递。

虽然信号是 Unix 的古老部分，但它们在不同的实现中具有不同的语义。基本上，它们涉及诸如“当处理另一个信号时发生信号会发生什么？”等问题。较老的 Linux libc 5 对于某些信号操作使用与较新的 GNU libc 库不同的语义集。在信号处理程序中调用 C 库函数通常是不安全的，甚至一些系统调用也不安全；你需要检查你做出的每个调用的文档，以查看它是否承诺在信号处理程序内部调用是安全的。有关更多信息，请参阅 glibc FAQ（在某些系统中，本地副本可在`/usr/doc/glibc-*/FAQ`中找到）。

对于新程序，只需使用 POSIX 信号系统（该系统本身基于 BSD 的工作）；这个集合得到了广泛的支持，并且没有一些较老信号系统的问题。POSIX 信号系统基于使用 sigset_t 数据类型，可以通过一系列操作来操作：sigemptyset()，sigfillset()，sigaddset()，sigdelset()和 sigismember()。你可以在 sigsetops(3)中了解这些内容。然后使用 sigaction(2)，sigprocmask(2)，sigpending(2)和 sigsuspend(2)来设置和管理信号处理（有关更多信息，请参阅它们的 man 页面）。

通常，使任何信号处理程序都非常简短和简单，并仔细寻找竞争条件。由于信号本质上是异步的，它们很容易引起竞争条件。

对于服务器来说，存在一个常见的约定：如果你收到 SIGHUP，你应该关闭任何日志文件，重新打开并重新读取配置文件，然后重新打开日志文件。这支持在不停止服务器的情况下重新配置，以及在无数据丢失的情况下进行日志轮换。如果你正在编写一个符合此约定的服务器，请支持它。

Michal Zalewski [2001] 编写了一篇关于如何利用信号处理程序以及如何消除信号竞争问题的优秀教程。我鼓励查看他的总结以获取更多信息；以下是我的建议，与 Michal 的工作类似：

+   在可能的情况下，让信号处理程序无条件地设置一个特定的标志，并不要做其他任何事情。

+   如果你必须使用更复杂的信号处理程序，请仅使用特别指定为在信号处理程序中安全使用的调用。特别是，在 C 语言中不要使用 malloc()或 free()（在大多数系统中它们没有针对信号的保护），也不要使用依赖于它们的许多函数（例如 printf()系列和 syslog()）。你可以尝试通过检查全局标志（以避免重新进入）来“包装”对不安全库调用的调用，但我不会推荐这样做。

+   在程序的所有非原子操作期间阻止信号传递，并在信号处理程序内部阻止信号传递。

# 3.6. 配额和限制

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/quotas.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/quotas.html)

许多类 Unix 系统都有支持文件系统配额和进程资源限制的机制。这当然包括 Linux。这些机制特别有用，可以防止拒绝服务攻击；通过限制每个用户可用的资源，你可以使单个用户难以耗尽所有系统资源。在这里要注意术语，因为文件系统配额和进程资源限制都有“硬”限制和“软”限制，但这两个术语的含义略有不同。

你可以为每个挂载点定义存储（文件系统）配额限制，限制可使用的存储块数和/或唯一文件（inode）数，并且你可以为指定的用户或指定的组设置这样的限制。一个“硬”配额限制是一个永远不会超过的限制，而一个“软”配额可以暂时超过。参见 quota(1)、quotactl(2) 和 quotaon(8)。

rlimit 机制支持大量的进程配额，例如文件大小、子进程数量、打开的文件数量等。有一个“软”限制（也称为当前限制）和一个“硬”限制（也称为上限）。软限制在任何时候都不能超过，但可以通过调用将其提高到硬限制的值。参见 getrlimit(2)、setrlimit(2)、getrusage(2)、sysconf(3) 和 ulimit(1)。请注意，有几种方法可以设置这些限制，包括 pam_limits 模块。

# 3.7\. 动态链接库

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/dlls.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/dlls.html)

几乎所有程序都依赖于库来执行。在大多数现代类 Unix 系统中，包括 Linux，程序默认编译为使用**动态链接库**（DLLs）。这样，你可以更新一个库，并且使用该库的所有程序将使用新的（希望是改进的）版本，如果它们能够做到的话。

动态链接库通常放置在一到几个特殊的目录中。这些目录通常包括 `/lib`、`/usr/lib`、用于 PAM 模块的 `/lib/security`、用于 X-windows 的 `/usr/X11R6/lib` 以及 `/usr/local/lib`。在你的程序中，你应该使用这些标准约定，特别是在调试期间，你不应该使用从当前目录计算出的值作为动态链接库的源（攻击者可能能够添加他们自己的“库”值）。

对于库的命名和为其创建符号链接有特殊的约定，结果是你可以更新库，同时仍然支持想要使用这些库旧版、非向后兼容版本的程序。还有方法在执行特定程序时覆盖特定的库或甚至只是库中的特定函数。这是类 Unix 系统相对于类似 Windows 系统的真正优势；我相信类 Unix 系统在处理库更新方面有一个更好的系统，这也是 Unix 和 Linux 系统被认为比基于 Windows 的系统更稳定的一个原因。

在基于 GNU glibc 的系统上，包括所有 Linux 系统，程序启动时自动搜索的目录列表存储在文件 `/etc/ld.so.conf` 中。许多源自 Red Hat 的发行版通常不会在文件 `/etc/ld.so.conf` 中包含 `/usr/local/lib`。我认为这是一个错误，将 `/usr/local/lib` 添加到 `/etc/ld.so.conf` 是在 Red Hat 衍生系统上运行许多程序时常见的“修复”方法。如果你只想覆盖库中的几个函数，但保留库的其余部分，你可以在 `/etc/ld.so.preload` 中输入覆盖库的名称（.o 文件）；这些“预加载”库将优先于标准集。这个预加载文件通常用于紧急补丁；发行版在交付时通常不会包含此类文件。在程序启动时搜索所有这些目录会非常耗时，因此实际上使用了一种缓存安排。默认情况下，程序 `ldconfig(8)` 会读取文件 `/etc/ld.so.conf`，在动态链接目录中设置适当的符号链接（以便它们遵循标准约定），然后写入缓存到 `/etc/ld.so.cache`，然后其他程序使用这个缓存。因此，每次添加 DLL、删除 DLL 或 DLL 目录集发生变化时，都必须运行 `ldconfig`；在安装库时，包管理器通常会执行运行 `ldconfig` 的步骤之一。因此，在启动时，程序使用动态加载器读取文件 `/etc/ld.so.cache` 并加载它需要的库。

各种环境变量可以控制此过程，实际上存在允许你覆盖此过程的环境变量（例如，你可以临时替换不同的库以用于特定执行）。在 Linux 中，环境变量 LD_LIBRARY_PATH 是一系列以冒号分隔的目录，在这些目录中首先搜索库，然后再搜索标准目录集；这在调试新库或使用非标准库进行特殊目的时很有用，但请确保你信任可以控制这些目录的人。变量 LD_PRELOAD 列出具有覆盖标准集的函数的对象文件，就像 /etc/ld.so.preload 所做的那样。变量 LD_DEBUG 显示调试信息；如果设置为“all”，则在动态链接过程发生时将显示大量关于该过程的信息。

如果没有采取特殊措施，允许用户控制动态链接库将对 setuid/setgid 程序造成灾难性后果。因此，在 GNU glibc 实现中，如果程序是 setuid 或 setgid，这些变量（以及其他类似变量）将被忽略或大大限制其功能。GNU glibc 库通过检查程序的凭证来确定程序是否是 setuid 或 setgid；如果 UID 和 EUID 不同，或者 GID 和 EGID 不同，库假定程序是 setuid/setgid（或从其中派生出来）的，因此大大限制了其控制链接的能力。如果你加载了 GNU glibc 库，你可以看到这一点；特别是查看文件 elf/rtld.c 和 sysdeps/generic/dl-sysdep.c。这意味着如果你使 UID 和 GID 等于 EUID 和 EGID，然后调用一个程序，这些变量将完全生效。其他类 Unix 系统以不同的方式处理这种情况，但出于相同的原因：setuid/setgid 程序不应过度受到环境变量的影响。请注意，图形用户界面工具包通常允许用户控制动态链接库，因为直接调用图形用户界面工具包的可执行文件永远不应该设置为 setuid（或拥有其他特殊权限）。有关如何开发安全的 GUI 应用程序的更多信息，请参阅第 7.4.4 节。

对于 Linux 系统，你可以从我的文档中获得更多信息，即[*程序库 HOWTO*](http://www.dwheeler.com/program-library)。

# 3.8\. 审计

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/audit.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/audit.html)

不同类 Unix 系统在审计处理上有所不同。在 Linux 中，最常用的“审计”机制是 syslogd(8)，通常与 klogd(8)协同工作。您还可能想查看 wtmp(5)、utmp(5)、lastlog(8)和 acct(2)。一些服务器程序（如 Apache 网络服务器）也有它们自己的审计跟踪机制。根据 FHS（文件系统层次结构标准），审计日志应存储在/var/log 或其子目录中。

# 3.10\. Unix-like 系统的专用安全扩展

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/unix-extensions.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/unix-extensions.html)

大量的研究和开发已经投入到扩展 Unix-like 系统以支持各种社区的安全需求中。例如，一些 Unix-like 系统已经扩展以支持美国军方对多层次安全的需求。如果您正在开发软件，您应该尝试设计软件以便它可以在这些扩展中运行。

FreeBSD 有一个新的系统调用，[jail(2)](http://docs.freebsd.org/44doc/papers/jail/jail.html)。jail 系统调用支持将环境子分区为许多虚拟机（在某种程度上，是一个“超级 chroot”）；其最流行的用途是为互联网服务提供商环境提供虚拟机服务。在 jail 内部，所有进程（即使是 root 拥有的进程）的请求范围都限制在 jail 内。当 FreeBSD 系统在全新安装后启动时，不会有任何进程在 jail 中。当一个进程被放入 jail 时，它以及由此进程创建的所有后代进程也将位于该 jail 中。一旦进入 jail，对文件名空间的访问将受到 chroot(2)风格的限制（典型的 chroot 逃逸路径被阻止），绑定网络资源的能力限制在特定的 IP 地址，操纵系统资源和执行特权操作的能力急剧减少，与其他进程交互的能力仅限于同一 jail 内的进程。请注意，每个 jail 都绑定到一个单独的 IP 地址；jail 内的进程可能无法使用任何其他 IP 地址进行出站或入站连接。更多信息请参阅[OnLamp.com 关于 FreeBSD Jails 的文章](http://www.onlamp.com/pub/a/bsd/2003/09/04/jails.html)。

在 Linux 中可用的某些扩展，例如 POSIX 能力和特殊的挂载时选项，已经讨论过了。这里列举了为 Linux 系统创建受限执行环境的一些努力，有众多不同的方法。Linux 2.6 添加了“Linux 安全模块”（LSM）接口，允许管理员插入模块以执行更复杂的访问控制系统。美国国家安全局（NSA）开发了[安全增强型 Linux（Flask）](http://www.nsa.gov/selinux)（SELinux），它支持使用专用语言定义安全策略，并强制执行该策略。最初 SELinux 是作为一组独立的补丁开发的，但现在它使用 LSM，NSA 已将 SELinux 内核模块提交给 Linux 开发者，以便包含在标准内核中。[Medusa DS9](http://medusa.fornax.sk) 通过在内核级别支持用户空间授权服务器来扩展 Linux。[LIDS](http://www.lids.org) 保护文件和进程，允许管理员“锁定”他们的系统。“基于规则集的访问控制”系统 [RSBAC](http://www.rsbac.de) 基于 Abrams 和 LaPadula 的通用访问控制框架（GFAC），提供了一种基于多个内核模块的灵活访问控制系统。[Subterfugue](http://subterfugue.org) 是一个用于“观察和玩弄软件现实”的框架；它可以拦截系统调用并更改它们的参数和/或更改它们的返回值以实现沙箱、跟踪器等；它可以在没有任何更改的情况下在 Linux 2.4 下运行（它不需要任何内核修改）。[Janus](http://www.cs.berkeley.edu/~daw/janus) 是一个用于在受限执行环境中沙箱化不受信任应用程序的安全工具。有些人甚至使用了[用户模式 Linux](http://user-mode-linux.sourceforge.net)，它实现了“Linux 在 Linux 上”，作为一种沙箱实现。由于实现更复杂的安全模型有如此多的不同方法，Linus Torvalds 要求开发一种通用方法，以便可以插入不同的安全策略；有关更多信息，请参阅[`mail.wirex.com/mailman/listinfo/linux-security-module`](http://mail.wirex.com/mailman/listinfo/linux-security-module)。

在各种类 Unix 系统上还有许多其他用于安全性的扩展，但这些内容实际上超出了本文档的范围。
