# 2.1\. Unix、Linux 和开源/自由软件的历史

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/history.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/history.html)

## 2.1.1\. Unix

在 1969-1970 年，肯尼思·汤普森、丹尼斯·里奇和其他在 AT&T 贝尔实验室的人开始在一个很少使用的 PDP-7 上开发一个小型操作系统。这个操作系统很快被命名为 Unix，是对一个早期操作系统项目 MULTICS 的双关语。在 1972-1973 年，系统用 C 语言重写，这是一个具有前瞻性的不寻常步骤：由于这个决定，Unix 成为了第一个可以切换并超越其原始硬件的广泛使用的操作系统。Unix 还增加了其他创新，部分得益于贝尔实验室和学术界之间的协同作用。1979 年，Unix 的“第七版”（V7）版本发布，是所有现有 Unix 系统的祖先。

从这一点开始，Unix 的历史变得有些复杂。以伯克利大学为首的学术界开发了一个名为伯克利软件发行版（BSD）的变种，而 AT&T 则继续在“System III”和后来的“System V”名称下开发 Unix。在 20 世纪 80 年代末至 90 年代初，这两种主要流派之间爆发了“战争”。经过多年，每个变种都采纳了对方的一些关键特性。在商业上，System V 赢得了“标准战争”（大多数接口都纳入了正式标准），大多数硬件供应商转向了 AT&T 的 System V。然而，System V 最终吸收了许多 BSD 的创新，因此产生的系统更像是两个分支的合并。BSD 分支并没有消亡，反而广泛应用于研究、PC 硬件和专用服务器（例如，许多网站使用 BSD 的衍生版本）。

结果是出现了许多不同的 Unix 版本，它们都基于原始的第七版。大多数 Unix 版本都是专有的，由各自的硬件供应商维护，例如，Sun Solaris 是 System V 的一个变种。Unix 的 BSD 分支最终有三个版本成为开源：FreeBSD（专注于 PC 硬件的安装简便性）、NetBSD（专注于多种不同的 CPU 架构）以及 NetBSD 的一个变种，OpenBSD（专注于安全性）。更多关于 Unix 历史的通用信息可以在[`www.datametrics.com/tech/unix/uxhistry/brf-hist.htm`](http://www.datametrics.com/tech/unix/uxhistry/brf-hist.htm)、[`perso.wanadoo.fr/levenez/unix`](http://perso.wanadoo.fr/levenez/unix)和[`www.crackmonkey.org/unix.html`](http://www.crackmonkey.org/unix.html)（注意，Microsoft Windows 系统无法读取最后一个链接）。[Unix Heritage Society](http://www.tuhs.org)引用了 Unix 历史的多个来源。关于 BSD 历史的更多信息可以在 Mckusick 1999 和 ftp://ftp.freebsd.org/pub/FreeBSD/FreeBSD-current/src/share/misc/bsd-family-tree 中找到。

一篇稍微有些过时但有趣的倡导文章，提出了使用 Unix-like 系统（而不是 Microsoft 的产品）的论点，是[John Kirch 的论文“Microsoft Windows NT Server 4.0 versus UNIX”](http://web.archive.org/web/20010801155417/www.unix-vs-nt.org/kirch)。

## 2.1.2\. 自由软件基金会

1984 年，理查德·斯托尔曼的免费软件基金会（FSF）开始了 GNU 项目，这是一个旨在创建 Unix 操作系统的免费版本的项目。斯托尔曼所说的“免费”，是指软件可以被自由使用、阅读、修改和重新分发。FSF 成功地构建了大量有用的组件，包括 C 编译器（gcc）、令人印象深刻的文本编辑器（emacs）以及一系列基本工具。然而，在 20 世纪 90 年代，FSF 在开发操作系统内核方面遇到了困难[FSF 1998]；没有内核，他们关于一个完全免费的操作系统的梦想将无法实现。

## 2.1.3\. Linux

1991 年，林纳斯·托瓦兹开始开发一个操作系统内核，他将其命名为“Linux”[Torvalds 1999]。这个内核可以与 FSF 材料和其他组件（特别是某些 BSD 组件和 MIT 的 X 窗口软件）结合，以产生一个可以自由修改且非常有用的操作系统。本书将内核本身称为“Linux 内核”，而将整个组合称为“Linux”。请注意，许多人使用“GNU/Linux”这个术语来指代这个组合。

在 Linux 社区中，不同的组织以不同的方式组合了可用的组件。每种组合被称为“发行版”，开发发行版的组织被称为“发行商”。常见的发行版包括 Red Hat、Mandrake、SuSE、Caldera、Corel 和 Debian。不同的发行版之间存在差异，但所有发行版都基于相同的基础：Linux 内核和 GNU glibc 库。由于两者都受“copyleft”风格许可的保护，对这些基础所做的更改通常必须对所有用户开放，这是 Linux 发行版在其基础上的一个统一力量，而在 BSD 和 AT&T 派生的 Unix 系统之间则不存在这种力量。本书不针对任何特定的 Linux 发行版；当它讨论 Linux 时，假设 Linux 内核版本为 2.2 或更高，C 库 glibc 为 2.1 或更高，这对于几乎所有当前的主要 Linux 发行版都是有效的假设。

## 2.1.4\. 开源/免费软件

对免费共享软件的兴趣增加，使得定义和解释它变得越来越必要。一个常用的术语是“开源软件”，它在[OSI 1999]中有进一步的定义。埃里克·雷蒙德（Eric Raymond）[1997, 1998]撰写了几篇开创性的文章，考察了其各种开发过程。另一个广泛使用的术语是“免费软件”，其中的“免费”代表“自由”：通常的解释是“言论自由，而非啤酒免费”。这两个短语都不完美。“免费软件”这个术语常被误解为那些可免费获得可执行文件，但源代码无法查看、修改或重新分发的程序。相反，“开源”这个术语有时（滥用）被用来指代源代码可见，但对使用、修改或重新分发有限制的软件。本书使用“开源”这个术语，其含义是指源代码可以自由使用、查看、修改和重新分发的软件；更详细的定义包含在[开源定义](http://www.opensource.org/osd.html)中。在某些情况下，动机上的差异被提出；那些偏好“免费软件”这个术语的人希望强烈强调自由的需求，而使用这个术语的人可能有其他动机（例如，更高的可靠性）或只是希望显得不那么强硬。关于这个免费软件定义及其背后的动机的信息，可以在[`www.fsf.org`](http://www.fsf.org)找到。

对阅读开源软件和免费软件倡导文章感兴趣的人应该查看[`www.opensource.org`](http://www.opensource.org)和[`www.fsf.org`](http://www.fsf.org)。还有其他一些文档考察了这类软件，例如，米勒（Miller）在 1995 年发现，开源软件在可靠性方面明显优于专有软件（使用他们的测量技术，该技术测量由于随机输入而崩溃的抵抗力）。

## 2.1.5. 比较 Linux 和 Unix

这本书使用“Unix-like”这个术语来描述有意模仿 Unix 的系统。特别是，“Unix-like”这个术语包括所有主要的 Unix 变种和 Linux 发行版。请注意，许多人简单地使用“Unix”这个术语来描述这些系统。最初，“Unix”这个术语指的是 AT&T 开发的一个特定产品。如今，开放集团拥有 Unix 商标，并将其定义为“全球单一 UNIX 规范”。

Linux 并非源自 Unix 源代码，但其接口有意模仿 Unix。因此，Unix 中学到的经验教训通常适用于两者，包括安全信息。本书中的大部分信息适用于任何 Unix-like 系统。为了使使用 Linux 的用户能够利用 Linux 的功能，特意添加了针对 Linux 的特定信息。

类 Unix 系统共享许多安全机制，尽管存在细微的差异，并不是所有系统都有所有机制可用。所有系统都包括每个进程的用户和组 ID（UID 和 GID）以及具有读取、写入和执行权限的文件系统（对于用户、组和其它）。参见 Thompson [1974] 和 Bach [1986] 关于 Unix 系统的一般信息，包括它们的基本安全机制。第三章总结了 Unix 和 Linux 的关键安全特性。

# 2.2. 安全原则

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/security-principles.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/security-principles.html)

有许多通用的安全原则，你应该熟悉；关于信息安全的一般信息，一个很好的地方是信息保障技术框架（IATF）[NSA 2000]。NIST 已经确定了高级的“普遍接受的原则和实践”[Swanson 1996]。你也可以查看关于计算机安全的一般教科书，例如[Pfleeger 1997]。NIST 特别出版物 800-27 描述了若干好的工程原则（尽管，由于它们是抽象的，它们对于实际构建安全程序是不够的——因此有这本书）；你可以在[`csrc.nist.gov/publications/nistpubs/800-27/sp800-27.pdf`](http://csrc.nist.gov/publications/nistpubs/800-27/sp800-27.pdf)获取副本。这里总结了几个安全原则。

通常，计算机安全目标（或目标）是用三个总体目标来描述的：

+   *保密性*（也称为机密性），意味着计算系统的资产只能由授权方读取。

+   *完整性*，意味着资产只能以授权的方式由授权方修改或删除。

+   *可用性*，意味着资产能够及时地（根据系统要求确定）被授权方访问。未能达到这个目标被称为拒绝服务。

有些人定义了额外的重大安全目标，而有些人将这些额外目标视为这三个目标的特殊情况。例如，有些人将不可否认性作为一个目标单独识别；这是指即使发送者或接收者后来希望否认，也能“证明”发送者发送了或接收者接收了消息（或两者都是）的能力。隐私有时与机密性分开处理；有些人将其定义为保护用户的机密性（例如，他们的身份），而不是数据。大多数目标都需要身份验证和认证，这有时被列为一个单独的目标。通常，审计（也称为问责制）被确定为理想的安全目标。有时，“访问控制”和“真实性”也被单独列出。例如，美国国防部（DoD）在其指令 3600.1 中将“信息安全”定义为“通过确保可用性、完整性、真实性、机密性和不可否认性来保护信息及其信息系统的信息操作（IO）。这包括通过结合保护、检测和反应能力来恢复信息系统。”

在任何情况下，重要的是要确定你程序的整体安全目标，无论你如何将它们分组，这样你就可以知道何时实现了它们。

有时，这些目标是对已知威胁集的回应，有时某些目标是由法律要求的。例如，对于美国的银行和其他金融机构，有一项新的隐私法称为“格拉姆-利奇-布莱利”（GLB）法案。该法案要求披露共享的个人信息和保护这些数据的方法，要求披露将与其他第三方共享的个人信息公开，并指导机构让客户有机会选择退出数据共享。[Jones 2000]

有时安全性与某些其他通用系统/软件工程原则之间会产生冲突。例如，安全性有时会干扰“易用性”，安装一个安全配置可能比一个“简单”但不够安全的安装需要更多的努力。通常，这种明显的冲突可以通过重新思考问题来解决，例如，通常可以设计一个既安全又易于使用的系统。有时，安全性与抽象（信息隐藏）之间也会产生冲突；例如，某些高级库例程可能被安全实现，也可能没有，但它们的规范不会告诉你。最后，如果你的应用程序必须保证安全，那么如果你不能确定其他方法，你必须亲自去做——是的，库应该被修复，但你的用户会因为你在库例程选择上的糟糕决定而受到伤害。

一个好的通用安全原则是“深度防御”；你应该有多个防御机制（“层”）就位，设计成攻击者必须击败多个机制才能成功攻击。

关于如何设计安全程序的一般原则，请参阅第 7.1 节。

# 2.3. 为什么程序员会编写不安全的代码？

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/why-write-insecure.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/why-write-insecure.html)

许多程序员并不打算编写不安全的代码——但仍然这样做了。以下是一些所谓的理由。其中大部分是由 Aleph One 在 Bugtraq 上收集和总结的（1998 年 12 月 17 日的一篇帖子）：

+   在大多数学校中，没有课程涉及计算机安全。即使有计算机安全课程，他们通常也不会讨论如何编写整个安全程序。许多这样的课程只研究某些领域，如密码学或协议。这些很重要，但它们通常不会讨论常见的现实世界问题，如缓冲区溢出、字符串格式化和输入检查。我相信这是最重要的一个问题；即使那些在大学和学院接受教育的程序员也很少有机会学习如何编写安全程序，但我们却依赖这些人编写安全程序。

+   编程书籍/课程没有教授安全/安全编程技术。实际上，直到最近，还没有关于如何编写安全程序的书籍（这本书是其中之一）。

+   没有人使用形式化验证方法。

+   C 是一种不安全的语言，标准 C 库中的字符串函数也是不安全的。这一点尤为重要，因为 C 的使用非常广泛——“简单”的 C 使用方式允许危险的利用。

+   程序员没有考虑“多用户”。

+   程序员是人类，人类是懒惰的。因此，程序员通常会采用“简单”的方法而不是安全的方法——一旦成功，他们往往后来也未能修复它。

+   大多数程序员并不是优秀的程序员。

+   大多数程序员不是安全专家；他们并不经常像攻击者那样思考。

+   大多数安全专家不是程序员。这是某些 Bugtraq 贡献者做出的声明，但并不清楚这个说法是否真的正确。

+   大多数计算机安全模型都很糟糕。

+   有大量的“破损”的遗留软件。修复这些软件（以消除安全漏洞或使其与更严格的安全策略兼容）是困难的。

+   消费者并不关心安全性。（就我个人而言，我希望能有消费者开始关心安全性；一个不断被利用的计算机系统既不实用也不用户友好。此外，许多消费者甚至不知道存在这个问题，认为这种情况不会发生在他们身上，或者认为事情不能变得更好。）

+   安全性需要额外的发展时间。

+   安全成本包括额外的测试（红队等）。

# 2.4\. 开源软件对安全有益吗？

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/open-source-security.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/open-source-security.html)

安全从业者就开源方法对安全的影响进行了很多辩论。一个关键问题是开源使得源代码对所有人开放，包括攻击者和防御者，而理性的人对此情况最终影响持不同意见。（注意 - 您可以通过访问本书的主网站 [`www.dwheeler.com/secure-programs`](http://www.dwheeler.com/secure-programs) 获取这篇论文的最新版本）。

## 2.4.1\. 各专家的观点

首先，让我们看看安全专家们有什么看法。

Bruce Schneier 是计算机安全和密码学方面的知名专家。他认为聪明的工程师应该“要求任何与安全相关的代码都应该是开源的” [Schneier 1999]，他还讨论了必须满足的一些前提条件，以使开源软件更安全。AES 加密算法的开发商 Vincent Rijmen 认为，Linux 的开源特性为发现和修复安全漏洞提供了更好的途径，“不仅因为更多的人可以查看它，更重要的是，这种模式迫使人们编写更清晰的代码，并遵守标准。这反过来又促进了安全审查” [Rijmen 2000]。

Elias Levy (Aleph1) 是最受欢迎的安全讨论组之一 - Bugtraq 的前版主。他在文章 ["开源软件真的比封闭源代码更安全吗？](http://www.securityfocus.com/commentary/19) 中讨论了开源软件安全方面的一些问题。他的总结是：

> 那么，这一切是否意味着开源软件在安全漏洞方面并不比封闭源代码软件更好？不。开源软件确实有可能比其封闭源代码版本更安全。但不要误解，仅仅因为是开源的并不能保证其安全性。

威尔特菲尔德·迪菲（Whitfield Diffie）是公钥加密（所有互联网安全的基础）的共同发明者，同时也是太阳微系统公司（Sun Microsystems）的首席安全官和高级工程师。在他的 2003 年文章 [风险业务：保持安全为秘密](http://zdnet.com.com/2100-1107-980938.html) 中，他辩称，专有供应商声称他们的软件因为保密而更安全是胡说八道。他识别并反驳了专有供应商提出的两个主要论点：（1）代码的发布对攻击者比其他人更有利，因为许多敌对的目光也可以查看开源代码，以及（2）少数专家的眼睛比几个随机的人更好。他首先指出，虽然给程序员访问软件的一部分并不能保证他们会仔细研究它，但有一群程序员可以预期会非常关心：那些要么个人使用该软件，要么为依赖该软件的企业工作的人。“事实上，审计企业依赖的软件以保障其自身安全是企业管理信息安全的自然功能。”然后他反驳第二个论点，指出：“至于开源对对手的有用性超过对用户的优势这一观点，这一论点与安全领域最重要的原则之一相悖：一个不能轻易更改的秘密应被视为一个漏洞。”他最后总结说：

> “在计算机软件中依赖保密来保证安全是不切实际的。你可能能够将程序的精确工作原理从公众视野中隐藏起来，但你能否阻止严肃的对手对其进行逆向工程？可能不能。”

约翰·维加（John Viega）的文章 ["开源安全神话](http://dev-opensourceit.earthweb.com/news/000526_security.html)" 也讨论了相关问题，并这样总结：

> 开源软件项目可能比封闭源代码项目更安全。然而，那些可以使开源程序更安全的因素——源代码的可用性，以及大量用户可用于寻找和修复安全漏洞的事实——也可能使人们陷入一种虚假的安全感。

[迈克尔·H·沃菲尔德的《关于开源安全的沉思》](http://www.linuxworld.com/linuxworld/lw-1998-11/lw-11-ramparts.html)对开源软件对安全性的影响持非常积极的态度。相比之下，弗雷德·施奈德不相信开源有助于安全性，他说“没有理由相信那些检查（开源）源代码的众多眼睛能够成功识别出导致系统安全性被破坏的漏洞”并声称“代码中的漏洞不是攻击的主要手段” [施奈德 2000]。他还声称开源排除了对构建过程的控制，尽管在实践中确实存在这样的控制——所有主要的开源程序都有一个或几个官方版本，有“所有者”在声誉上有所赌注。彼得·G·纽曼讨论了“开箱软件”（其中源代码可用，可能仅在特定条件下），他说“开箱软件真的能提高系统安全性吗？我的回答是，它本身并不能，尽管潜力是相当大的” [纽曼 2000]。在红帽（一家开源公司）的赞助下，TruSecure 公司开发了一份关于他们为什么认为开源在安全性方面更有效的论文 [TruSecure 2001]。[娜塔莉·沃克·惠洛克在 IBM DeveloperWorks 上的文章](http://www-106.ibm.com/developerworks/linux/library/l-oss.html?open&I=252,t=gr,p=SeclmpOS)也讨论了利弊。布赖恩·威滕、卡尔·兰德韦尔和迈克尔·卡洛扬尼迪斯 [威滕 2001] 在 IEEE Software 上发表了一篇文章，初步得出结论，认为源代码可用应该有利于系统安全性；他们指出：

> “从这次讨论中，我们可以得出四个额外的结论。首先，访问源代码使用户能够提高系统安全性——如果他们具备这样做的能力和资源。其次，有限的测试表明，在某些情况下，开源的生命周期会产生对非恶意故障更不脆弱的系统。第三，对三种操作系统的调查表明，在 12 个月的时间里，一个开源操作系统在已知但未修补的漏洞方面的暴露程度低于两个专有操作系统的任何一个。最后，只要不安全的系统更有利可图，封闭和专有系统开发模型就会面临不利于部署和支持更安全系统的障碍。尽管有这些结论，但在这个重要问题上的争论还处于形成阶段，迫切需要能够反映提供给客户的实际安全性的指标。”

Scott A. Hissam 和 Daniel Plakosh 在他们的论文《“开源软件中的信任与漏洞”》（[“Trust and Vulnerability in Open Source Software”](http://www.ics.uci.edu/~wscacchi/Papers/New/IEE_hissam.pdf)）中讨论了开源软件的优缺点。与其他论文一样，他们指出，尽管软件是开放的，可以接受审查，但这并不意味着这种审查实际上已经完成。实际上，他们指出，这是所有软件（无论是开源还是封闭）的一个普遍问题——许多人是否审查了任何特定的软件，这往往是可疑的。一个有趣的观点是，他们证明了攻击者可以从对开源/自由软件程序（Linux）进行的补丁中了解到封闭源代码程序（Windows）中的漏洞。在这个例子中，Linux 开发者在攻击者尝试攻击之前修复了一个漏洞，攻击者正确地推测，Windows 中可能还存在类似的问题（确实如此）。除非禁止开源/自由软件程序，否则这种学习是难以阻止的。因此，开源/自由软件程序的存在可以揭示执行相同功能的开源/自由软件和专有软件的漏洞——但在这个例子中，开源/自由软件程序是首先被修复的。

## 2.4.2. 为什么关闭源代码不能阻止攻击

有观点认为，没有源代码的系统更安全，因为攻击者可获取的信息较少，因此应该更难发现漏洞。然而，这种论点存在许多弱点，因为尽管源代码在尝试为程序添加新功能时非常重要，但攻击者通常不需要源代码来发现漏洞。此外，这种论点还假设你总能保守源代码的秘密，这往往是不真实的。

首先，重要的是要区分“破坏性”行为和“建设性”行为。在现实世界中，毁掉一辆车比建造一辆车容易得多。在软件世界中，发现和利用漏洞比向软件添加新的重要功能要容易得多。由于这种差异，攻击者相对于防御者有许多优势。软件开发者必须努力确保他们的代码中没有任何与安全相关的错误，而攻击者只需要找到一处。开发者主要被雇佣来使他们的程序运行……攻击者不需要使程序运行，他们只需要找到一处弱点。而且，正如我稍后将描述的，攻击一个程序所需的信息比修改一个程序所需的信息要少。

通常，攻击者（针对开源和闭源程序）首先了解程序可能存在的各种安全问题。隐藏这种信息是没有意义的；它已经公开了，而且无论如何，防御者需要这种信息来自卫。然后，攻击者使用技术来尝试找到这些问题；我将这些技术分为“动态”技术（运行程序）和“静态”技术（检查程序的代码——无论是源代码还是机器码）。

在“动态”方法中，攻击者运行程序，向其发送数据（通常是问题数据），并查看程序的响应是否表明了常见的漏洞。开源和闭源程序在这里没有区别，因为攻击者没有查看代码。

攻击者也可能查看代码，采用“静态”方法。对于开源软件，他们可能会查看源代码并搜索其中的模式。对于闭源软件，你可以搜索机器码（通常以汇编语言格式呈现以简化任务）中表明安全问题的模式。实际上，有几种工具可以做到这一点。攻击者还可能使用称为“反汇编器”的工具，将机器码转换回源代码，然后在源代码中搜索易受攻击的模式（就像他们在开源软件中搜索漏洞一样）。参见 Flake [2001] 中关于如何检查闭源代码中的安全漏洞的讨论（例如，使用反汇编器）。这一点很重要：即使攻击者想使用源代码来寻找漏洞，闭源程序也没有优势，因为攻击者可以使用反汇编器重新创建产品的源代码（用于分析），或者使用二进制扫描工具。

非开发者可能会问：“如果反汇编器可以从机器码创建源代码，那么为什么开发者说他们需要源代码而不是仅仅机器码？”问题在于，尽管开发者不需要源代码来发现安全问题，但他们确实需要源代码来对程序进行实质性的改进。尽管反汇编器可以将机器码转换回某种“源代码”，但生成的源代码极难修改。通常，大多数可理解的名字都会丢失，所以你得到的不是“grand_total”这样的变量，而是“x123123”，不是“display_warning”这样的方法，而是“f123124”，而且代码本身可能还夹杂着汇编代码。此外，_ 所有 _ 注释和设计信息都会丢失。这对于发现安全问题来说并不是一个严重的问题，因为通常你是在寻找表明漏洞的模式，而不是内部变量或方法名称。因此，反汇编器和二进制代码扫描工具可以用来寻找攻击程序的方法，或者查看程序有多脆弱，但它们对于更新程序并没有帮助。

因此，当开发者意图添加功能时，他们会说“源代码至关重要”，但实际上，封闭源代码程序的源代码是隐藏的，这并不能很好地保护程序。事实上，仅使用二进制程序的用户在使用反编译器或二进制扫描工具时可能会遇到问题；一个勤奋的用户可能会知道一个可以利用的安全漏洞，但无法轻易修复，而且他们可能也无法说服供应商修复它。

这还假设你无论如何都能从攻击者那里保守源代码的秘密。例如，微软至少有几次被盗取了其源代码，至少有一次是从微软本身，至少有一次是从它共享数据的另一家公司。微软还有与各种政府、公司和教育机构共享其源代码的计划；其中一些组织包括攻击者，这些组织可能会被其他人攻击以获取源代码。我仅以此为例；许多公司必须共享源代码有许多原因。而且这还没有考虑到恶意泄露源代码的受损害员工。依赖于源代码的长期保密是一种自我欺骗；你可以推迟其发布，但如果它很重要，它迟早会被盗取。保守源代码的秘密对专有软件供应商来说在经济上是有意义的，作为一种鼓励客户购买产品和支持的方式，但这并不是一个强大的安全措施。

## 2.4.3. 为什么隐藏漏洞并不能使它们消失

有时人们会指出，一个存在但未知的漏洞无法被利用，因此系统“实际上很安全”。从理论上讲，这是正确的，但问题是，一旦有人发现了这个漏洞，发现者可能会利用这个漏洞而不是帮助修复它。未知漏洞并不会真正使漏洞消失；它只是意味着漏洞像定时炸弹一样，无法知道它们何时会被利用。从根本上讲，有人利用他们发现的漏洞的问题，对开源和封闭源代码系统都是一个问题。

有时候会提出一个相关的说法（尽管与开源/自由软件没有直接关系），即人们不应该发布关于漏洞的警告并讨论它们。在理论上这听起来不错，但问题是攻击者已经通过大量渠道分发关于漏洞的信息。简而言之，这种做法会让防御者处于脆弱状态，而对攻击者却没有任何抑制作用。在过去，公司积极尝试防止漏洞信息的披露，但经验表明，总的来说，公司只有在漏洞被广泛告知用户（用户随后可以要求修复漏洞）之后才会修复漏洞。这全是“完全披露”论点的组成部分。Gartner 集团在 CNET.com 文章《评论：炒作才是真正的问题 - 科技新闻》中有一段直率的评论。他们表示：

> 微软安全响应中心经理斯科特·库普的评论与关于信息分布的道德讨论相呼应，这种讨论由来已久，非常熟悉。例如，几个世纪以前，教会试图压制哥白尼和伽利略关于太阳是太阳系中心的学说...库普试图将最近微软产品中出现的漏洞归咎于“信息安全专业人士”的做法至多是不诚实的。也许，这也代表了试图将批评从构建这些产品的公司身上转移开来的尝试...各方的努力都有助于一个持续改进的过程。漏洞越广泛为人所知，修复的速度就越快。

## 2.4.4\. 如何识别开源/自由软件中的特洛伊木马

有时候人们会争论，由于开源程序没有单一公司的强制控制，这允许人们插入特洛伊木马和其他恶意代码。确实，特洛伊木马可以被插入到开源代码中，但它们也可以被插入到专有代码中。一个不满或受贿的员工可以插入恶意代码，而且在许多组织中，这种恶意代码被发现的可能性比在开源程序中要小得多。毕竟，组织外部没有人可以审查源代码，而且很少有公司会内部审查他们的代码（或者即使他们这样做，也很少能保证审查过的代码实际上就是使用的代码）。而且，一个闭源公司可以事后被起诉的观念几乎没有证据；几乎所有许可证都声明不承担任何保证，而且法院通常没有判定软件开发公司负有责任。

博兰德的 InterBase 服务器是一个有趣的案例。在 1992 年至 1994 年之间，博兰德故意在他们的数据库服务器“InterBase”中插入了一个“后门”。这个后门允许任何本地或远程用户操纵任何数据库对象并安装任意程序，在某些情况下甚至可能导致以“root”身份控制机器。这个漏洞在该产品中至少存在了 6 年——没有人可以审查该产品，博兰德也没有动力去除这个漏洞。然后博兰德在 2000 年 7 月发布了其源代码。"Firebird"项目开始使用源代码，并在 2000 年 12 月揭露了 InterBase 的这个严重安全问题。到 2001 年 1 月，CERT 宣布了这个后门的存在，作为[CERT 咨询 CA-2001-01](http://www.cert.org/advisories/CA-2001-01.html)。令人沮丧的是，这个后门可以通过简单地查看程序的 ASCII 转储（一个常见的黑客技巧）来轻松找到。一旦开源开发者通过审查代码发现了这个问题，它就很快得到了修复。你可以争论，通过保持密码未知，程序保持了安全，而开放源代码使得程序变得不那么安全。我认为这是无稽之谈，因为 ASCII 转储是微不足道的，并且作为标准的攻击技术而众所周知，并不是所有攻击者都有突然宣布漏洞的冲动——事实上，根本无法确定这个漏洞是否已经被多次利用。很明显，在源代码开放之后，随着时间的推移，源代码得到了审查，发现了漏洞并得到了修复。可以这样描述，原始代码是脆弱的，当它首次开源时，其漏洞更容易被利用，然后最终这些漏洞得到了修复。

## 2.4.5. 其他优势

开放源代码的优势不仅限于正在遭受攻击的软件，还扩展到了漏洞评估扫描器。漏洞评估扫描器故意在配置的系统内寻找漏洞。最近的一次网络计算评估发现，最好的扫描器（在许多方面，它发现了最多的合法漏洞）是 Nessus，一个开源扫描器[Forristal 2001]。

## 2.4.6. 总结

那么，底线是什么呢？我个人认为，当一个程序最初是封闭源代码，然后首次开源时，它通常对任何用户来说都更不安全（通过暴露漏洞），随着时间的推移（比如说几年），它有可能比封闭程序更安全。如果一个程序最初就是开源软件，公众的审查更有可能在它准备好供大量用户使用之前提高其安全性，但这个陈述有几个注意事项（这不是一个不可动摇的规则）。仅仅将一个程序开源并不会突然使其变得安全，而且一个程序是开源的并不保证其安全性：

+   首先，人们必须真正审查代码。这是辩论的关键点之一——人们真的会在开源项目中审查代码吗？各种因素可以减少审查的数量：是一个利基或很少使用的产品（那里很少有潜在的审查者），开发者数量少，以及使用很少使用的计算机语言。显然，只有一个开发者和没有其他任何类型贡献者的程序没有这种审查。另一方面，一个有主要作者和许多偶尔审查代码并做出贡献的人的程序表明，其他人正在审查代码（至少是为了创建贡献）。一般来说，如果有更多的审查者，通常有更高的可能性有人会识别出缺陷——这是“许多眼睛”理论的基础。请注意，例如，OpenBSD 项目持续检查程序的安全漏洞，因此其最内层的组件肯定已经经过了长时间的审查。由于 OSS/FS 讨论通常是公开进行的，这种审查水平是潜在用户可以自己判断的。

    一个可能特别降低审查可能性的因素实际上并不是开源。一些供应商喜欢将他们的“披露源代码”程序（也称为“源代码可用”）定位为开源，但由于程序所有者拥有广泛的独家权利，其他人将很少有动力为所有者免费工作在代码上。即使是具有异常不对称权利的开源许可证（如 MPL）也存在这个问题。毕竟，如果其他人将有权获得他们的成果而他们自己没有（正如布鲁斯·佩雷斯所说，“谁愿意成为别人的无偿雇员？”），人们自愿参与的意愿就会降低。特别是，由于最有激励力的审查者往往是试图修改程序的人，这种参与的不激励因素减少了“眼睛”的数量。伊利亚斯·利维在他的关于开源安全性的文章中犯了错误；他举的软件被破解的例子（例如，TIS 的 Gauntlet）在那时并不是开源的。

+   第二，至少一些开发和审查代码的人必须知道如何编写安全程序。希望这本书的存在能有所帮助。显然，如果没有人知道要寻找什么，那么“许多眼睛”也没有用。请注意，并非每个人都必须知道如何编写安全程序，只要那些知道的人正在审查代码更改即可。

+   第三，一旦发现问题，就需要迅速修复并分发其修复方案。开源系统通常能迅速修复问题，但分发并不总是顺利。例如，OpenBSD 开发者对代码进行安全漏洞审查做得非常出色——但他们并不总是将识别出的问题报告给原始开发者。因此，在某个系统中可能存在修复版本，但在另一个系统中漏洞仍然存在。我相信随着时间的推移，这个问题正在减少，因为没有人“下游”喜欢反复修复相同的问题。当然，确保安全补丁实际上安装在了最终用户系统上，对于开源和闭源软件都是一个问题。

开源的另一个优点是，如果你发现一个问题，你可以立即修复它。这在闭源中是没有对应的。

简而言之，开源软件对安全性的影响在安全社区中仍然是一个主要的争论点，尽管大量知名专家认为它具有更大的安全潜力。

# 2.5. 安全程序类型

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/types-of-programs.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/types-of-programs.html)

许多不同类型的程序可能需要是安全程序（如本书中定义的术语）。一些常见的类型包括：

+   用作远程数据查看器的应用程序。用作查看器（如文字处理程序或文件格式查看器）的程序通常会被要求查看由不受信任的用户发送的远程数据（此请求可能由网络浏览器自动调用）。显然，不应允许不受信任的用户输入导致应用程序运行任意程序。通常支持初始化宏（在数据显示时运行）是不明智的；如果必须这样做，那么你必须创建一个安全沙盒（这是一个复杂且容易出错的任务，几乎从未成功，这也是为什么一开始就不应该支持宏的原因）。小心像缓冲区溢出这样的问题，这些问题在第六章中讨论过，可能会允许不受信任的用户强制查看器运行任意程序。

+   管理员（root）使用的应用程序。这些程序不应信任非管理员可以控制的信息。

+   本地服务器（也称为守护进程）。

+   可通过网络访问的服务器（有时称为网络守护进程）。

+   基于 Web 的应用程序（包括 CGI 脚本）。这些是网络可访问服务器的特殊情况，但由于它们非常普遍，因此值得单独分类。这类程序是通过 Web 服务器间接调用的，Web 服务器可以过滤掉一些攻击，但仍然有许多攻击必须承受。

+   小程序（即下载到客户端以自动执行的程序）。这是 Java 特别著名的事情，尽管其他语言（如 Python）也支持移动代码。这里有几个安全观点；客户端小程序基础设施的实现者必须确保只允许“安全”的操作，而小程序的编写者必须处理恶意主机的难题（换句话说，你通常不能信任客户端）。有一些研究试图处理在恶意主机上运行小程序的问题，但坦白说，我对这些方法的价值持怀疑态度，这个主题足够奇特，所以我在这里没有进一步讨论。

+   setuid/setgid 程序。这些程序由本地用户调用，在执行时，立即获得程序所有者及其/或所有者组的权限。在许多方面，这些是最难保护的程序，因为它们的大多数输入都受不受信任用户的控制，其中一些输入并不明显。

本书将这些不同类型程序的问题合并为一个单一集合。这种方法的缺点是，这里确定的一些问题并不适用于所有类型的程序。特别是，setuid/setgid 程序有许多意外的输入，这里的一些指南仅适用于它们。然而，事情并不那么简单明了，因为某个特定的程序可能跨越这些边界（例如，CGI 脚本可能是 setuid 或 setgid，或者配置成具有相同效果的方式），而且一些程序被分成几个可执行文件，每个可执行文件都可以被视为不同的“类型”的程序。考虑所有这些程序类型的优点是，我们可以考虑所有问题，而无需试图将不适当的类别应用于程序。正如将看到的，许多原则适用于所有需要被保护的程序。

这本书在某种程度上偏向于用 C 语言编写的程序，并对其他语言如 C++、Perl、PHP、Python、Ada95 和 Java 做了一些注释。这是因为 C 语言是 Unix-like 系统上实现安全程序最常用的语言（除了 CGI 脚本，它们通常使用 Perl、PHP 或 Python 等语言）。此外，大多数其他语言的实现都调用了 C 库。这并不是说 C 语言在实现这一目的上“最好”，这里描述的大多数原则无论使用哪种编程语言都适用。

# 2.6. 偏执是一种美德

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/paranoia.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/paranoia.html)

编写安全程序的主要困难在于，编写它们需要不同的思维方式，简而言之，就是偏执的思维方式。原因是错误的（也称为缺陷或错误）的影响可能非常不同。

正常的非安全程序有很多错误。虽然这些错误是不受欢迎的，但这些错误通常涉及罕见或不寻常的情况，如果用户偶然遇到一个错误，他们将来会尽量避免以这种方式使用该工具。

在安全程序中，情况相反。某些用户会故意寻找并造成罕见或不寻常的情况，希望这样的攻击能给他们带来不应得的特权。因此，在编写安全程序时，偏执是一种美德。

# 2.7. 为什么我写这份文档？

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/why-write.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/why-write.html)

我被问到的其中一个问题是“你为什么写这本书”？以下是我的回答：在过去的几年里，我发现许多应用开发者似乎一直在重复犯同样的安全错误。审计员逐渐发现了问题，但最好是在代码中一开始就避免这些问题。我相信问题的一部分是，没有单一、明显的地方让开发者去获取如何避免已知陷阱的信息。这些信息是公开可用的，但往往很难找到，过时，不完整，或者有其他问题。大多数此类信息根本不讨论 Linux，尽管它已经被广泛使用！这导致了答案：我编写这本书的目的是希望未来的软件开发者不会重复过去的错误，从而构建更安全的系统。你可以在[`www.linuxsecurity.com/feature_stories/feature_story-6.html`](http://www.linuxsecurity.com/feature_stories/feature_story-6.html)看到对此的更深入讨论。

一个相关的问题是“你为什么写自己的书而不是仅仅参考其他文档”？有几个答案：

+   这部分信息大部分是分散的；将关键信息放在一个有组织的文档中，使其更容易使用。

+   其中一些信息并非为程序员编写，而是为管理员或用户编写的。

+   大部分可用的信息都强调了可移植的结构（在所有类 Unix 系统上都能工作的结构），但并未讨论 Linux。通常最好为了避免可移植性问题而避免使用 Linux 特有的功能，但有时 Linux 特有的功能确实能帮助提高安全性。即使希望实现非 Linux 的可移植性，当在 Linux 上运行时，你可能仍然希望支持 Linux 特有的功能。而且，通过强调 Linux，我可以包括对那些针对 Linux 但可能不适用于其他人的信息的引用。

# 2.8. 设计和实现指南的来源

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/sources-of-guidelines.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/sources-of-guidelines.html)

几份文档有助于描述如何编写安全程序（或者，也可以说是如何发现现有程序中的安全漏洞），并且是本书其余部分所强调的指南的基础。

对于通用服务器和 setuid/setgid 程序，有许多有价值的文档（尽管在没有相关参考的情况下，一些文档可能难以找到）。

Matt Bishop [1996, 1997] 在这个主题上开发了几个极其有价值的论文和演示文稿，实际上他有一个专门针对这个主题的网页，网址是 [`olympus.cs.ucdavis.edu/~bishop/secprog.html`](http://olympus.cs.ucdavis.edu/~bishop/secprog.html)。AUSCERT 发布了一个编程检查清单 [[AUSCERT 1996]](ftp://ftp.auscert.org.au/pub/auscert/papers/secure_programming_checklist)，部分基于 Garfinkel 和 Spafford 的书中第二十三章讨论如何编写安全的 SUID 和网络程序 [[Garfinkel 1996]](http://www.oreilly.com/catalog/puis)。[Galvin [1998a]](http://www.sunworld.com/swol-04-1998/swol-04-security.html) 描述了一个简单的流程和检查清单，用于开发安全程序；他后来在 [Galvin [1998b]](http://www.sunworld.com/sunworldonline/swol-08-1998/swol-08-security.html) 中更新了检查清单。[Sitaker [1999]](http://www.pobox.com/~kragen/security-holes.html) 为“Linux 安全审计”团队提供了一份需要搜索的问题清单。[Shostack [1999]](http://www.homeport.org/~adam/review.html) 定义了另一个用于审查安全敏感代码的检查清单。NCSA [[NCSA]](http://www.ncsa.uiuc.edu/General/Grid/ACES/security/programming) 提供了一套简洁但有用的安全编程指南。其他有用的信息来源包括 *Secure Unix Programming FAQ* [[Al-Herbish 1999]](http://www.whitefang.com/sup/)、*Security-Audit’s Frequently Asked Questions* [[Graham 1999]](http://lsap.org/faq.txt) 和 [Ranum [1998]](http://www.clark.net/pub/mjr/pubs/pdf/)。一些推荐必须谨慎对待，例如，BSD setuid(7) 手册页 [[Unknown]](http://www.homeport.org/~adam/setuid.7.html) 推荐使用 access(3)，但没有提到通常伴随它的危险竞争条件。Wood [1985] 在其“程序员安全”章节中提供了一些有用但过时的建议。[Bellovin [1994]](http://www.research.att.com/~smb/talks) 包括有用的指南和一些具体示例，例如如何重构 ftpd 实现以使其更简单、更安全。FreeBSD 提供了一些指南 [FreeBSD [1999]](http://www.freebsd.org/security/security.html) [[Quintero 1999]](http://developer.gnome.org/doc/guides/programming-guidelines/book1.html) 主要关注 GNOME 编程指南，但包括一个关于安全考虑的章节。[Venema 1996](http://www.fish.com/security/murphy.html) 提供了一些编程安全程序时常见的错误（广为人知或可预测的密码、用恶意数据烧伤自己、用户可访问数据中的秘密以及依赖于其他程序）的详细讨论（附带示例）。[[Sibert 1996]](http://www.fish.com/security/maldata.html) 描述了恶意数据引起的威胁。Michael Bacarella 的文章 [The Peon’s Guide To Secure System Development](http://m.bacarella.com/papers/secsoft/html) 提供了一套很好的简短指南。

有许多文档提供了使用通用网关接口（CGI）与网络接口的程序的安全指南。这些包括 [Van Biesbrouck [1996]](http://www.csclub.uwaterloo.ca/u/mlvanbie/cgisec), [Gundavaram [unknown]](http://language.perl.com/CPAN/doc/FAQs/cgi/perl-cgi-faq.html), [[Garfinkle 1997]](http://webreview.com/wr/pub/97/08/08/bookshelf) [Kim [1996]](http://www.eekim.com/pubs/cgibook), [Phillips [1995]](http://www.go2net.com/people/paulp/cgi-security/safe-cgi.txt), [Stein [1999]](http://www.w3.org/Security/Faq/www-security-faq.html), [[Peteanu 2000]](http://members.home.net/razvan.peteanu), 和 [[Advosys 2000]](http://advosys.ca/tips/web-security.html)。

有许多特定于语言的文档，这些文档在本书的语言特定部分中进一步讨论。例如，Perl 发行版包括 [perlsec(1)](http://www.perl.com/pub/doc/manual/html/pod/perlsec.html)，它描述了如何更安全地使用 Perl。位于 [`www.cs.princeton.edu/sip`](http://www.cs.princeton.edu/sip) 的安全互联网编程网站对计算机安全问题感兴趣，但专注于移动代码系统，如 Java、ActiveX 和 JavaScript；Ed Felten（其创始人之一）合著了一本关于保护 Java 的书 ([[McGraw 1999]](http://www.securingjava.com))，在 第 10.6 节 中进行了讨论。Sun 的安全代码指南主要针对 Java 和 C 提供了一些指南；它可在 [`java.sun.com/security/seccodeguide.html`](http://java.sun.com/security/seccodeguide.html) 获取。

Yoder [1998] 收集了一系列在处理应用程序安全时可以使用的模式。它并不是一套具体的指南，而是一套编程中常用的模式，可能对你有用。Schmoo 团队维护了一个网页，链接到有关如何编写安全代码的信息 [`www.shmoo.com/securecode`](http://www.shmoo.com/securecode)。

还有许多文档从另一个方向描述了这个问题（即，“如何破解一个系统”）。一个例子是 McClure [1999]，在互联网上有大量从这个角度的材料。还有关于计算机体系结构的更普遍的文档，讨论了如何开发攻击来利用它们，例如 [LSD 2001]。蜜网项目一直在收集有关攻击者实际攻击方式的信息（包括统计数据）；更多信息请访问他们的网站 [`project.honeynet.org`](http://project.honeynet.org)。[通过示例进行不安全编程](http://community.corest.com/~gera/InsecureProgramming/) 提供了一系列不安全的程序，旨在用作练习攻击不安全程序的练习。

关于现有程序中已识别出的漏洞，也有大量信息。这可以是一组有用的“不要这样做”的例子，尽管从大量具体例子中提取更普遍的指导方针需要付出努力。有一些邮件列表讨论安全问题；其中最著名的是 [Bugtraq](http://SecurityFocus.com/forums/bugtraq/faq.html)，它除了其他事情外，还开发了一个漏洞列表。CERT 协调中心（CERT/CC）是互联网安全问题的主要报告中心，负责报告漏洞。CERT/CC 不时发布警告，提供对严重安全问题和其影响的描述，以及获取补丁或绕过细节的说明；更多信息请参阅 [`www.cert.org`](http://www.cert.org)。请注意，最初 CERT 是一个小型计算机紧急响应小组，但现在官方上“CERT”并不代表任何含义。能源部的 [计算机事件咨询能力（CIAC）](http://ciac.llnl.gov/ciac) 也报告漏洞。这些不同的团体可能会识别出相同的漏洞，但使用不同的名称。为了解决这个问题，MITRE 支持通用漏洞和暴露（CVE）列表，为所有由他人识别的已知漏洞和安全暴露创建一个唯一的标识符（“名称”）；请参阅 [`www.cve.mitre.org`](http://www.cve.mitre.org)。NIST 的 ICAT 是计算机漏洞的可搜索目录，对每个 CVE 漏洞进行分类，以便以后可以搜索和比较；请参阅 [`csrc.nist.gov/icat`](http://csrc.nist.gov/icat)。

本书是对我认为最有用和重要的指导方针的总结。我的目标是编写一本好书，让优秀的程序员阅读后能够相当充分地准备编写一个安全的程序。没有任何单一文件能够真正达到这个目标，但我相信这种尝试是值得的。我的目标是在这两个极端之间找到一个平衡点：“所有可能指导方针的完整列表”（这将是没有尽头的且无法阅读的）和在线上可获得的“各种简短”列表之间（它们很简洁，但省略了许多关键问题）。在不确定的情况下，我会包含指导方针；我相信在这种情况下，将信息提供给所有人这个“一站式”文件是更好的选择。这里呈现的组织是我自己的（每个列表都有自己的不同结构），其中一些指导方针（特别是 Linux 特有的，如关于能力和 FSUID 值）也是我自己的。强烈推荐阅读上面列出的所有参考文档，尽管我意识到对于许多人来说这是不切实际的。

# 2.10\. 文档约定

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/conventions.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/conventions.html)

系统手册页以*名称(编号)*的格式引用，其中*编号*是手册的章节号。表示“不指向任何地方”的指针值称为 NULL；在大多数需要指针的情况下，C 编译器会将整数 0 转换为 NULL 值，但请注意，C 标准中没有任何内容要求 NULL 实际上通过一系列全零位来实现。C 和 C++将字符“\0”（ASCII 0）特别处理，本书中这个值被称为 NIL（这通常被称为“NUL”，但“NUL”和“NULL”听起来很相似）。函数和方法名称始终使用正确的字母大小写，即使这意味着某些句子必须以小写字母开头。我使用“Unix-like”这个术语来表示 Unix、Linux 或其他底层模型与 Unix 非常相似的系统；我不能说 POSIX，因为像 Windows 2000 这样的系统虽然实现了 POSIX 的部分内容，但具有截然不同的安全模型。

攻击者被称为“攻击者”、“黑客”或“对手”，而不是“黑客”。一些记者错误地使用“黑客”一词来代替“攻击者”；这本书避免这种误用，因为许多 Linux 和 Unix 开发者将自己称为“黑客”，在传统上非邪恶的意义上使用这个术语。对于许多 Linux 和 Unix 开发者来说，“黑客”这个术语仍然仅仅意味着一个专家或爱好者，尤其是在计算机方面。确实，有些黑客会进行恶意或侵入性的行为，但许多其他黑客并没有，声称所有黑客都进行恶意活动是不公平的。许多其他词汇表和书籍也指出，并非所有黑客都是攻击者。例如，行业咨询委员会的信息保障（IA）特别兴趣小组（SIG）的[信息保障词汇表](https://web.archive.org/web/20021214041204/http://www.iaconline.org/sig_infoassure.html)将黑客定义为“一个喜欢深入了解计算机和计算机网络内部工作原理的人。这个术语在负面语境中被误用，而应该使用‘黑客’。” [《术语文件》](http://www.catb.org/~esr/jargon)对黑客有一个[长而复杂的定义](http://www.catb.org/~esr/jargon/html/entry/hacker.html)，从“一个喜欢探索可编程系统的细节以及如何扩展其功能的人”开始，与大多数只愿意学习最少必要知识的用户形成对比；它指出，尽管有些人使用这个术语来表示“一个试图通过四处窥探来发现敏感信息的有恶意的人”，但它也指出这个定义已被弃用，并且正确的术语应该是“黑客”。

本书使用的是*逻辑*引号系统，而不是误导性的*排字工*引号系统。这意味着，如果标点符号不是被引用材料的一部分，则引用的信息中不包含任何尾随标点。排字工的引号系统会导致无关字符被放置在引号内；这在诗歌中可能没有影响，但在准确性很重要的情况下却是一个严重问题。排字工的引号系统常常会伪造引语（因为它包括了引语中不存在的标点），在代码或计算机命令中可能会造成灾难性的错误。逻辑引号系统在各种出版物中广泛使用，包括[*《术语文件》*](http://www.catb.org/jargon/html/writing-style.html)、[维基百科](http://en.wikipedia.org/wiki/Wikipedia:Logical_quotation_on_Wikipedia)和美国语言学协会。本书使用标准的美国（而不是英国）拼写。
