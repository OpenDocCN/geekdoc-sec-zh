# 11.1\. 密码

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/passwords.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/passwords.html)

在可能的情况下，不要编写处理密码的代码。特别是，如果应用程序是本地的，尽量依赖用户的正常登录认证。如果应用程序是 CGI 脚本，尽量依赖 web 服务器尽可能提供保护——但请参见以下关于在 web 服务器中处理认证的内容。如果应用程序通过网络进行，尽量避免以明文形式发送密码（如果可能），因为它很容易被网络嗅探器捕获并在以后重用。“使用算法中固定的某个密钥或使用某种遮蔽算法加密”密码基本上与发送明文密码相同。

在通过网络传输密码时，进行加密认证并加密连接。（以下我们将讨论网络认证，这通常使用 SSL/TLS 来实现。）

当实现用户使用密码登录的系统（如许多服务器）时，*永远*不要以原样存储密码（即，永远不要以明文形式存储密码）。今天的一个常见问题是攻击者可能能够短暂地入侵系统，或获取数据备份；在这种情况下，他们可以伪造每个用户账户，至少在该系统上，通常在许多其他系统上。

今天，对于许多用户使用密码登录的系统，最低可接受的方法是使用*包含每个用户盐值且使用专为该目的设计的故意慢速哈希函数的加密哈希*。为了简洁，这些被称为“加盐哈希”（尽管许多人如果只满足前两个标准，也会使用“加盐哈希”这个术语）。让我们简要地看看这意味着什么，以及为什么每个部分都是必要的：

+   加密哈希：加密哈希函数，如 SHA-512，将数据转换为“指纹”，这种指纹很难逆转。如果使用哈希函数，攻击者不能直接看到密码是什么，而必须以某种方式根据指纹确定密码。

+   每个用户的盐值：攻击者可以通过预先对许多常见密码进行哈希处理，然后查看是否有任何密码与预先计算的哈希值匹配来对抗简单的加密哈希。这可以通过为每个用户创建一个额外的随机值，称为“盐”，并将其用作哈希数据的一部分来对抗。这些数据需要为每个用户存储（未加密）。盐值应使用加密伪随机数生成器生成，并且它应该至少有 128 位（根据 NIST SP 800-132）。

+   密钥派生/迭代函数：存储的值应使用密钥派生或密钥拉伸函数创建；这些函数通过多次迭代某些操作而故意变得稍微慢一些。这种缓慢是为了在正常操作中无关紧要，但额外的周期大大阻碍了试图对特定高价值用户账户进行密码猜测的攻击者。密钥派生函数反复使用加密哈希、加密或 HMAC 方法。一个真正常见的密钥派生函数是 PBKDF2（基于密码的密钥派生函数 2）；这是 RSA 实验室的公钥密码学标准（PKCS）#5 v2.0、RFC 2898，以及在美国国家标准与技术研究院（NIST）特别出版物 800-132 中的“基于密码的密钥派生推荐”。然而，PBKDF2 可以在 GPU 和专用硬件上快速实现，特别是 GPU 在今天是广泛可用的。今天，你应该优先考虑像 bcrypt 这样的迭代算法，它旨在更好地对抗使用 GPU 和专用硬件的攻击者。

如果你的应用程序允许用户设置他们的密码，请检查密码并只允许“好的”密码（例如，不在字典中，具有某些最小长度等）。你可能想查看有关如何选择一个好的密码的信息，例如[`consult.cern.ch/writeup/security/security_3.html`](http://consult.cern.ch/writeup/security/security_3.html)。如果你能使用 PAM，那将是个不错的选择，因为它支持可插拔的密码检查器。

# 11.2. 网络认证

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/web-authentication.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/web-authentication.html)

在网络上，通常通过使用 SSL 或 TLS 以及服务器证书来对用户进行身份验证 - 但认证用户身份并不像认证服务器那样简单。SSL 和 TLS 确实支持客户端证书，但在实际使用中存在许多实际问题（例如，网络浏览器不支持单一用户证书格式，用户发现安装它们很困难）。你可以从许多地方了解如何设置数字证书，例如[Petbrain](http://www.petbrain.com/modules.php?op=modload&name=pki&file=index)。使用 Java 或 JavaScript 也有其自身的问题，因为许多用户禁用了它们，一些防火墙过滤掉了它们，而且它们通常运行较慢。在大多数情况下，要求每个用户安装插件也是不切实际的，尽管如果系统仅用于相对较少用户的企业内部网，这可能适用。

如果你正在构建一个企业内部网应用程序，你应该通常使用你的用户使用的任何认证系统。类 Unix 系统倾向于使用 Kerberos、NIS+或 LDAP。你可能还需要处理基于 Windows 的认证方案（这些可以被视为 Kerberos 和 LDAP 的专有变体）。因此，如果你的组织依赖 Kerberos，设计你的系统以使用 Kerberos。由于组织可能会（将！）在一段时间内更改他们的认证系统，因此尽量将认证系统与你的应用程序的其他部分分开。文章[构建和实施单点登录解决方案](http://www-106.ibm.com/developerworks/java/library/wa-singlesign/?ca=dgr-lnxw02CASsso)讨论了一些实现企业内部网单点登录（SSO）的方法。

许多认证技术对于互联网应用来说不起作用或者效果不佳。在某些情况下，一种可行的方法是使用“基本认证”，这基本上是所有浏览器和服务器内置的功能。不幸的是，基本认证发送未加密的密码，这使得密码容易被窃取；基本认证本身仅对无价值信息非常有用。您可以将认证信息存储在用户选择的 URL 中，但对于大多数情况，您绝对不应该这样做——不仅 URL 会在网络上未加密地发送（就像基本认证一样），而且还有太多其他方式可以让这些信息泄露给他人（例如，通过许多浏览器存储的浏览器历史记录日志、代理日志以及通过 Referer:字段发送到其他网站）。您可以使用 SSL/TLS 连接包裹与 Web 服务器的所有通信（这将对其进行加密）；这是安全的（取决于您如何操作），如果您有重要数据，这是必要的，但请注意，这会以性能为代价。您还可以使用“摘要认证”，这会暴露通信，但至少在暴露用于认证用户的底层密码之前会验证用户。摘要认证旨在成为低价值通信的简单部分解决方案，但摘要认证在浏览器和服务器之间并没有广泛地以互操作的方式得到支持。实际上，正如 2002 年 3 月 18 日的 eWeek 文章中提到的，微软的 Web 客户端（Internet Explorer）和 Web 服务器（IIS）错误地实现了标准（RFC 2617），因此无法与其他服务器或浏览器一起工作。由于微软不认为这种错误实现是一个严重问题，因此大多数客户拥有正确工作的程序还需要很长时间。

因此，目前存储网络认证信息最常用的技术是通过 cookies。cookies 并非真正为此目的而设计，但它们可以用来支持认证——但使用它们的方式有很多是错误的，这会创建安全漏洞，所以请小心。有关 cookies 的更多信息，请参阅 IETF RFC 2965，以及关于它们的旧规范。请注意，为了使用 cookies，某些浏览器（例如，Microsoft Internet Explorer 6）可能会坚持要求您有一个隐私配置文件（在服务器的根目录上命名为 p3p.xml）。

注意，有些用户不接受 cookies，所以这个解决方案仍然存在一些问题。如果你想支持这些用户，你应该通过 HTML 表单隐藏字段来回发送这个身份验证信息（因为几乎所有的浏览器都支持它们，无需担心）。你会使用与 cookies 相同的方法——你只是使用不同的技术来发送从用户到服务器的数据。当然，如果你实施这种方法，你需要包括设置以确保这些页面不会被缓存供他人使用。然而，虽然我认为避免 cookies 是更好的选择，但在实践中，这些其他方法通常需要更多的开发工作。由于对于许多应用程序开发人员来说，在大规模上实施这一点非常困难，所以我目前并不强调这些方法。我更愿意描述一个既安全又易于实施的方法，而不是强调那些难以正确实施的方法（无论是开发者还是用户）。然而，如果你可以不费太多力气做到这一点，那么请务必支持使用表单隐藏字段和加密链接（例如，SSL/TLS）发送身份验证信息。与所有 cookies 一样，对于这些 cookies，你应该打开 HttpOnly 标志，除非你有必须能够读取 cookie 的网页浏览器脚本。

Fu [2001] 讨论了网络上的客户端身份验证，以及一个建议的方法，这是我为大多数网站推荐的方法。基本思路是将客户端身份验证分为两部分，即“登录过程”和“后续请求”。在登录过程中，服务器请求用户的用户名和密码，用户提供这些信息，然后服务器回复一个“身份验证令牌”。在后续请求中，客户端（网页浏览器）将身份验证令牌发送到服务器（连同其请求一起）；服务器验证令牌的有效性，如果有效，则处理请求。关于网络身份验证的另一个良好信息来源是 Seifried [2001]。

一些网络身份验证技术的一个严重问题是它们容易受到称为“会话固定”的问题的影响。在会话固定攻击中，攻击者在用户甚至登录目标服务器之前就固定了用户的会话 ID，从而消除了之后获取用户会话 ID 的需要。基本上，攻击者获取了一个账户，然后诱骗另一个用户使用攻击者的账户——通常是通过创建一个特殊的超链接并诱骗用户点击它。描述会话固定的一个好论文是 Mitja Kolsek [2002]的论文（http://www.acros.si/papers/session_fixation.pdf）。你使用的网络身份验证系统应该能够抵御会话固定。

一个涵盖网站身份验证的良好通用清单是[Mark Burnett 在 SecurityFocus 上的文章](http://www.securityfocus.com/infocus/1688)。

## 11.2.1. 网络身份验证：登录

登录过程通常实现为一个 HTML 表单；我建议使用字段名“username”和“password”，这样网络浏览器可以自动执行一些有用的操作。确保密码是通过加密连接发送的（使用 SSL 或 TLS，通过 https:连接）——否则，窃听者可能会收集密码。确保所有密码文本字段在 HTML 中都被标记为密码，这样密码文本就不会对任何可以看到用户屏幕的人来说可见。

如果用户名和密码字段都填写了，不要尝试自动以该用户登录。相反，显示带有用户名和密码字段的登录表单；这可以让用户验证他们是否真的想以该用户身份登录。如果你不这样做，攻击者将能够利用这个弱点执行会话固定攻击。偏执的系统可能只想简单地忽略密码字段，并让用户填写它，但这会干扰浏览器，因为浏览器可以存储用户的密码。

当用户发送用户名和密码时，必须将其与用户账户数据库进行核对。这个数据库不应该“明文”存储密码，因为如果有人获得了这个数据库的副本，他们就会突然获得所有人的密码（而且用户经常重复使用密码）。有些人使用 crypt()来处理这个问题，但 crypt 只能处理小输入，所以我建议使用不同的方法（这是我的方法 - Fu [2001]没有讨论这一点）。相反，用户数据库应该存储用户名、盐和该用户的密码散列。这里的“盐”只是一个随机的字符序列，用于使攻击者即使获得了密码数据库也难以确定密码——我建议使用一个 8 字符的随机序列。它不需要是密码学上随机的，只需与其他用户不同即可。密码散列应该通过连接“服务器密钥 1”，用户的密码和盐，然后运行一个密码学安全的散列算法来计算。服务器密钥 1 是这个服务器独有的密钥——请将其与密码数据库分开。如果有人拥有服务器密钥 1，并且他们也拥有密码数据库，那么他们就可以运行程序来破解用户密码；由于它不需要记忆，它可以是一个长而复杂的密码。

因此，当用户创建他们的账户时，密码会被散列并放置在密码数据库中。当用户尝试登录时，声称的密码会被散列，并与数据库中的散列进行比较（它们必须相等）。当用户更改密码时，他们应该输入旧密码和新密码两次（以确保他们没有输入错误）；再次确保这些密码的字符在屏幕上不可见。

默认情况下，不要使用 cookies 在客户端的网页浏览器上保存密码 - 用户有时可能会使用共享的客户端（比如在一些咖啡馆）。如果您愿意，可以给用户提供在浏览器上“保存密码”的选项，但如果是这样，请确保密码仅通过“安全”连接传输，并确保用户必须明确请求它（不要默认这样做）。

确保页面被标记为不缓存，否则代理服务器可能会将该页面重新提供给其他用户。

用户成功登录后，服务器需要通过 cookie 向客户端发送一个“认证令牌”，这将在下文中描述。

## 11.2.2\. 在 Web 上进行认证：后续操作

用户登录后，服务器会向客户端发送一个包含认证令牌的 cookie，该令牌将从那时起使用。使用单独的认证令牌，这样用户就不需要不断登录，密码也不需要不断来回发送，如果需要的话，可以使用未加密的通信。一个建议的令牌（忽略会话固定攻击）看起来像这样：

```
   exp=t&data=s&digest=m
```

其中 t 是令牌的过期时间（比如，几小时），而数据 s 标识用户（比如，用户名或会话 ID）。摘要是对其他字段的键控摘要。请随意更改“数据”字段的名称以使其更具描述性（例如，用户名和/或 sessionid）。如果您有多个数据字段（例如，既有用户名又有会话 ID），请确保摘要使用您正在验证的所有字段的字段名称和数据值；使用一个模式（比如“%%”，“+”，或“&”）将它们连接起来，这个模式不能出现在任何字段数据值中。如稍后所述，包含用户名是个好主意。键控摘要应该是令牌中其他信息的加密散列，使用不同的服务器密钥 2 进行键控。键控摘要应使用 HMAC-MD5 或 HMAC-SHA1，使用不同的服务器密钥（key2），尽管在某些目的上简单地使用 SHA1 可能就足够了（或者如果风险较低，甚至可以使用 MD5）。Key2 容易受到暴力猜测攻击，因此它应该很长（比如 12+个字符）且难以猜测；它不需要容易被记住。如果这个 key2 被泄露，任何人都可以认证到服务器，但更换 key2 很容易 - 当您更换时，它将简单地强制当前“已登录”的用户重新认证。有关更多详细信息，请参阅傅[2001]。

这种方法存在潜在弱点。我担心傅的方法，如最初描述的那样，对会话固定攻击（来自几个不同方向，这里不想深入讨论）比较脆弱。因此，我现在建议修改傅的方法，并使用这种令牌格式：

```
   exp=t&data=s&client=c&digest=m
```

这与原始的 Fu 方法相同，并且本书的早期版本（在 2002 年 12 月之前）并没有建议这样做。这种修改为客户端添加了一个新的 "client" 字段，以唯一标识客户端当前的地理位置/身份。客户端字段中的数据应该是其他人尝试使用该账户时可能会改变的数据；理想情况下，其新值应该是不可猜测的，尽管在实践中很难实现。理想情况下，客户端字段应该是客户端的 SSL 客户端证书，但目前这个建议很难满足。至少，它应该是用户的 IP 地址（从服务器端感知，并记住为 IPv6 的更长地址进行规划）。不幸的是，这种修改并不能完全防止会话固定攻击（因为如果攻击者能够确定用户会发送的内容，攻击者可能能够向服务器发送请求并说服客户端接受这些值）。然而，它确实增加了对攻击的抵抗力。再次强调，摘要现在必须包括所有其他数据。

这里有一个例子。如果一个用户成功登录到 foobar.com，你可能会将过期日期设定为 2002-12-30T1800（让我们假设我们暂时以 ASCII 文本格式传输），用户名作为 "fred"，客户端会话作为 "1234"，并且你可能会确定客户端的 IP 地址是 5.6.7.8。如果你使用简单的 SHA-1 带密钥的摘要（并且使用前缀密钥的其余数据），服务器密钥 2 的值为 "rM!V^m~v*Dzx"，摘要可以计算如下：

```
  exp=2002-12-30T1800&user=fred&session=1234&client=5.6.7.8
```

通过对服务器密钥 2，例如，运行加密散列代码，然后是数据，可以计算出一个带密钥的摘要；在这种情况下，摘要将是：

```
 101cebfcc6ff86bc483e0538f616e9f5e9894d94
```

从那时起，服务器必须检查过期时间并重新计算此身份验证令牌的摘要，并且只有当摘要正确时才接受客户端请求。如果没有令牌，服务器应该回复用户登录页面（带有一个隐藏的表单字段，显示成功登录后应去的地方）。

在重要屏幕上显示用户名是谨慎的做法，特别是可以帮助防止会话固定攻击。如果用户对他们的用户名有反馈，他们可能会注意到他们没有预期的用户名。如果可能有一个意外的用户名（例如，一个共享同一台机器的家庭），这总是有帮助的。重要屏幕的例子包括上传文件时应保持私密的那些时候。

一个奇怪的实施问题：尽管对于 cookie 的“Expires:”（过期时间）字段的规定允许时区，但结果发现，一些版本的 Microsoft 的 Internet Explorer 没有正确实现 cookie 过期时的时区。因此，您需要始终在 cookie 过期时间中使用 UTC 时间（也称为 Zulu 时间）以实现最大程度的可移植性。通常来说，使用 UTC 时间对于时间值是个好主意，并在必要时将其转换为人类显示的时间，因为这样可以消除其他时区和夏令时问题。

如果你在身份验证令牌中包含一个 sessionid，你可以进一步限制访问。你的服务器可以“跟踪”用户在特定会话中看到的页面，并且只允许从那个点访问其他适当的页面（例如，只有那些直接链接到那些页面）。例如，如果一个用户被授予访问 foo.html 页面的权限，并且 foo.html 页面指向资源 bar1.jpg 和 bar2.png，那么对 bar4.cgi 的访问可以被拒绝。你甚至可以结束会话，但只有当身份验证信息有效时才这样做（否则，这会让攻击者有可能对其他用户造成拒绝服务攻击）。这将在一定程度上限制攻击者的访问权限，即使他们成功劫持了会话，尽管显然一个有时间和身份验证令牌的攻击者可以像普通用户一样“遍历”链接。

一个决定是是否要求将身份验证令牌和/或数据通过安全连接（例如，SSL）发送。如果你在明文（非安全）中发送身份验证令牌，拦截令牌的人可以执行用户直到过期时间可以做的任何事情。另外，当你通过未加密的链接发送数据时，存在攻击者未察觉更改的风险；如果你担心有人可能会在途中更改数据，那么你需要对传输中的数据进行身份验证。仅加密本身并不能保证身份验证，但它确实使得损坏更容易被发现，并且典型的库可以在 TLS/SSL 连接中支持加密和身份验证。一般来说，如果你正在加密消息，你也应该对其进行身份验证。如果你的需求不同，一个替代方案是创建两个身份验证令牌 - 一个仅用于“安全”连接的重要操作，而另一个用于不太重要的操作。确保用于“安全”连接的令牌被标记，以便仅使用安全连接（通常是加密的 SSL/TLS 连接）。如果用户实际上没有区别，身份验证令牌可以完全省略“数据”。

再次强调，确保带有此身份验证令牌的页面不被缓存。还有其他合理的方案；本文的目的是提供至少一种安全解决方案。可能有许多不同的变体。

## 11.2.3. 网络上的身份验证：登出

你应该始终为用户提供一种“注销”机制——这对于使用共享浏览器的客户（比如在图书馆）特别有帮助。你的“注销”例程的任务很简单——只需取消客户端的认证令牌。

# 11.3. 随机数

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/random-numbers.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/random-numbers.html)

在许多情况下，安全程序必须生成不可被攻击者猜测的“随机”数字。例如包括会话密钥、公钥或私钥、对称密钥、许多协议中使用的非 ces 和 IVs、盐等。理想情况下，你应该使用真正的随机数据源来生成随机数，例如基于放射性衰变（通过精确的盖革计数器点击时间）、大气噪声或电路中的热噪声的值。一些计算机有一个作为真实随机值生成器的硬件组件，如果可用，你应该使用它。

然而，大多数计算机没有生成真正随机值的硬件，所以在大多数情况下，你需要一种足够随机的方法来生成随机数，以至于对手无法预测它。一般来说，这意味着你需要三样东西：

+   一个“不可预测”的状态；通常这是通过测量底层设备（如按键、磁盘驱动器臂抖动等）的时间差异来完成的，这样攻击者就无法控制。

+   一个密码学上强大的伪随机数生成器（PRNG），它使用状态来生成“随机”数字。

+   一个大数量的比特（既包括种子也包括用于结果的值）。如果你只有几个可能的值，那么拥有一个强大的 PRNG 是没有意义的，因为这会使攻击者使用蛮力攻击变得容易。然而，所需的比特数取决于具体情况，因为这些通常用作加密密钥，所以适用于密钥的正常规则适用。对于一个对称密钥（结果），我会使用至少 112 比特（3DES），128 比特会更好一些，160 比特或更多会更安全。

通常，伪随机数生成器（PRNG）使用状态来生成一些值，然后使用其中的一些值和其他不可预测的输入来更新状态。攻击这些系统有众多方法。例如，如果攻击者可以控制或查看状态（或其部分）的输入，攻击者可能能够确定你所谓的“随机”数字。

对于伪随机数生成器（PRNGs）的一个真实危险是，大多数计算机语言库包括大量不适用于安全目的的伪随机数生成器（PRNGs）。让我再说一遍：*不要使用典型的随机数生成器进行安全目的*。典型的库 PRNGs 旨在用于模拟、游戏等；它们在安全功能（如密钥生成）中并不足够随机。大多数非加密库 PRNGs 是“线性同余生成器”的某种变体，其中“下一个”随机值计算为"(aX+b)modm"（其中 X 是前一个值）。好的线性同余生成器速度快，具有有用的统计特性，使其适用于其预期用途。这类 PRNGs 的问题在于攻击者可以轻易地推断出未来的值（尽管它们可能看起来是随机的）。用于快速生成随机数的其他算法，如二次生成器和立方生成器，也已被破解[Schneier 1996]。简而言之，你必须在安全应用程序中生成随机数时使用加密强 PRNGs - 普通的随机数库是不够的。

未能正确生成真正的随机值用于密钥已导致一系列问题，包括 Kerberos、X 窗口系统和 NFS 中的漏洞[Venema 1996]。

如果可能的话，你应该使用专为创建加密安全随机值而设计的系统服务（通常由操作系统提供）。例如，Linux 内核（自 1.3.30 版本起）包含一个随机数生成器，这对于许多安全目的来说是足够的。这个随机数生成器将来自设备驱动程序和其他来源的环境噪声收集到一个熵池中。当作为/dev/random 访问时，只有在熵池中估计的噪声位数内才会返回随机字节（当熵池为空时，调用会阻塞，直到收集到更多的环境噪声）。当作为/dev/urandom 访问时，即使熵池耗尽，也会返回请求的所有字节。如果你在 Linux 上使用随机值进行加密目的（例如，生成密钥），请使用/dev/random。*BSD 系统也包含/dev/random。拥有 SUNWski 包的 Solaris 用户也有/dev/random。请注意，如果可用硬件随机数生成器及其驱动程序已安装，则将使用它。更多信息可在系统文档 random(4)中找到。

在其他系统上，您需要找到另一种方法来获取真正的随机结果。对于其他类 Unix 系统，一个可能的选择是熵收集守护进程（Entropy Gathering Daemon，简称 EGD），它监控系统活动并将其散列成随机值；您可以在[`www.lothar.com/tech/crypto`](http://www.lothar.com/tech/crypto)获取它。您可以考虑在伪随机数生成器的输出上使用加密散列函数。通过使用散列算法，即使伪随机数生成器最终是可猜测的，这也意味着攻击者现在还必须破解散列函数。

如果您必须自己实现一个强大的伪随机数生成器，一个选择是使用加密强度高（且不受专利限制）的 Yarrow 算法；您可以在[`www.counterpane.com/yarrow.html`](http://www.counterpane.com/yarrow.html)了解更多关于 Yarrow 的信息。一些其他伪随机数生成器可能很有用，但许多广泛使用的伪随机数生成器都有已知的弱点，这些弱点可能或可能不会影响您的应用程序。在您自己实现伪随机数生成器之前，请查阅文献，例如[Kelsey 1998]和[McGraw 2000a]。您还应该检查[IETF RFC 1750](http://www.ietf.org/rfc/rfc1750.txt)。NIST 有一些有用的信息；请参阅[NIST 出版物 800-22](http://csrc.nist.gov/publications/nistpubs/800-22/sp-800-22-051501.pdf)和[NIST 勘误表](http://csrc.nist.gov/publications/nistpubs/800-22/errata-sheet.pdf)。您还应该了解[diehard 测试](http://stat.fsu.edu/~geo/diehard.html)。您可能还想检查标题为“Intel 如何检查其伪随机数生成器”的论文，但不幸的是，这篇论文现在似乎不可用。

# 11.4. 在用户内存中特别保护秘密（密码和密钥）

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/protect-secrets.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/protect-secrets.html)

如果您的应用程序必须处理密码或非公开密钥（例如会话密钥、私钥或秘密密钥），尽量隐藏它们并在使用后立即覆盖它们，以使它们暴露最小。塔尔·加芬克尔（Tal Garfinkel）已经表明，许多程序未能这样做，这是一个严重的问题。[`www.newscientist.com/news/news.jsp?id=ns99995064`](http://www.newscientist.com/news/news.jsp?id=ns99995064)

类似 Linux 这样的系统支持 mlock() 和 mlockall() 调用来防止内存被分页到磁盘（因为某人可能稍后从交换文件中获取到密钥）。请注意，在 Linux 2.6.9 版本之前，这是一个特权系统调用，它会产生自己的问题（我是否应该授予程序超级用户权限以便它可以调用 mlock，如果它不需要这些权限的话？）。Linux 内核 2.6.9 及更高版本允许用户进程锁定一定量的内存，因此在这种情况下，始终使用 mlock() 来锁定用于存储私有密钥和/或有价值密码的内存。将锁定内存的数量保持在最低限度，将其视为一种宝贵资源。如果一个系统允许用户锁定大量内存，那么很容易使整个系统停止运行（通过强制其耗尽内存），这就是为什么这种请求是特权或严格受限的原因。

此外，如果你的程序处理这样的秘密值，请确保禁用创建核心转储（通过 ulimit）。否则，攻击者可能能够停止程序并在数据转储中找到秘密值。

警惕——通常，进程可以通过调试器的调用来监控其他进程（例如，通过 ptrace(2) 和 /proc 伪文件系统）。内核通常在进程设置为 setuid 或 setgid 时保护这些监控例程（在少数古老的系统中，如果没有，实际上没有很好的防御方法，除了升级）。因此，如果你的进程管理秘密值，你可能应该将其设置为 setgid 或 setuid（到一个不同的非特权组或用户），以强制性地抑制这种监控。除非你需要它设置为 setuid，否则使用 setgid（因为它授予较少的权限）。

然后还有能够实际覆盖值的问题，这通常与语言和编译器有关。在许多语言中，你需要确保将此类信息存储在可变位置，然后覆盖这些位置。例如，在 Java 中，不要使用类型 String 来存储密码，因为 String 是不可变的（它们将在垃圾回收后重新使用，可能是在遥远的未来）。相反，在 Java 中使用 char[] 来存储密码，这样它就可以立即被覆盖。在 Ada 中，使用类型 String（字符数组），而不是类型 Unbounded_String，以确保你可以控制内容。

在许多语言（包括 C 和 C++）中，请注意编译器不会优化掉用于覆盖值的“死代码”——因为在这种情况下它不是死代码。许多编译器，包括许多 C/C++编译器，会删除不再使用的存储写入操作——这通常被称为“死存储消除”。不幸的是，如果写入操作确实是覆盖秘密值的操作，这意味着看似正确的代码将被无声地丢弃。Ada 提供了 pragma Inspection_Point；在清除内存的代码之后放置它，这样就可以确保包含秘密的对象确实被删除（并且覆盖不会被优化掉）。

安迪·波利亚科夫（Andy Polyakov）于 2002 年 11 月 7 日在 Bugtraq 论坛上发表的一篇文章报道，C/C++编译器 gcc 版本 3 或更高、SGI MIPSpro 以及微软编译器消除了简单内联调用 memset 以覆盖秘密的行为。这是 C 和 C++标准所允许的。其他 C/C++编译器（如版本低于 3 的 gcc）在所有优化级别上保留了 memset 的内联调用，这表明问题与编译器有关。仅仅声明目标数据是易变的并不能在所有编译器上有所帮助；MIPSpro 和微软编译器都忽略了简单的“易变化”。仅仅“触摸”秘密数据的第一个字节也无济于事；他发现 MIPSpro 和 GCC>=3 巧妙地使第一个字节为空，其余部分保持不变（这实际上相当聪明——问题是编译器的聪明才智干扰了我们的目标）。似乎在所有平台上都有效的一种方法是编写自己的 memset 实现，其中第一个参数具有内部“易变化”（此代码基于迈克尔·霍华德（Michael Howard）提出的[解决方案](http://online.securityfocus.com/archive/82/298061/2002-10-27/2002-11-02/0)）：

```
  void *guaranteed_memset(void *v,int c,size_t n)
  { volatile char *p=v; while (n--) *p++=c; return v; }
```

然后将此定义放入外部文件中，以强制函数为外部函数（在相应的.h 文件中定义函数，并在调用者中#include 该文件，就像通常那样）。这种方法在任何优化级别上似乎都是安全的（即使函数被内联）。

# 11.5\. 密码学算法和协议

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/crypto.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/crypto.html)

通常，密码学算法和协议对于保持系统安全是必要的，尤其是在通过不受信任的网络（如互联网）进行通信时。在可能的情况下，使用密码学技术来验证信息和保护信息不被泄露（但不要假设简单的加密会自动进行验证）。通常，您需要使用一系列可用的工具来保护您的应用程序。

对于背景信息和代码，你可能需要查看经典文本“应用密码学” [Schneier 1996]。新闻组“sci.crypt”有一系列常见问题解答；你可以在许多地方找到它们，包括[`www.landfield.com/faqs/cryptography-faq`](http://www.landfield.com/faqs/cryptography-faq)。Linux 特定的资源包括[`marc.mutz.com/Encryption-HOWTO/`](http://marc.mutz.com/Encryption-HOWTO/)上的 Linux 加密 HOWTO。关于协议如何使用基本算法的讨论可以在[Opplinger 1998]中找到。关于如何在协议中应用密码学的一组有用论文可以在[Stallings 1996]中找到。以下只是几点评论；这些领域相当专业，在其他地方有更全面的介绍。

密码学协议和算法很难正确实现，因此不要自行创建。相反，在可能的情况下，使用广泛使用、经过深入分析和被接受为安全的协议和算法。当你必须创建任何东西时，应进行广泛的公众审查，并确保专业安全分析师检查其中是否存在问题。特别是，除非你是密码学专家，知道自己在做什么，并计划花费数年时间对算法进行专业审查，否则不要自行创建加密算法。创建有效的加密算法（如果有的话）是专家的工作。

许多算法都获得了专利；即使所有者目前允许“免费使用”，没有签署的合同，他们以后总是可以改变主意，这会让你面临极大的风险。一般来说，避免所有专利算法——在大多数情况下，有未获专利的方法，技术上至少同样好或更好，这样做可以避免大量法律问题。

另一个复杂的问题是许多国家以某种方式规范或限制密码学。有关法律问题的调查可以在“密码法调查”网站上找到，[`rechten.kub.nl/koops/cryptolaw/`](http://rechten.kub.nl/koops/cryptolaw/).

通常，你的软件应该提供一种拒绝“太小”密钥的方法，并让用户设置“太小”是什么。对于 RSA 密钥，512 位太小，不能使用。越来越多的证据表明，RSA 密钥的 1024 位也不够；Bernstein 提出了简化 RSA 暴力破解的技术，基于它的其他工作（如 Shamir 和 Tromer 的“使用 TWIRL 设备分解大数”）现在表明，1024 位密钥可以在一年内被价值 1000 万美元的设备破解。如果你真的想要一个安全的系统，你可能希望将 2048 位设为 RSA 密钥的最小值。有关 RSA 的更多信息，请参阅[RSA 对 Bernstein 工作的评论](http://www.rsasecurity.com/rsalabs/technotes/bernstein.html)。关于密钥长度和其他一般密码学算法问题的更广泛讨论，请参阅[NIST 2001 年 11 月的密钥管理研讨会](http://csrc.nist.gov/encryption/kms/key-management-guideline-(workshop).pdf).

## 11.5.1\. 密码学协议

当你需要一个安全协议时，尽量使用符合标准的协议，例如 IPSec、SSL（即将成为 TLS）、SSH、S/MIME、OpenPGP/GnuPG/PGP 和 Kerberos。每个协议都有其优缺点；它们在功能上有些重叠，但通常用于不同的领域：

+   互联网协议安全（IPSec）。IPSec 在 IP 数据包级别提供加密和/或认证。然而，IPSec 通常只保证两个通信主机的真实性，而不是用户。作为一个实际问题，IPSec 通常需要操作系统（并非所有操作系统都实现）的低级支持以及一个必须配置的额外密钥环服务器。由于 IPSec 可以用作“隧道”来保护属于多个用户和多个主机的数据包，因此它特别适用于构建虚拟专用网络（VPN）和连接远程机器。到目前为止，它很少用于保护个人客户端到服务器的通信。新的互联网协议版本 IPv6 自带了 IPSec“内置”，但 IPSec 也可以与更常见的 IPv4 协议一起工作。请注意，如果你使用 IPSec，不要在没有认证的情况下使用加密模式，因为认证也充当完整性保护。

+   安全套接字层（SSL）/TLS。SSL/TLS 在 TCP 上工作，并通过 TCP 隧道传输其它协议，增加加密、服务器认证以及可选的客户端认证（但使用 SSL/TLS 认证客户端需要客户端配置 X.509 客户端证书，这很少做）。SSL 版本 3 被广泛使用；TLS 是对 SSL 的后期调整，增强了其安全性并提高了其灵活性。目前正在进行从 SSLv3 到 TLS 的缓慢过渡，因为实现可以轻松尝试使用 TLS，然后在用户干预之前回退到 SSLv3。不幸的是，一些不良的 SSLv3 实现导致回退出现问题，因此你可能需要一个首选项设置，允许用户在必要时跳过使用 TLS。不要使用 SSL 版本 2，它有一些严重的安全弱点。

    SSL/TLS 是保护 http（网页）交易的主要方法。每次你使用“https://”URL 时，你就是在使用 SSL/TLS。经常使用 SSL/TLS 的其它协议包括 POP3 和 IMAP。SSL/TLS 通常使用一个与未加密端口不同的单独的 TCP/IP 端口号，这让 IETF 有些不高兴（因为它消耗了双倍的端口；有解决这个问题的方法）。在程序中使用 SSL 相对容易，因为大多数库实现都允许程序员使用类似于标准套接字操作的操作，如 SSL_connect()、SSL_write()、SSL_read()等。一个广泛使用的开源/自由软件 SSL 实现（以及其它功能）是 OpenSSL，可在[`www.openssl.org`](http://www.openssl.org)找到。

+   OpenPGP 和 S/MIME。有两种竞争性、本质上不兼容的标准用于保护电子邮件：OpenPGP 和 S/MIME。OpenPHP 基于 PGP 应用程序；一个 OSS/FS 实现是来自 [`www.gnupg.org`](http://www.gnupg.org) 的 GNU Privacy Guard。目前，它们的证书通常不可互换；正在进行工作以修复这个问题。

+   SSH。SSH 是在互联网上确保“远程终端”的主要方法，它还包括用于隧道 X Windows 会话的方法。然而，它已被扩展以支持单点登录和 TCP 流的一般安全隧道，因此它通常也用于确保其他数据流（如 CVS 访问）。SSH 最受欢迎的实现是 OpenSSH [`www.openssh.com`](http://www.openssh.com)，这是一个 OSS/FS。SSH 的典型用途允许客户端验证服务器确实是服务器，然后用户输入密码以验证用户身份（密码被加密并发送到另一系统进行验证）。SSH 的当前版本可以存储私钥，使用户不必每次都输入密码。为了防止中间人攻击，SSH 记录与服务器通信时的密钥信息；这意味着 SSH 的典型使用在第一次连接期间容易受到中间人攻击，但之后可以检测到问题。相比之下，SSL 通常使用证书颁发机构，这消除了第一次连接问题，但需要向证书颁发机构进行特殊设置（以及支付！）。

+   Kerberos。Kerberos 是一种用于单点登录和对用户进行中央认证和密钥分发服务器的身份验证的协议。Kerberos 通过向经过身份验证的用户提供“票据”，授予他们访问网络上各种服务的权限来工作。当客户端随后联系服务器时，服务器可以验证这些票据。Kerberos 是在局域网中确保和支持身份验证的主要方法，以及建立共享密钥（因此，需要与其他算法一起使用以实现通信的实际保护）。请注意，要使用 Kerberos，客户端和服务器都必须包含使用它的代码，并且由于并非每个人都有 Kerberos 设置，这必须是可选的——这使某些程序中使用 Kerberos 复杂化。然而，Kerberos 被广泛使用。

许多这些协议允许你选择多种不同的算法，所以你仍然需要为算法（例如，加密）选择合理的默认值。

## 11.5.2. 对称密钥加密算法

在许多国家，加密算法的实现的使用、出口和/或进口受到限制，法律可能会迅速变化。在尝试使用密码学构建应用程序之前，找出规则是什么。

对于密钥（大量数据）加密算法，应仅使用公开出版并经年累月攻击考验的加密算法，并检查其专利状态。我建议使用新的高级加密标准（AES），也称为 Rijndahl——许多密码学家对其进行了分析，并未发现任何严重弱点，我相信它已经经过了足够的分析，现在可以信赖。2002 年 8 月，研究人员 Fuller 和 Millar 发现了一种密码的数学性质，虽然这不是一种攻击，但可能被利用并转化为攻击（这种方法实际上可能对某些其他算法也有严重影响）。然而，迄今为止，全球范围内的深入分析尚未提供 AES 实际上存在漏洞的严重证据（参见[Landau 2004]了解更多关于 Rijndael 的技术信息）。当然，始终关注未来的工作总是值得的。AES 的一个很好的替代算法是 Serpent 算法，它稍微慢一些，但非常难以攻击。对于许多应用来说，三重 DES 是一个非常优秀的加密算法；它有一个相当长的密钥（112 位），没有专利问题，并且有非常长的抵御攻击的历史（它在公开文献中比任何其他具有合理密钥长度的加密算法都经受住了更长时间的攻击，因此，如果正确实现，它可能是最安全的公开可用对称加密算法）。然而，当在软件中实现时，三重 DES 非常慢，因此三重 DES 可以被认为是“最安全但最慢的”。Twofish 似乎是一个很好的加密算法，但还有一些悬而未决的问题——Sean Murphy 和 Fauzan Mirza 表明，Twofish 具有许多学者担忧的特性（尽管到目前为止还没有人能够利用这些特性）。MARS 对“新和独特”的攻击具有高度抵抗力，但它更复杂，在小容量智能卡上不实用。目前，我会避免使用 Twofish——它很可能永远不会被利用，但很难确定，而且还有没有这些担忧的替代算法。不要使用 IDEA——它受到美国和欧洲专利的限制。不要使用诸如使用常数或常数字符串的 XOR、ROT（旋转）方案、Vinegere 密码等愚蠢的算法——这些算法可以用今天的计算机轻易破解。不要使用“双 DES”（使用两次 DES）——这会受到三重 DES 避免的“中间人”攻击。无论如何，您的协议应该支持多种加密算法；这样，当一种加密算法被破解时，用户可以切换到另一种算法。

对于对称密钥加密（例如，用于大量加密），如果你想在 2016 年之前保持信息保密（每增加 18 个月的安全时间增加一个比特）[Blaze 1996]，不要使用小于 90 比特的密钥长度。对于加密无价值的数据，旧的 DES 算法有一些价值，但使用现代硬件，使用暴力破解很容易破解 DES 的 56 位密钥。如果你使用 DES，不要仅仅使用 ASCII 文本密钥作为密钥——奇偶校验位位于最低位（而不是最高位），因此大多数 DES 算法将使用对手熟知的密钥值进行加密；相反，创建密钥的散列并正确设置奇偶校验位（并注意加密例程的错误报告）。所谓的“可导出”加密算法只有 40 比特的有效密钥长度，实际上毫无价值；在 1996 年，攻击者可以用 10000 美元在 12 分钟内破解这样的密钥，或者使用闲置的计算机时间在几天内破解，在两种情况下，破解时间每 18 个月减半。

区块加密算法可以在多种模式下使用，例如“电子密码本”（ECB）和“密文块链接”（CBC）。在几乎所有情况下，应使用 CBC，而不要使用 ECB 模式——在 ECB 模式下，同一块数据在流中总是返回相同的结果，这通常足以揭示加密的内容。许多模式，包括 CBC 模式，都需要一个“初始化向量”（IV）。IV 不需要是保密的，但它需要不可预测，以免被攻击者预测。不要在会话之间重复使用 IV——每次开始会话时都使用一个新的 IV。

有许多不同的流加密算法，但其中许多都有专利限制。我知道 WAKE 没有专利或技术问题。RC4 是 RSA 数据安全公司的商业机密；自那以后已经泄露，我知道没有真正的法律障碍阻止其使用，但 RSA 数据安全公司经常威胁要对使用 RC4 的用户提起诉讼（RSA 数据安全公司能做什么并不完全清楚，但毫无疑问，他们可以在毫无价值的诉讼案件中缠住用户）。如果你使用 RC4，请按预期使用它——特别是，始终丢弃它生成的第一个 256 字节，否则你将容易受到攻击。SEAL 由 IBM 拥有专利——所以不要使用它。SOBER 也是专利产品；专利所有者声称，如果请求许可，它将允许许多免费使用，但这为以后的使用设置了障碍。更有趣的是，块加密算法可以用于将它们转换为流密码的模式，希望使用流密码的用户应考虑这种方法（你将能够在更多公开可用的算法之间进行选择）。

## 11.5.3. 公钥算法

对于公钥加密（用于签名和发送密钥等），只有少数广泛部署的算法。其中最广泛使用的算法之一是 RSA；RSA 的算法在美国获得了专利，但该专利已于 2000 年 9 月到期，因此 RSA 可以免费使用。切勿使用 RSA 直接解密或签名攻击者直接给出的原始值并暴露结果，因为这可能会暴露私钥（在实践中这不是问题，因为大多数协议都涉及签名用户计算出的哈希值——而不是原始值——或者不暴露结果）。切勿多次解密或签名相同的原始值（原始值可能会被暴露）。这两种情况都可以通过始终添加随机填充（PGP 就是这样做的）来解决——通常的方法称为最优非对称加密填充（OAEP）。

Diffie-Hellman 密钥交换算法广泛用于允许双方达成会话密钥的协议。它本身并不能保证双方就是他们所说的那样，或者没有中间人，但它确实强烈有助于防御被动监听者；其专利已于 1997 年到期。如果您使用 Diffie-Hellman 创建共享秘密，请确保先对其进行哈希处理（如果直接使用其共享值，则存在攻击）。

NIST 开发了数字签名标准（DSS）（它是 ElGamal 密码系统的修改版）用于数字签名生成和验证；其开发的一个条件是它必须是专利免费的。

与典型的对称密钥相比，RSA、Diffie-Hellman 和 El Gamal 的技术需要更多的位数才能达到等效的安全性；在这些系统中，1024 位的密钥被认为大约等同于 80 位的对称密钥。512 位的 RSA 密钥被认为完全不安全；Nicko van Someren 已经证明，仅使用现有的办公硬件（更不用说专门为此设计的设备）就可以在 6 周内分解这样的小 RSA 密钥。在过去，1024 位的 RSA 密钥被认为相当安全，但最近在分解算法方面的进步（例如，D. J. Bernstein 的工作）引发了担忧，也许即使是 1024 位对于 RSA 密钥来说也可能不够。Netcraft 在 2012 年指出，CA/B 论坛（证书颁发机构联盟）和 NIST 都建议到 2013 年逐步淘汰 2048 位以下的 RSA 公钥。今天，RSA 用户应该使用至少 2048 位长的密钥。

如果你需要一个需要更少位数的公钥（例如，用于智能卡），那么你可能可以使用椭圆曲线密码学（IEEE P1363 提供了一些建议曲线；找到曲线是困难的）。然而，要小心 - 椭圆曲线密码学没有专利，但某些加速技术是受专利保护的。椭圆曲线密码学足够快，对于其通常用于加密会话/批量加密密钥的使用，实际上并不需要这些加速技术。一般来说，你不应该尝试使用椭圆曲线进行批量加密；对称算法更快，并且更适合这项工作。

## 11.5.4\. 密码学散列算法

一些程序需要一个单向加密散列算法，也就是说，一个函数可以接受任意数量的数据并生成一个固定长度的数字，这个数字对攻击者来说很难逆转（例如，攻击者很难创建一组不同的数据来生成相同的值）。从历史上看，MD5 被广泛使用，但到了 20 世纪 90 年代，就有警告称 MD5 已经变得过于脆弱[van Oorschot 1994] [Dobbertin 1996]。此后，论文表明 MD5 根本不能作为加密散列来信任 - 请参阅[`cryptography.hyperlink.cz/MD5_collisions.html`](http://cryptography.hyperlink.cz/MD5_collisions.html)。不要使用原始的 SHA（现在称为“SHA-0”）；SHA-0 与 MD5 有相同的弱点。在 MD5 被破解后，SHA-1 成为了典型的首选，并且它多年来一直表现良好。然而，SHA-1 今天也已经变得过于脆弱；SHA-1 永远不应该用于新程序的安全，现有程序应该实现替代散列算法。今天的程序应该使用更好、更安全的散列算法，如 SHA-256 / SHA-384 / SHA-512 或更新的 SHA-3。

## 11.5.5\. 完整性检查

在通信时，你需要某种完整性检查（不要仅仅依赖于加密，因为攻击者可以诱导信息变化为“随机”值）。这可以通过散列算法来完成，但不要直接使用散列函数（这会让用户暴露于“扩展”攻击 - 攻击者可以使用散列值，添加他们选择的数据，并计算新的散列）。通常的方法是“HMAC”，它计算完整性检查如下

```
   H(k xor opad, H(k xor ipad, data)).
```

其中 H 是散列函数，k 是密钥。这在 IETF RFC 2104 中有详细定义。

注意，在 HMAC 方法中，接收者可以伪造与发送者相同的数据。这通常不是问题，但如果必须避免这种情况，则应使用公钥方法，并让发送者使用发送者私钥“签名”数据 - 这避免了这种伪造攻击，但成本更高，并且对于大多数环境来说并不必要。

## 11.5.6\. 随机消息认证模式（RMAC）

[NIST 已经开发和提出了一个新的模式](http://csrc.nist.gov/CryptoToolkit/modes)，用于使用名为[随机消息认证码（RMAC）](http://www.counterpane.com/crypto-gram-0301.html)的加密算法。RMAC 旨在用作消息认证码技术。

尽管有一个正式的证明表明 RMAC 是安全的，但这个证明依赖于一个高度可疑的假设，即底层加密算法符合“理想密码模型”——特别是，该算法能够抵御各种专业攻击，包括相关密钥攻击。不幸的是，许多算法的相关密钥攻击研究不足；这并不是人们在分析加密算法时最关心的一种属性或攻击。已知三重 DES 没有这种特性，而且其他广泛接受的算法如 AES 是否具有这种特性尚不清楚（似乎 AES 在相关密钥攻击方面至少比常规攻击要弱）。

目前最好的建议是“不要使用 RMAC”。还有其他进行消息认证的方法，例如结合加密散列算法的 HMAC（例如，HMAC-SHA1）。HMAC 并不是同一回事（例如，技术上它不包括 nonce，因此你应该更早地更换密钥），但 HMAC 的理论弱点仅仅是理论上的，而 RMAC 在现实世界中的问题似乎更为重要。

## 11.5.7\. 其他加密问题

你应该加密并包含重要数据的完整性检查。不要依赖于加密同时提供完整性——攻击者可能能够将位改变为不同的值，尽管攻击者可能无法将其改变为特定的值，但仅仅改变值可能就足够了。一般来说，你应该为完整性和保密性使用不同的密钥，以避免某些微妙的攻击。

一个讨论得不够充分的问题是“流量分析”问题。也就是说，即使消息被加密且加密没有被破解，对手也可能仅从加密消息中了解到很多信息。例如，如果两家公司的总裁开始交换大量加密电子邮件，这可能表明两家公司正在考虑合并。另一个例子，许多 SSH 实现被发现存在密码交换的弱点：观察者可以查看数据包并确定密码的长度（或长度范围），即使他们无法确定密码本身。他们还可以确定有关密码的其他信息，这些信息在破解密码时非常有帮助。

确保不要使问题可以部分解决，并在信任环境（信任谁）改变时使用不同的密钥。不要长时间使用相同的密钥——过一段时间后，更换会话密钥或密码，这样对手将不得不从头开始。

通常，你应该压缩将要加密的内容——这确实会增加一个固定的头部，这并不太好，但它消除了消息其余部分中的许多模式，同时也使结果更小，因此如果压缩可能使结果更小，通常被视为“双赢”。

在相关笔记中，如果你必须创建自己的通信协议，检查之前发生的问题。例如，Bellovin [1989] 对 TCP/IP 协议套件中安全问题的回顾，以及 Bruce Schneier [1998] 和 Mudge 对微软 PPTP 实现的破解及其后续工作，都可能对你有所帮助。再次强调，确保任何新的协议都得到广泛的公众审查，并且尽可能重用已有的内容。

# 11.6. 使用 PAM

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/use-pam.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/use-pam.html)

可插入认证模块（PAM）是一种灵活的用户认证机制。许多类 Unix 系统支持 PAM，包括 Solaris、几乎所有 Linux 发行版（例如，Red Hat Linux、Caldera 和 Debian 版本 2.2）以及 FreeBSD 版本 3.1。通过使用 PAM，你的程序可以独立于认证方案（密码、智能卡等）。基本上，你的程序调用 PAM，它在运行时通过检查由本地系统管理员设置的配置来确定所需的“认证模块”。如果你正在编写需要认证的程序（例如，输入密码），你应该包括对 PAM 的支持。你可以在[`www.kernel.org/pub/linux/libs/pam/index.html`](http://www.kernel.org/pub/linux/libs/pam/index.html)了解更多关于 Linux-PAM 项目的信息。

# 11.7. 工具

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/tools.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/tools.html)

一些工具可能有助于你在发布结果之前检测到安全漏洞。当然，它们不能找到所有这类问题，但它们可以帮助捕捉那些否则可能被忽视的问题。以下是一些工具，强调开源/免费软件工具。

一种明显的工具类型是检查源代码以搜索已知潜在安全问题的模式（例如，以通常导致安全漏洞的方式调用库函数）。这类程序被称为“源代码扫描器”。以下是一些这样的工具：

+   Flawfinder，是我开发的；它可以在[`www.dwheeler.com/flawfinder`](http://www.dwheeler.com/flawfinder)找到。这也是一个扫描 C/C++源代码以查找常见问题的程序，并且也遵循 GPL 许可。与 RATS 不同，Flawfinder 是用 Python 实现的。RATS 和 Flawfinder 的开发者已同意找到一种方式合作，以创建一个单一的“最佳”开源程序。

+   来自 Secure Software Solutions 的 RATS（Rough Auditing Tool for Security）可在 [`www.securesw.com/rats`](http://www.securesw.com/rats) 获取。此程序扫描 C/C++ 源代码以查找常见问题，并采用 GPL 许可。

+   来自 Cigital（以前称为 Reliable Software Technologies，RST）的 ITS4 也用于静态检查 C/C++ 代码。它免费提供给非商业用途，包括其源代码以及某些修改和再分发权利。请注意，这不是按照 [开源定义](http://www.opensource.org/osd.html)（OSD）定义的“开源”发布的——特别是，OSD 第 6 点禁止在开源许可证中包含“仅限非商业用途”条款。ITS4 可在 [`www.rstcorp.com/its4`](http://www.rstcorp.com/its4) 获取。

+   Splint（以前称为 LCLint）是一个用于静态检查 C 程序的工具。只需最小努力，Splint 就可以用作更好的 lint。如果投入更多努力添加程序注释，Splint 可以执行比任何标准 lint 更强的检查。例如，它可以用于静态检测可能的缓冲区溢出。该软件采用 GPL 许可，可在 [`www.splint.org`](http://www.splint.org) 获取。

+   cqual 是一个用于在 C 程序中查找错误的类型分析工具。cqual 通过额外的用户定义类型限定符扩展了 C 的类型系统，例如，它可以标记值是“受污染”或“未受污染”（类似于 Perl 的 taint 检查）。程序员在程序的一些地方进行注释，cqual 执行限定符推理来检查注释是否正确。cqual 使用基于 emacs 的界面 Program Analysis Mode 展示分析结果。当前版本的 cqual 可以检测 C 程序中潜在的格式字符串漏洞。cqual 的早期版本 Carillon 曾被用于在 C 程序中查找 Y2K 错误。该软件采用 GPL 许可，可在 [`www.cs.berkeley.edu/Research/Aiken/cqual`](http://www.cs.berkeley.edu/Research/Aiken/cqual) 获取。

+   Smatch 是一个通用的 C/C++ 程序静态分析工具。它生成描述程序中静态确定的状态的输出，并提供一个 API，让您定义查询潜在不良情况的查询。它最初是为分析 Linux 内核而设计的。更多信息请访问 [`smatch.sourceforge.net`](http://smatch.sourceforge.net)。

+   Cyclone 是一种类似于 C 的语言，旨在消除 C 的安全弱点。理论上，你可以总是切换到一个“更安全”的语言，但这并不总是有帮助（一种语言可以帮助你避免常见的错误，但它无法读懂你的心思）。[John Viega 已经对 Cyclone 进行了评估](http://www.securityfocus.com/guest/9094)，在 2001 年 12 月他说：“Cyclone 确实是一种很棒的语言。它是一种感觉上没有削弱任何权力的 C 方言，但它增加了强大的安全保证，以及许多对程序员来说真正有益的特性。不幸的是，Cyclone 还没有准备好进入主流。即使不考虑其限制，它也没有提供足够的优势，使其优于 Java（甚至是一个好的工具集的 C），以至于值得承担使用这种仍然非常年轻的技术所带来的风险。也许在几年后，Cyclone 将成熟为一个强大、广泛支持的编程语言，在效率上几乎与 C 相当。如果那一天到来，你肯定会看到我彻底放弃 C。” Cyclone 编译器已经在 GPL 和 LGPL 下发布。你可以从 [Cyclone 网站](http://www.research.att.com/projects/cyclone) 获取更多信息。

一些工具试图在运行时检测潜在的安全漏洞，要么是为了对抗它们，至少也是为了警告开发者。Crispin Cowan 的许多工作，如 StackGuard，都属于这一类。

有几种工具试图检测各种 C/C++ 内存管理问题；这些工具实际上是通用的软件质量提升工具，并不特定于安全领域，但内存管理问题确实可能引起安全问题。一个特别强大的工具是 [Valgrind](http://developer.kde.org/~sewardj)，它可以检测各种内存管理问题（例如使用未初始化的内存、在内存释放后读取/写入内存、在 malloc 分配的块末尾读取/写入内存以及内存泄漏）。另一个这样的工具是 Bruce Perens 的 Electric Fence (efence)，它可以检测某些内存管理错误。[Memwatch](http://www.linkdata.se/sourcecode.html)（公有领域）和 [YAMD](http://odin.ac.hmc.edu/~neldredge/yamd/)（GPL）可以检测 C 和 C++ 的内存分配问题。你甚至可以使用 GNU C 库的 malloc 库的内置功能，该库具有 MALLOC_CHECK_ 环境变量（更多信息请参阅其手册页）。还有很多其他的工具。

另一种方法是创建测试模式并运行程序，试图找到程序中的弱点。以下是一些这样的工具：

+   BFBTester，即暴力破解二进制测试器，在 GPL 许可证下发布。这个程序对二进制程序进行快速安全检查。BFBTester 执行单个和多个参数命令行溢出以及环境变量溢出的检查。版本 2.0 及以上版本还可以监视临时文件创建活动（以检查是否使用不安全的临时文件名）。曾经 BFBTester 在 Linux 上无法运行（由于 Linux 的 POSIX 线程实现中的技术问题），但这个问题已在 2.0.1 版本中得到修复。更多信息可以在[`bfbtester.sourceforge.net/`](http://bfbtester.sourceforge.net/)找到。

+   [fuzz](http://fuzz.sourceforge.net)程序是一个用于测试其他软件的工具。它通过向正在评估的程序发送随机数据来测试程序。这个工具并不真正特定于安全。

+   [SPIKE](http://www.immunitysec.com/spike.html)是一个“模糊测试创建套件”，即它是一个用于创建“随机”测试以发现安全问题的工具包。SPIKE 工具包特别设计用于通过模拟网络协议客户端进行协议分析，SPIKE proXy 是基于 SPIKE 构建的用于测试 Web 应用的工具。SPIKE 包含一些预定义的测试。SPIKE 在 GPL 许可证下发布。

有许多工具试图帮助你深入了解正在运行的程序，这些工具在尝试寻找代码中的安全问题时也非常有用。这包括符号调试器（如 gdb）和跟踪程序（如 strace 和 ltrace）。一个支持分析运行代码的有趣程序是[Fenris](http://razor.bindview.com/tools/fenris)（GPL 许可证）。其文档将 Fenris 描述为“一个多用途跟踪器、有状态分析器和部分反编译器，旨在简化错误跟踪、安全审计、代码、算法或协议分析——提供结构化程序跟踪、关于内部构造的一般信息、执行路径、内存操作、I/O、条件表达式等等。”Fenris 实际上提供了一套完整的工具，包括广泛的取证功能，以及[一个不错的 Linux 调试 GUI](http://lcamtuf.coredump.cx/fdesk.jpg)。其他一些有潜力的开源工具，这些工具可能适合调试或代码分析，可以在[`lcamtuf.coredump.cx/fenris/debug-tools.html`](http://lcamtuf.coredump.cx/fenris/debug-tools.html)找到。在这些工具中，另一个有趣的程序是 Subterfugue，它允许你控制程序在每次系统调用中发生的事情。

如果你正在构建一种常见的、存在许多标准潜在缺陷的产品（如 ftp 服务器或防火墙），你可能会发现标准安全扫描工具很有用。其中一个很好的工具是[Nessus](http://www.nessus.org)；还有很多其他的。这类工具在进行回归测试时非常有用，但鉴于它们本质上使用的是过去特定漏洞和常见配置错误的列表，它们在寻找新程序中的问题可能不太有帮助。

通常，您需要调用其他工具来实现您安全的基础设施。[开源 PKI 书籍](http://ospkibook.sourceforge.net)描述了实现公钥基础设施（PKI）的多个开源程序。

当然，在一个不安全平台配置上运行“安全”的程序几乎没有意义。您可能想检查加固系统，这些系统试图配置或修改系统以使其更难以受到攻击。对于 Linux，一个加固系统是 Bastille Linux，可在[`www.bastille-linux.org`](http://www.bastille-linux.org)找到。

另一个安全工具列表可在[`www.insecure.org/tools.html`](http://www.insecure.org/tools.html)找到。

# 11.9\. 编写审计记录

> [`dwheeler.com/secure-programs/Secure-Programs-HOWTO/write-audit-records.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/write-audit-records.html)

为程序启动、会话启动和可疑活动编写审计日志。可能有价值的信息包括日期、时间、uid、euid、gid、egid、终端信息、进程 ID 和命令行值。您可能会发现 syslog(3)函数对实现审计日志很有帮助。一个尴尬的问题是，任何日志系统都应该能够记录大量信息（因为这些信息可能非常有用），但如果信息处理不当，信息本身可能会被用来创建攻击。毕竟，攻击者控制着发送给程序的一些输入。在记录可能攻击者发送的数据时，确定一个“预期”字符列表，并转义任何“意外”字符，以防止日志被破坏。不这样做可能会真正成为问题；用户可能包括控制字符（尤其是 NIL 或行尾），这可能会引起真正的问题。例如，如果攻击者嵌入换行符，他们可以随后通过在换行符后跟随所需的日志条目来伪造日志条目。遗憾的是，似乎没有标准惯例来转义这些字符。我倾向于 URL 转义机制（%hh，其中 hh 是转义字节的十六进制值），但还有其他一些，包括 C 语言惯例（\ooo 表示八进制值和\X，其中 X 是特殊符号，例如\n 表示换行）。还有缩写系统（例如^I 是控制-I），尽管它不能优雅地处理大于 127 的字节值。 

存在着用户可能通过执行大量事件来创建拒绝服务攻击（或者至少停止审计）的危险，这些事件会截断审计记录，直到系统耗尽存储记录的资源。应对这种威胁的一种方法是对审计记录记录进行速率限制；如果“太多”的审计记录被截断，就故意降低响应速度。你可以尝试只针对可疑攻击者降低响应速度，但在许多情况下，单个攻击者可以伪装成许多潜在的用户。

选择“可疑活动”当然取决于程序的功能和预期用途。任何未能通过之前讨论的过滤检查的输入都肯定是一个候选者（例如，包含 NIL）。那些无法由正常使用产生的输入可能需要被记录，例如，一个 CGI 程序中某些必需的字段以可疑的方式缺失。任何包含诸如 /etc/passwd 或 /etc/shadow 或类似内容的输入在许多情况下都非常可疑。同样，尝试访问 Windows“注册表”文件或.pwl 文件也是非常可疑的。

不要在审计记录中记录密码。人们经常不小心为不同的系统输入密码，因此记录密码可能会允许系统管理员突破管理员域之外的计算机。

# 11.10. 物理辐射

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/physical-emissions.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/physical-emissions.html)

虽然这确实超出了本书的范围，但重要的是要记住，计算和通信设备会泄露大量信息，这使得它们难以真正实现安全。许多人知道 TEMPEST 要求，这些要求涉及计算机、显示器、键盘和其他组件的射频辐射，这些辐射可以被窃听。显示器的光线也可以被窃听，即使它从办公室墙壁上反射到很远的距离 [Kuhn 2002]。调制解调器的指示灯也足以确定底层通信。

# 11.11. 杂项

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/miscellaneous.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/miscellaneous.html)

以下是一些我似乎无法放在其他地方的杂项安全指南：

在程序使用它们之前，至少检查一下它的某些假设（例如，在程序开始时）。例如，如果你依赖于某个目录上的“粘性”位被设置，就测试它；这样的测试花费的时间很少，可能会防止严重的问题。如果你担心某些测试的执行时间，至少在安装时执行测试，或者更好的是，在应用程序启动时至少执行测试。

如果你有一个内置的脚本语言，该语言可能允许设置一个环境变量，这会不利于调用脚本的程序。对此进行防御。

如果你需要一个复杂的配置语言，确保该语言有一个注释字符，并包含一些注释出的安全示例。通常，“#”用于注释，表示“此行剩余部分是注释”。

如果可能的话，不要创建 setuid 或 setgid 的 root 程序；让用户以 root 身份登录。

在代码上签名。这样，其他人可以检查所提供的内容是否与发送的内容一致。

在某些应用程序中，你可能需要担心时间攻击，其中时间或 CPU 利用率的差异足以泄露重要信息。这种攻击已被用于从智能卡中获取密钥信息，例如。Mauro Lacy 发表了一篇题为[远程时间技术](http://maurol.com.ar/security/RTT.pdf)的论文，表明在某些情况下，你可以仅通过 CPU（可以使用论文中描述的技术远程检测）所付出的努力来确定是否存在给定的用户 ID。处理这类问题的唯一方法是在不必要的情况下也确保付出同样的努力。问题是，在某些情况下，这可能会使系统更容易受到拒绝服务攻击，因为它无法优化掉不必要的操作。

考虑静态链接安全程序。通过确保安全程序不使用动态链接库机制来对抗对动态链接库机制的攻击。然而，这有几个缺点。这可能会增加磁盘和内存的使用（由于同一例程的多个副本）。更糟糕的是，它使得更新库（例如，为了安全漏洞）变得更加困难——在大多数系统中，它们不会自动更新，必须单独跟踪和实施。

在阅读代码时，要考虑所有匹配失败的情况。例如，如果有一个 switch 语句，当没有任何一个 case 匹配时会发生什么？如果有一个“if”语句，当条件为假时会发生什么？

仅“删除”一个文件并不能从磁盘上消除文件的数据；在大多数系统中，这仅仅是将内容标记为“已删除”并使其有资格被以后重新使用，而且数据通常至少暂时存储在其他地方（如内存、交换文件和临时文件）。实际上，面对一个坚定的攻击者，覆盖数据是不够的。关于擦除磁性媒体问题的经典论文是彼得·古特曼的论文[“Secure Deletion of Data from Magnetic and Solid-State Memory”](http://www-tac.cisco.com/Support_Library/field_alerts/fn13070.html)。一个坚定的对手还可以使用其他手段，例如监控计算机的电磁辐射（军事系统必须遵守 TEMPEST 规则以克服这一点）和/或秘密攻击（如隐藏在键盘中的显示器）。

当修复安全漏洞时，考虑添加一个“警告”来检测和记录尝试利用（现在已修复的）漏洞的尝试。这将降低攻击的可能性，特别是如果攻击者无法预先确定攻击是否有效，因为它暴露了正在进行的攻击。简而言之，它将漏洞转化为入侵检测系统。这也表明，在身份验证之前暴露服务器程序的版本通常对安全来说是个坏主意，因为这样做使得攻击者很容易只使用那些会成功的攻击。一些程序允许用户故意“撒谎”关于他们的版本，这样攻击者就会使用“错误的攻击”并被检测到。此外，如果漏洞可以通过网络触发，请确保安全扫描器可以检测到该漏洞。我建议联系 Nessus([`www.nessus.org`](http://www.nessus.org))并确保他们的开源安全扫描器可以检测到问题。这样，那些不检查软件升级的用户至少可以在他们的安全漏洞扫描过程中（如果他们应该这样做的话）了解到这个问题。

在你的文档中始终包含报告安全问题的联系方式。你还应该支持至少一个用于报告安全问题的常见电子邮件地址（security-alert@SITE、secure@SITE 或 security@SITE）；让 support@SITE 和 info@SITE 也能正常工作通常是个好主意。准备好支持那些需要报告安全漏洞的行业做法，例如[完全披露政策（RFPolicy）](http://www.wiretrip.net/rfp/policy.html)和 IETF 互联网草案“负责任漏洞披露流程”。与报告安全漏洞的人迅速合作非常重要；记住，他们通过向你报告问题是在帮你，他们没有义务这样做。一旦问题得到解决，特别重要的是要给予报告漏洞的人适当的认可（除非他们要求不这样做）。许多报告者只提供信息以获得认可，通常认为应该给予报告者认可。一些供应商认为人们永远不应该向公众报告漏洞；这个论点的问题是，这曾经很常见，结果是供应商否认了漏洞，而他们的客户则连续多年被不断攻击。

在领导软件开发项目时，遵循最佳实践和常见惯例。如果你在领导开源软件/免费软件项目，可以在[免费软件项目管理 HOWTO](http://www.tldp.org/HOWTO/Software-Proj-Mgmt-HOWTO/index.html)和[软件发布实践 HOWTO](http://www.tldp.org/HOWTO/Software-Release-Practice-HOWTO/index.html)中找到一些有用的指南；你还应该阅读[大教堂与市集](http://www.catb.org/~esr/writings/cathedral-bazaar)。

不时地回顾像这样的安全指南。至少重新阅读第十二章的结论，并且随时可以回到引言重新开始！
