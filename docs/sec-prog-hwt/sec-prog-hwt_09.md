# 9.1\. 最小化反馈

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/minimize-feedback.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/minimize-feedback.html)

避免向不受信任的用户提供过多信息；简单地成功或失败，如果失败，只需说它失败了，并尽量减少失败原因的信息。简而言之，如果可能危及安全，就向不受信任的用户*最小化反馈*，并将详细信息发送到审计跟踪日志。例如：

+   如果你的程序需要某种用户身份验证（例如，你正在编写网络服务或登录程序），在用户进行身份验证之前尽可能少地提供信息。特别是，在身份验证之前避免透露你的程序版本号。否则，如果发现你的程序某个版本存在漏洞，那么没有升级到该版本的用户的广告就会向攻击者表明他们是易受攻击的。

+   如果你的程序接受密码，不要回显它；这又是一种可以看到密码的方式。

我建议在开发早期就实现审计日志。审计日志对于调试来说非常方便（因为它们被设计用来记录有用的信息而不干扰正常操作），如果它们与程序的其余部分并行开发，你更有可能将有用的状态信息包含在日志中。

# 9.2\. 不要包含注释

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/no-comments.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/no-comments.html)

当返回信息时，除非你确信接收用户能够查看它们，否则不要包含任何“注释”。这对于生成文件（如 HTML）的 Web 应用程序来说是一个特别的问题。通常，Web 应用程序程序员希望对其工作进行注释（这是可以的），但不是简单地将其注释留在代码中，而是将注释包含在返回给用户的生成文件（通常是 HTML 或 XML）中。问题是这些注释有时会提供有关系统工作方式的见解，这有助于攻击者。

# 9.3\. 处理完整/无响应的输出

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/handle-full-output.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/handle-full-output.html)

用户可能使安全程序的输出通道阻塞或对用户无响应。例如，一个网页浏览器可能会被故意停止或其 TCP/IP 通道响应变慢。安全程序应处理此类情况，特别是它应快速释放锁（最好在回复之前），这样就不会为拒绝服务攻击创造机会。始终在出站网络导向的写请求上设置超时。

# 9.4. 控制数据格式化（格式字符串）

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/control-formatting.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/control-formatting.html)

计算机语言中的许多输出例程都有一个控制生成格式的参数，例如，一个*格式字符串*语言。在 C 语言中，最明显的例子是 printf()函数族（包括 printf()、sprintf()、snprintf()、fprintf()等）。C 语言中的其他例子包括 syslog()（用于写入系统日志信息）和 setproctitle()（用于设置显示进程标识符信息的字符串）。许多以“err”或“warn”开头、包含“log”或以“printf”结尾的函数都值得考虑。Python 包括“%”操作，它在字符串上以类似的方式控制格式化。许多程序和库定义了格式化函数，通常是通过调用内置例程并进行额外处理（例如，glib 的 g_snprintf()例程）。

格式化语言本质上是一些小型的编程语言——因此，允许攻击者控制格式化字符串的开发者实际上是在运行攻击者编写的程序！令人惊讶的是，许多人似乎忘记了这些格式化能力的力量，并使用不受信任用户的数据作为格式化参数。这里的指导方针很明确——永远不要使用不受信任用户未过滤的数据作为格式化参数。未能遵循此指导方针通常会导致格式化字符串漏洞（也称为格式化漏洞）。或许通过以下示例可以最好地说明这一点：

```
   /* Wrong way: */
  printf(string_from_untrusted_user);
  /* Right ways: */
  printf("%s", string_from_untrusted_user); /* safe */
  fputs(string_from_untrusted_user); /* better for simple strings */
```

如果攻击者控制了格式化信息，攻击者可以通过精心选择格式来造成各种破坏。C 语言的 printf()函数就是一个很好的例子——有好多方法可以可能地利用 printf()函数中的用户控制的格式字符串。这包括通过创建一个长的格式字符串（这可能导致攻击者完全控制程序），使用未传递的参数的转换规范（导致意外数据被插入），以及创建产生完全不可预期的结果值的格式（例如通过前缀或后缀添加不合适的数据，导致后续使用时出现问题）。特别是 printf 的%n 转换规范，它将到目前为止写入的字符数写入指针参数；利用这个，攻击者可以覆盖原本打算打印的值！攻击者甚至可以覆盖几乎任意的位置，因为攻击者可以指定一个实际上未传递的“参数”。%n 转换规范是 C 语言从其开始就是标准的一部分，所有 C 标准都要求使用它，并且被真实程序所使用。在 2000 年，Greg KH 对源代码进行了快速搜索，并确定了使用%n 的程序有 BitchX（一个 irc 客户端）、Nedit（一个程序编辑器）和 SourceNavigator（一个程序编辑器/IDE/调试器），并且无疑还有更多。弃用%n 可能是一个好主意，但即使没有%n，也可能存在重大问题。许多论文详细讨论了这些攻击，例如，你可以看到[Avoiding security holes when developing an application - Part 4: format strings](http://www-syntim.inria.fr/fractales/Staff/Raynal/LinuxMag/SecProg/Art4/index.html)。

由于在许多情况下结果会被发送回用户，这种攻击也可以用来暴露堆栈的内部信息。这些信息随后可以被用来绕过 StackGuard 和 ProPolice 等堆栈保护系统；StackGuard 使用常量“canary”值来检测攻击，但如果堆栈的内容可以被显示，canary 的当前值就会被暴露，这使得软件再次容易受到堆栈溢出攻击的攻击。

格式化字符串几乎总是应该是一个常量字符串，可能涉及一个函数调用来实现国际化查找（例如，通过 gettext 的 _()）。请注意，这个查找必须限制在程序控制的值，即用户必须被允许只从程序控制的消息文件中选择。在使用用户数据之前过滤用户数据是可能的（例如，通过设计一个列出格式字符串合法字符的过滤器，如[A-Za-z0-9]），但通常更好的做法是简单地通过使用常量格式字符串或 fputs()来防止问题。请注意，尽管我将其列为“输出”问题，但这可能在输出之前在程序内部引起问题（因为输出例程可能正在保存到文件，或者甚至只是通过 snprintf()生成内部状态）。

输入格式化导致安全问题的这个问题并非空穴来风；参见 CERT 咨询 CA-2000-13，其中有一个利用这种弱点的攻击示例。有关这些问题的更多利用信息，请参阅 Pascal Bouchareine 的电子邮件文章，标题为“[论文]格式错误”，发表于 2000 年 7 月 18 日的[Bugtraq](http://www.securityfocus.com)。截至 2000 年 12 月，gcc 编译器的开发版本支持对不安全的格式字符串使用发出警告信息，试图帮助开发者避免这些问题。

当然，这都引发了一个问题，即国际化查找是否实际上安全。如果你正在创建自己的国际化查找例程，确保不受信任的用户只能指定合法的区域设置，而不能指定其他东西，比如任意路径。

显然，你希望将国际化创建的字符串限制在你信任的范围内。否则，攻击者可能会利用这种能力来利用格式字符串中的弱点，尤其是在 C/C++程序中。这已经在 Bugtraq 上成为讨论的话题（例如，参见 John Levon 在 2000 年 7 月 26 日的 Bugtraq 帖子）。有关更多信息，请参阅第 5.10.3 节中关于允许用户仅选择合法语言值的讨论。

虽然这实际上是一个编程错误，但值得提一下，不同国家用不同的方式表示数字，特别是小数点（.）和逗号（,）都用来分隔整数和它的分数部分。如果你保存或加载数据，你需要确保活动区域设置不会干扰数据处理。否则，一个法语用户可能无法与一个英语用户交换数据，因为存储和检索的数据将使用不同的分隔符。我不知道这被用作安全问题，但这是可以想象的。

# 9.5. 控制输出中的字符编码

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/output-character-encoding.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/output-character-encoding.html)

通常，一个安全的程序必须确保它同步其客户端与安全程序所做的任何假设。一个经常影响 Web 应用程序的问题就是它们忘记指定输出内容的字符编码。如果所有数据都来自可信源，这并不是一个问题，但如果部分数据来自不可信源，不可信源可能会悄悄地插入使用与安全程序预期不同的编码的数据。这为跨站恶意内容攻击打开了大门；有关更多信息，请参阅第 5.12 节。

[CERT 关于恶意代码缓解的技术提示](http://www.cert.org/tech_tips/malicious_code_mitigation.html)很好地解释了未指定字符编码的问题，因此我在这里引用它：

> 许多网页没有定义字符编码（HTTP 中的"charset"参数）。在 HTML 和 HTTP 的早期版本中，如果未定义字符编码，则默认为 ISO-8859-1。实际上，许多浏览器的默认值不同，因此无法依赖于默认值是 ISO-8859-1。HTML 版本 4 合法化了这一点——如果未指定字符编码，则可以使用任何字符编码。
> 
> 如果网络服务器没有指定使用的字符编码，它就无法判断哪些字符是特殊的。未指定字符编码的网页大多数情况下都能正常工作，因为大多数字符集将相同的字符分配给低于 128 的字节值。但是，高于 128 的值中哪些是特殊的呢？一些 16 位字符编码方案为特殊字符如"<"提供了额外的多字节表示。一些浏览器识别这种替代编码并对其做出反应。这是一种“正确”的行为，但它使得使用恶意脚本的攻击更难以预防。服务器根本不知道哪些字节序列代表特殊字符。
> 
> 例如，UTF-7 为"<"和">"提供了替代编码，并且几个流行的浏览器将它们识别为标签的开始和结束。这不是那些浏览器的错误。如果字符编码确实是 UTF-7，那么这是正确的行为。问题是，可能会出现浏览器和服务器对编码有不同看法的情况。

幸运的是，尽管解释这个问题很棘手，但在 HTML 中的解决方案却很简单。在 HTML 头部，只需指定字符集，如下面来自 CERT 的示例：

```
 <HTML>
<HEAD>
<META http-equiv="Content-Type"
content="text/html; charset=ISO-8859-1">
<TITLE>HTML SAMPLE</TITLE>
</HEAD>
<BODY>
<P>This is a sample HTML page
</BODY>
</HTML>
```

从技术角度来看，一个更好的方法是将字符编码作为 HTTP 协议输出的一部分设置，尽管一些库使得这个过程更加困难。这种方法在技术上更好，因为它不会强迫客户端检查头部以确定一个字符编码，以便能够读取头部中的 META 信息。当然，在实践中，一个无法正确读取上述 META 信息并正确使用的浏览器在市场上不会成功，但这是一个不同的问题。无论如何，这仅仅意味着服务器需要作为 HTTP 协议的一部分发送一个带有期望值的“charset”。不幸的是，很难热情地推荐这种（技术上更好）的方法，因为一些较旧的 HTTP/1.0 客户端没有正确处理显式的 charset 参数。尽管 HTTP/1.1 规范要求客户端遵守该参数，但这种情况足够可疑，您可能应该将其用作强制使用正确字符编码的辅助手段，而不是您唯一的机制。

# 9.6\. 防止包含/配置文件访问

> 原文：[`dwheeler.com/secure-programs/Secure-Programs-HOWTO/prevent-include-access.html`](https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/prevent-include-access.html)

在开发基于 Web 的应用程序时，不允许用户访问（读取）程序包含文件和配置文件等文件。这些数据可能提供足够的信息（例如，密码）来破解系统。请注意，这个指南有时也适用于其他类型的应用程序。您可以采取几种行动来实现这一点，包括：

+   将包含/配置文件放置在 Web 文档根目录之外（这样网络服务器永远不会提供这些文件）。实际上，除非有某些原因需要这些文件在文档根目录内，否则这是最好的方法。

+   配置网络服务器，使其不会将包含文件作为文本提供服务。例如，如果您正在使用 Apache，您可以添加一个处理程序或操作来处理 .inc 文件，如下所示：

    ```
  <Files *.inc>
       Order allow,deny
       Deny from all
     </Files>
    ```  |
    | --- |

+   将包含文件放置在受保护的目录中（使用 .htaccess），并将它们指定为不会提供服务的文件。

+   使用过滤器来拒绝文件访问。对于 Apache，可以使用以下方法：

    ```
  <Files ~ "\.phpincludes">
        Order allow,deny
        Deny from all
     </Files>
    ```  |
    | --- |

    如果您需要完整的正则表达式来匹配文件名，在 Apache 中，您可以使用 FilesMatch 指令。

+   如果您的包含文件是一个有效的脚本文件，您的服务器将解析它，请确保它不会对用户提供的参数进行操作，并且它被设计为安全的。

这些方法不能保护您免受那些可以访问您文件所在目录的用户，如果这些目录是公开可读的。您可以更改文件的权限，使得只有 web 服务器的 uid/gid 可以读取这些文件。然而，如果用户能够让 web 服务器运行他自己的脚本（用户只需编写脚本以访问您的文件），这种方法将不起作用。从根本上讲，如果您的网站托管在与其他不可信人士共享的服务器上，那么保护系统会更困难。一种方法是运行多个具有不同权限的 web 服务程序；这提供了更多的安全性，但在实践中却很痛苦。另一种方法是设置这些文件只能由您的 uid/gid 读取，并且让服务器以“您的”权限运行脚本。这种后一种方法也有其自身的问题：这意味着服务器的一些部分必须具有 root 权限，而且脚本可能拥有比必要的更多权限。
